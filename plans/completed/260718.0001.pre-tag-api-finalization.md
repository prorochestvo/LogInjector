# Task Breakdown

## Overview

Finalize the public API before the v1.0.6 tag is cut. The release branch already
contains deliberate breaking changes (httptap extraction, no shim); this plan folds in
the remaining breaking cleanups surfaced by the 2026-07-17 API review — dead error
returns, lying names, prunable surface — plus support for the blessed consumption
pattern now documented in the README (stdlib front loggers via `StdLog`, sink-side
timestamps), and two pre-tag bug fixes (file-rotation restart behaviour, duplicate
hook levels). Once the tag exists, none of the breaking items can ever land again.

## Superseding decisions (2026-07-18)

These override any conflicting statement below — the body predates them.

- **Target version is `v1.0.6`, not `v1.1.0`.** The `1.x` line was opened by an accidental
  `v1.0.0` publish that cannot be unpublished (the Go module proxy and checksum DB cache
  tagged versions immutably), so the whole `1.0.x` line is treated as pre-stable and simply
  moves forward: `v1.0.6` = `v1.0.5` + 1 patch. The SemVer violation (breaking changes under
  a patch bump) is knowingly accepted — every consumer is in-house, pinned, and upgraded
  only under the user's control, so the number is effectively cosmetic. Task O renumbers the
  existing `[1.1.0]` CHANGELOG section to `[1.0.6]` and fixes the compare links
  (`v1.0.5...v1.0.6`). Where the body names `1.1.0` as the *target*, read `1.0.6`; mentions
  of the *existing* `[1.1.0]` CHANGELOG section refer to the section being renumbered.
- **KEEP `NewPublicErrorDetails` / `PublicDetailsError` / `Details()` — do NOT reverse commit
  992130b.** Re-verified across all seven consumers at the user's request: each ships its own
  local `PublicError` exposing `Details() string` plus a variadic `NewPublicError(details
  ...string)` constructor (`beacon`, `hive`, `com_lingocrm_api`, `neuro_compass` in
  `internal/errors.go`; `kz_qazaqmap`, `kz_behappy` in a `response/publicerror.go`). That is
  exactly the shape `NewPublicErrorDetails(parts ...string)` + `Details()` was added to
  absorb — the alias is the deliberate migration target for that reinvented pattern. "Zero
  current uses" reflects that no consumer has upgraded past `v1.0.5`, not that the shape is
  unwanted. Task 2 must NOT cut these three symbols and keeps their CHANGELOG `Added` bullet;
  all other Task 2 cuts stand.

## Assumptions

- **Version = `v1.0.6`** (full rationale in Superseding decisions above). Breaking changes
  ship under a patch bump; knowingly accepted because all consumers are in-house and pinned.
  No `/v2` module-path bump. The tag is not yet cut (`git tag` latest is `v1.0.5`), so no
  published contract is broken.
- **No shims, no deprecation window** for any removed/renamed symbol — matches the
  httptap-extraction precedent recorded in `CHANGELOG.md` (`[1.1.0]` BREAKING section).
- **Decoration convention is settled:** sinks stamp timestamps, emitters use
  `log.Lmsgprefix` only. Already documented in `README.md` (lines 79–81). Tasks K/L
  implement the code side of this.
- **There are SEVEN in-house consumers, not two.** The audit found `go.mod` references
  to `prorochestvo/loginjector` in: `beacon`, `hive`, `com_lingocrm_api`, `kz_qazaqmap`,
  `kz_qazaqmap_api`, `kz_behappy`, `neuro_compass` (`developments/tmp/kz_behappy` is a
  duplicate checkout, ignored). Every one pins an OLD version: six pin `v1.0.5`,
  `kz_qazaqmap_api` pins `v1.0.3`. **None** are on `v1.1.0`. The audit therefore reports
  usage against the *pre-httptap*, pre-`levels/`, pre-`NewPublicErrorDetails` surface; a

- **`kz_qazaqmap_api` (v1.0.3) is already broken against the current tree.** It uses

### Task M: Fix the rotation restart bug in `RotatingFileHandler`

- Description: Layered on Task G. Two related defects in the cyclic handler
  (`handler.go`, pre-refactor lines cited):
  1. **Index always restarts at 1** (`handler.go:115`, `index := 1`). After a process
     restart the handler writes fresh lines into `prefix.00000001.log`, which is the
     lexicographically-oldest file — and `verifyFiles` (`handler.go:441-453`) sorts
     ascending and prunes from the front, so it deletes the *newest* data before older
     data. The handler's own godoc admits this limitation (`handler.go:102-104`).
  2. **Size seeding covers only the construction-time file.** The current seed logic
     (`handler.go:120-127`) stats index-1 and seeds `fileSize` from it — good for the
     first file, but when the handler rotates *into a pre-existing higher-index file*
     (from a prior run), it resets `fileSize = 0` (`handler.go:162`) and opens that file
     in append mode, so its counter starts at 0 and the file grows to ~2× the cap before
     rotating again.

  Fix: at construction, `filepath.Glob` the folder for `prefix.*.log`, parse the hex
  index out of each name, and resume `index` at the **highest** existing index (not 1);
  seed `fileSize` from *that* file's on-disk size. Additionally, on every rotation, after
  advancing `index` and computing the new filename, `os.Stat` the new target and seed
  `fileSize` from its existing size (0 if absent) rather than blindly resetting to 0 —
  this closes sub-bug 2 for the rotate-into-preexisting case. `WithFreshStart()` still
  forces a clean index-1 start (truncate + `index = 1` + `fileSize = 0`), overriding the
  resume-highest logic.
- Acceptance Criteria:
  - New subtest "resumes at highest existing index across restart": pre-create
    `prefix.00000001.log`, `.00000002.log`, `.00000003.log` with content; construct a
    fresh `RotatingFileHandler` (no `WithFreshStart`); the first write lands in
    `.00000003` (or `.00000004` if `.00000003` is already at cap), NOT `.00000001`.
  - New subtest "does not delete newest data on restart": with `maxFiles` at the limit,
    after a simulated restart + a few writes, the file containing the most recent write
    is NOT among the pruned files.
  - New subtest "rotating into a pre-existing file seeds its size": pre-create the
    *next* index file near the cap; a single write that rotates into it triggers a
    further rotation instead of letting it grow to ~2× cap.
  - `WithFreshStart()` still starts at index 1 with a truncated file (the migrated
    fresh-start test stays green).
  - Index parsing tolerates unexpected filenames in the folder (a foreign `x.log` or a
    malformed index) without panicking — it skips them and falls back to the max of what
    it could parse, or 1 if none parse.
  - Suite green under `-race`; the `...ForRaceCondition` coverage still passes.
- Pitfalls & edge cases:
  - Index format is `%0.8X` uppercase hex (`handler.go:116`) — parse with
    `strconv.ParseUint(hexPart, 16, 32)`. Glob returns full paths; strip
    `folder`+prefix+`.`+ and the `.`+ext suffix carefully; reject names that do not match
    the exact `prefix.<8 hex>.log` shape.
  - Interaction with `verifyFiles`: resuming at the highest index means the newest file
    is now the lexicographically-*largest*, so ascending-sort front-pruning correctly
    removes the oldest — the fix makes `verifyFiles` correct, do not also change
    `verifyFiles`.
  - Off-by-one: if the highest existing file is already at/over the cap, the first write
    should append then rotate (existing overflow logic handles this once `fileSize` is
    seeded from the real size) — verify the seed makes the *first* write rotate rather
    than the second.
  - Do not break the fail-soft `seedErr`-on-first-Write contract; a Glob error should be
    surfaced the same way (stashed, joined into the first Write, then cleared).
- Complexity: Hard
- Code Example:
  ```go
  // at construction, after the prefix validation:
  index := 1
  var fileSize uint64
  var seedErr error
  if matches, err := filepath.Glob(filepath.Join(folder, prefix+".*."+defaultFileExtension)); err != nil {
      seedErr = err
  } else {
      for _, p := range matches {
          base := filepath.Base(p)
          mid := strings.TrimSuffix(strings.TrimPrefix(base, prefix+"."), "."+defaultFileExtension)
          if len(mid) != 8 { // reject non-8-hex middles (foreign files)
              continue
          }
          if n, e := strconv.ParseUint(mid, 16, 32); e == nil && int(n) > index {
              index = int(n)
          }
      }
      // seed size from the resolved highest-index file (0 if it does not exist yet).
      if fi, e := os.Stat(filepath.Join(folder, fmt.Sprintf("%s.%08X.%s", prefix, index, defaultFileExtension))); e == nil {
          fileSize = uint64(fi.Size())
      } else if !os.IsNotExist(e) {
          seedErr = errors.Join(seedErr, e)
      }
  }
  ```

### Task N: Dedupe hook levels and document the ">=X alert" guidance

- Description: Two defects in `Hook` (`logger.go:93-118`):
  1. `Hook(w, X, X)` (or `Hook(w, X, additional...)` with a repeat) appends one `*hook`
     entry per level occurrence sharing one wrapped writer (`logger.go:103-115`);
     `WriteLog` then collects *both* matches (`logger.go:173-177`) and writes the message
     twice to the same sink. Dedupe the level set at registration: build the ordered set
     of distinct levels (`level` plus `additional`, first occurrence wins) and append one
     `*hook` per distinct level. `TestLogger_Unhook` (`logger_test.go:97-98`) currently
     registers `Hook(b, Warning, Severe, Severe)` and asserts `Len(hooks, 3)` — that
     assertion must change to `Len(hooks, 2)` after dedup.
  2. Add a godoc note on `Hook` (and a README "Hooks" section note, Task P) that
     "alert on level ≥ X" must be expressed with a `WithMinLevel(X, handler)` handler,
     NOT by enumerating exact-level hooks. Reason: hooks fire on *exact* level match
     (`logger.go:174`, `level == h.Level`), so enumerating `Error, Severe, Critical` and
     forgetting one silently drops that level — this made `com_lingocrm_api` miss
     `Critical` (value 6 in `levels/`) in its Telegram alerts. **Live evidence:**
     `com_lingocrm_api/internal/logger.go:126` registers its Telegram hook as
     `l.Logger.Hook(hndl, LogLevelError, LogLevelSevere)` — Error and Severe only,
     Critical omitted. This is the exact bug the guidance prevents; the note should cite
     `WithMinLevel(LogLevelError, hndl)` as the correct form.
- Acceptance Criteria:
  - `Hook(w, X, X, X)` registers exactly one `*hook`; a `WriteLog(X, msg)` writes the
    message to that sink exactly once (assert the sink buffer contains the message once,
    not doubled).
  - `Hook(w, A, B, A)` registers two hooks (levels A and B), preserving first-seen order.
  - `TestLogger_Unhook` updated to assert `Len(hooks, 2)` for the `Warning, Severe,
    Severe` input, and still verifies a single `Unhook(id)` removes all entries sharing
    that ID.
  - New subtest under `TestLogger_Hook`: "duplicate levels dedupe" — register a hook with
    a repeated level, write once, assert the sink received exactly one copy.
  - `Hook` godoc documents the exact-match semantics and points to `WithMinLevel` for
    threshold alerting.
  - Suite green under `-race`.
- Pitfalls & edge cases:
  - `Unhook` (`logger.go:121-134`) removes by ID and must still remove all remaining
    entries for a deduped hook (now fewer entries per ID) — it already filters by ID, so
    it keeps working, but the test that asserted 3→0 now asserts 2→0.
  - Preserve the "one shared wrapped writer per Hook call" property (`logger.go:99-101`)
    so concurrent writes to different levels of the same hook never race — dedup reduces
    entry count but must not create a second wrapper.
  - Do not change the fire-below-minimum asymmetry (`TestLogger_HookBelowMinimum`,
    `logger_test.go:77`) — that is correct and orthogonal.
- Complexity: Medium

## Risks

- **All seven consumers are on OLD versions (six v1.0.5, one v1.0.3), none on v1.1.0.**
  The audit measures usage against the pre-httptap, pre-`levels/`,
  pre-`NewPublicErrorDetails` API. A 1.1.0-only symbol (e.g. `NewPublicErrorDetails`, the
  httptap constructors) showing "0 uses" partly reflects that nobody upgraded — but here
  the zero was *also* confirmed by source: consumers roll their own `Details()` type, so
  the library alias is genuinely unadopted, not merely un-upgraded-to. The distinction is
  documented per-symbol in Task 1 so the cut rests on evidence, not on a bare zero.
- **`CyclicOverwritingFilesHandler` is used by all seven consumers** — Task G's rename is
  the single highest-ripple change in the fleet. Every consumer's `logger.go` breaks on
  upgrade. Migration is out of scope here (deferred), but the CHANGELOG rename table must
  make the old→new mapping unmissable so the eventual upgrades are mechanical.
- **`kz_qazaqmap_api` (v1.0.3) is already broken against the current tree** (uses
  long-removed `LogEvent`/`CreateLogEvent`/`UseAsDefault`/`ExtractMethodTrace`/
  `NewHttpPayloadHandler`). It is also the sole user of `JoinAs` and `CloseOrIgnore`. Its
  upgrade is a separate, larger migration; do not let its single-consumer usages block
  cuts that the ≥2 rule already clears.
- **`fx_rate_monitor` does not exist** under either scanned root — not among the seven
  importers. Residual risk only if it turns up in an unscanned location; treat as absent.
- **`SafeBuffer` relocation couples test packages.** Seven `_test.go` files across two
  packages depend on it. A botched move (wrong package clause, black-box vs white-box
  mismatch) breaks the build in a way `gofmt`/`vet` will not catch until compile. The
  relocation must respect each test file's `package` line.
- **`NewPublicErrorDetails` (992130b) is KEPT, not reversed** (user decision, 2026-07-18).
  It is unused *today* only because no consumer has upgraded past v1.0.5; all seven reinvent
  the same `PublicError.Details()` shape locally, so the alias is the migration target.
  Shipping an as-yet-unused symbol is the accepted cost of that forward compatibility.
- **The parallel background audit produced false positives** (stdlib `log.Fatalf` counted
  as `Logger.Fatalf`; local `Details()` counted as loginjector's; consumer JSON wrappers
  counted as `Json*`). All were caught by direct source verification. Lesson for the
  implementer: do not trust a name-only grep across these consumers — several loginjector
  symbol names collide with the consumers' own identically-named local types and stdlib.
- **`NewFileLogger` file-format change (Task L)** alters on-disk log lines from
  `"<prefix> <msg>"` to `"<timestamp> <prefix> <msg>"`. Any consumer that greps/parses
  its own log files by column position breaks. Pre-tag this is acceptable, but it is a
  silent behaviour change — the CHANGELOG must call it out.
- **Default-console-sink change (Task F)** moves a bare `NewLogger(min)` from
  stderr/no-timestamp to stdout/timestamped. Anyone relying on the old default sees
  different output on a different stream. Documented, but easy to miss.
- **Shared test-file edits collide.** Tasks A, C, H, N, E all edit `logger_test.go`;
  A and H also edit `handler_test.go` and `httptap/payload_test.go`. Running these
  fully in parallel risks merge churn — sequence the `logger_test.go` edits or assign
  them to one engineer pass.
- **Two Hard tasks stacked (G→M)** in the file-rotation path — the highest-complexity,
  highest-regression-risk area. The existing race tests are the safety net; do not weaken
  them during the rename.

## Trade-offs

- **Version is `v1.0.6`, a patch bump carrying breaking changes** (decision: the `1.x` line
  is a write-off from an unrecallable accidental `v1.0.0` publish, so it is treated as
  pre-stable and just moves forward; all consumers are in-house and pinned, making the number
  cosmetic. Chosen over both a truthful `v2.0.0` — which would churn the import path across
  seven projects — and `v1.1.0`. The tag is not cut, so no published contract breaks).
- **No shims / no deprecation window** for any breaking change (decision: chose the
  clean break over compatibility aliases because the httptap extraction already set this
  precedent in the same release, and maintaining shims for a not-yet-published version is
  pure overhead).
- **Panic on nil in constructors (B) over returning an error** (decision: chose panic
  because a nil logger/next is a programmer error caught at wiring time, not a runtime
  condition a caller can meaningfully handle — mirrors `regexp.MustCompile`; the error
  return forced a dead `if err != nil` on every consumer).
- **Per-`Logger` hook counter under the existing lock (H) over a package-global atomic**
  (decision: chose the field-under-`l.m` because `Hook` already holds the write lock, so
  no new synchronization primitive is needed and IDs only need to be unique within one
  logger — which is exactly how `Unhook` uses them).
- **Preserve bug-for-bug behaviour in G, fix in M** (decision: chose splitting the rename
  from the fix so the refactor is reviewable as a pure no-behaviour-change move and the
  bug fix is isolated and independently testable — "refactor and fix in one commit" would
  make the diff impossible to reason about).
- **`SafeBuffer` → `internal/synctest/` over keeping it public** (decision: audit showed
  0 external consumers; chose de-exporting because it is test scaffolding that never
  belonged in the shipped public surface — a library's public API should not advertise a
  mutex-wrapped buffer that exists only to make the library's own tests race-clean).
- **`PrintHandler` kept over removed** (decision: audit showed 3 consumers — chose keeping
  it, repurposed from stderr-`println` to a plain stdout writer, over removal that would
  break three projects on upgrade. It loses only its default-handler status).
- **Final Task 2 cut list (audit-decided, rule: keep ≥2 consumers, cut below):**
  - CUT `JoinAs` — 1 consumer (kz_qazaqmap_api); trivially replaced by
    `log.SetOutput(l.WriterAs(level))`.
  - CUT `SilenceHandler` — 0 consumers; `io.Discard` covers it.
  - KEEP `PublicDetailsError` + `Details()` + `NewPublicErrorDetails` — 0 current uses, but
    re-verified as the deliberate migration target for the local `PublicError.Details()`
    shape all seven consumers reinvent. Commit 992130b is NOT reversed (user decision,
    2026-07-18).
  - CUT / de-export `SafeBuffer` + `NewBuffer` + `NewBufferString` — 0 external consumers;
    relocate to `internal/synctest/` for the library's own tests.
  - CUT `CloseOrPrintLn` — 0 consumers.
  - CUT `CloseOrIgnore` — 1 consumer (kz_qazaqmap_api); replace with inline
    `_ = c.Close()` or the retained `CloseOrLog`.
  - CUT all eight `Json*` + delete `json.go`/`json_test.go` — 0 loginjector consumers.
  - KEEP `PrintHandler` (3), `CloseOrLog` (1, not a candidate),
    `CloseOrPanic`/`CloseOrJoin`/`CloseOrLogError`, `NewPublicError`/`PublicError`,
    `TraceError`/`HttpError` (actively used).
