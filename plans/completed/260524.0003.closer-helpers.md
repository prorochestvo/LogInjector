# Task Breakdown

## Overview

Three consumer projects independently reinvented close-helper variants that
`loginjector` doesn't ship. The library's `closers.go` only has the four
`CloseOr{Panic,Log,PrintLn,Ignore}` helpers, each taking a single `io.Closer`
and choosing the error sink internally — none accept a caller-supplied
`io.Writer`, and none cover the `defer func(){ err = errors.Join(err, c.Close()) }()`
named-return idiom that is inlined across the consumers and also lives 4× in this
library's own `json.go`.

This plan extracts two missing helpers into `closers.go`, plus one small public
thread-safe buffer that duplicate-implementing consumers (and this library's own
tests) keep wanting:

1. **`CloseOrLogError(w io.Writer, c io.Closer)`** — close, and on error write the
   message to a caller-supplied `io.Writer` (e.g. `logger.WriterAs(level)`). This is
   the single biggest gap: all three consumers have it.
2. **`CloseOrJoin(errp *error, c io.Closer)`** — close and join the close error into
   a named-return `*error`, replacing the hand-inlined `errors.Join` defer idiom.
   This is the *exact* shape of the bug the library just fixed in `json.go` (where the
   joined result was discarded); a helper makes that bug structurally unrepeatable.
3. **`SafeBuffer`** — a minimal mutex-wrapped `bytes.Buffer` exposing
   `Read`/`Write`/`String`/`Bytes` plus `NewBuffer`/`NewBufferString` constructors. One
   consumer ships `threadsafe.Buffer`; it duplicates the library's internal `writer`
   thread-safety and is a genuinely useful concurrent test sink (today's tests reach
   for a bare `bytes.Buffer`, which is only safe because the logger serializes writes —
   a buffer that is safe on its own is strictly better for fan-out tests).

The ctx-keyed variant (`CloseOrLogErrorInContext`) is **deliberately excluded** — see
Trade-offs. Pulling `context` into a tiny, dependency-light library to serve one
consumer's app-specific convention is the wrong layer.

**Hard constraint:** additive only, backward compatible, ships as a new **minor**
version. Every existing `CloseOr*` signature stays byte-for-byte unchanged. The repo's
"no skipped errors" rule and its documented exceptions (`fmt.Fprint*` to loggers,
`Rollback()` in recovery, `.Close()` in `defer`/`t.Cleanup`) are respected throughout.

## Assumptions

- **Naming:** `CloseOrLogError` is the chosen name for the writer-targeted helper.
  Rationale: the existing family is `CloseOr<Action>`; `CloseOrLog` is taken (logs via
  stdlib `log`), so the writer-targeted variant needs a distinct, self-describing name.
  `CloseOrLogError(w, c)` reads as "close, or log the error (to this writer)". The
  `io.Writer` is the **first** parameter to mirror stdlib `fmt.Fprintf(w, …)` ordering
  and to keep the `io.Closer` last, consistent with the rest of the family taking the
  closer as the trailing/only arg. (Alternative `CloseOrWrite` was rejected: it reads as
  "close or write the closer's contents", which is misleading.)
- **`CloseOrLogError` write format:** writes `err.Error()` followed by a single
  trailing `\n`, so the output is one line per failure and plays well with a
  line-oriented logger writer (`WriterAs` forwards each `Write` as one log record). The
  `\n` matches `CloseOrLog` (stdlib `log.Println`) and `CloseOrPrintLn` (builtin
  `println` adds its own newline) behavior.
- **Nil handling is a contract, not a panic:** `CloseOrLogError` with a nil `w` falls
  back to a silent no-op for the write (still closes the closer); a nil `c` is a no-op
  entirely. `CloseOrJoin` with a nil `errp` still calls `Close` but has nowhere to
  store the error — it must not panic; it joins into nothing (documented). A nil `c` is
  a no-op. `SafeBuffer` zero value is usable (mirrors `bytes.Buffer`). These guards are
  explicit because consumers will pass `logger.WriterAs(level)` results and named
  returns from `defer`, and a nil there must degrade gracefully, not crash the caller's
  cleanup path.
- **`SafeBuffer` scope is minimal:** it wraps `bytes.Buffer` behind a `sync.Mutex` and
  exposes only `Read`, `Write`, `String`, `Bytes`, plus `NewBuffer([]byte)` and
  `NewBufferString(string)`. No `Len`, `Cap`, `Truncate`, `Reset`, `ReadString`, etc.
  until a consumer actually needs them — no speculative surface.
- **`Bytes()` snapshot semantics:** `SafeBuffer.Bytes()` returns a **copy**, not the
  live underlying slice. `bytes.Buffer.Bytes()` aliases internal storage, which a later
  `Write` can mutate or reallocate out from under a concurrent reader — returning the
  live slice would reintroduce exactly the data race the type exists to prevent. This is
  the one place `SafeBuffer` intentionally diverges from `bytes.Buffer` semantics, and
  it must be documented on the method.
- Go 1.22.2 module, `CGO_ENABLED=0`, only deps are `stretchr/testify` (tests) and
  `twinj/uuid` — no new dependencies are introduced by this plan.

## Tasks

### Task 1: Add `CloseOrLogError(w io.Writer, c io.Closer)` to `closers.go`
- **Description:** Add the writer-targeted close helper to `closers.go`. It closes `c`;
  if `Close` returns a non-nil error and `w` is non-nil, it writes `err.Error()+"\n"`
  to `w`. The write itself is the documented `fmt.Fprint*`-to-logger exception to the
  no-skipped-errors rule, so its error may be ignored — but ignore it **explicitly**
  with `_ =` and a comment, exactly as the rest of the codebase does (this is what the
  rule's exception covers; do not `errors.Join` the write error anywhere). Place it
  directly after `CloseOrLog` so the two log-style variants sit together. Godoc must
  start with `CloseOrLogError`, end with a period, and state: (a) it writes to the
  caller-supplied writer rather than stdlib `log`; (b) nil `w` → write is skipped; nil
  `c` → no-op; (c) the write error is intentionally discarded.
- **Acceptance Criteria:**
  - [ ] `CloseOrLogError(w io.Writer, c io.Closer)` exists in `closers.go`, signature
    exactly as specified, with a complete godoc comment.
  - [ ] On `Close` error with non-nil `w`: the error message text is written to `w`.
  - [ ] On `Close` returning nil: nothing is written to `w`.
  - [ ] Nil `w` with a failing `Close`: does not panic, closer is still closed.
  - [ ] Nil `c`: no-op, no panic, nothing written.
  - [ ] The internal write-error ignore is `_ =` with a one-line comment citing the
    `fmt.Fprint*`-to-logger exception (no naked `_ =`).
  - [ ] `go vet ./...` clean; `gofmt -l .` empty.
- **Pitfalls & edge cases:**
  - Easy to forget the nil-`w` guard and panic inside a `defer` cleanup path — that
    turns a benign close error into a crash. Guard `w != nil` before writing.
  - Do **not** reach for `fmt.Fprintf` if a plain `w.Write([]byte(err.Error()+"\n"))`
    suffices — but using `fmt.Fprintln(w, err)` is also acceptable and slightly cleaner;
    pick one and ignore its return per the documented exception. Either way the trailing
    newline must be present.
  - Don't accidentally write when `err == nil` (no "" line spam).
- **Complexity:** Easy

### Task 2: Add `CloseOrJoin(errp *error, c io.Closer)` to `closers.go`
- **Description:** Add the named-return join helper that replaces the
  `defer func(){ err = errors.Join(err, c.Close()) }()` idiom. It closes `c` and, if
  `errp != nil`, sets `*errp = errors.Join(*errp, c.Close())`. The intended call site is
  `defer CloseOrJoin(&err, c)` inside a function with a named `(… err error)` return.
  Because `errors.Join` drops nil arguments and returns nil when all args are nil, a
  successful close into a nil `*errp` correctly leaves it nil. Add `import "errors"` to
  `closers.go` (currently it imports only `io` and `log`). Godoc must start with
  `CloseOrJoin`, document the `defer CloseOrJoin(&err, c)` usage, and state the nil-`errp`
  and nil-`c` behavior.
- **Acceptance Criteria:**
  - [ ] `CloseOrJoin(errp *error, c io.Closer)` exists in `closers.go` with full godoc.
  - [ ] `defer CloseOrJoin(&err, c)` where `Close` errors and `err` was nil → `err`
    becomes the close error.
  - [ ] Where `Close` errors and `err` was already non-nil → `err` is the join of both
    (both retrievable via `errors.Is`/`errors.Unwrap` joined semantics).
  - [ ] Where `Close` returns nil → `*errp` is unchanged (stays nil if it was nil).
  - [ ] Nil `errp` with a failing `Close` → no panic, closer still closed (error has
    nowhere to go, documented).
  - [ ] Nil `c` → no-op, no panic.
  - [ ] `go vet ./...` clean; `gofmt -l .` empty.
- **Pitfalls & edge cases:**
  - Must read **and** write through `*errp`: `*errp = errors.Join(*errp, c.Close())`.
    Reading `*errp` only once before computing the join is fine, but evaluate `c.Close()`
    exactly once (don't call it twice in a guard + assignment).
  - The nil-`errp` guard must still call `Close` — the point of the helper is the close
    actually happening; only the error capture is conditional.
  - This is the helper that prevents recurrence of the recently-fixed `json.go` bug
    where `errors.Join(err, c.Close())` was computed and discarded. Do not, in turn,
    write a `CloseOrJoin` body that discards its own result.
- **Complexity:** Easy
- **Code Example** (call-site shape the helper is meant to enable — illustration only,
  not for implementation):
  ```go
  func writeThing(w io.WriteCloser) (err error) {
      defer CloseOrJoin(&err, w) // close error is joined into the named return
      _, err = w.Write(payload)
      return
  }
  ```

### Task 3: Add `SafeBuffer` in a new `buffer.go`
- **Description:** Add a public, minimal thread-safe buffer that wraps `bytes.Buffer`
  behind a `sync.Mutex`. New file `buffer.go` (it is conceptually separate from closers;
  don't bloat `closers.go`). Type `SafeBuffer struct { m sync.Mutex; b bytes.Buffer }`.
  Methods: `Write([]byte) (int, error)`, `Read([]byte) (int, error)`, `String() string`,
  `Bytes() []byte` (returns a **copy**, see Assumptions). Constructors `NewBuffer(buf
  []byte) *SafeBuffer` and `NewBufferString(s string) *SafeBuffer` mirroring the
  `bytes.NewBuffer`/`bytes.NewBufferString` constructors. Add a compile-time assertion
  `var _ io.ReadWriter = (*SafeBuffer)(nil)` so the type's `io.Reader`/`io.Writer`
  contract is checked at build time (matches the repo's interface-assertion convention).
  The zero value `SafeBuffer{}` must be usable (no constructor required), mirroring
  `bytes.Buffer`. Every exported identifier gets a godoc comment; the `Bytes()` godoc
  must call out the copy semantics and *why* (live-slice aliasing would race with
  concurrent `Write`).
- **Acceptance Criteria:**
  - [ ] `buffer.go` defines `SafeBuffer`, `NewBuffer`, `NewBufferString`, and the
    `Write`/`Read`/`String`/`Bytes` methods, each with godoc.
  - [ ] `var _ io.ReadWriter = (*SafeBuffer)(nil)` assertion present.
  - [ ] Zero-value `SafeBuffer{}` is usable for `Write` then `String`/`Bytes`.
  - [ ] `Bytes()` returns a copy: mutating the returned slice or a subsequent `Write`
    does not corrupt a previously returned slice (assert in test).
  - [ ] Concurrent `Write` from many goroutines under `-race` is clean and total bytes
    written equal the sum of payload lengths.
  - [ ] `go vet ./...` clean; `gofmt -l .` empty.
- **Pitfalls & edge cases:**
  - **The whole point is `-race` cleanliness.** Every method that touches `b` (including
    `String`, which calls `b.String()`, and `Bytes`) must take the mutex — `String` and
    `Bytes` are reads but they read mutable state and must be serialized against `Write`.
  - `Bytes()` copy: `append([]byte(nil), sb.b.Bytes()...)` under lock. Returning
    `sb.b.Bytes()` directly is a latent race and defeats the type's purpose.
  - Don't embed `bytes.Buffer` (anonymous field) — embedding re-exports `bytes.Buffer`'s
    full method set *unguarded* (e.g. `Truncate`, `Next`, `WriteByte`), every one of
    which would bypass the mutex and reintroduce races. Use a **named** field `b`.
  - Don't add `Len`/`Reset`/etc. on spec — out of scope; add when a consumer needs them.
  - `Read` returns `io.EOF` semantics from the underlying buffer; preserve them (just
    delegate under lock), don't reinterpret the error.
- **Complexity:** Medium

### Task 4: Tests for `CloseOrLogError` and `CloseOrJoin` in `closers_test.go`
- **Description:** Extend `closers_test.go`. Reuse the existing `fixedCloser`
  test double (`fixedCloser{err}` already returns a fixed error and has the
  `var _ io.Closer = (*fixedCloser)(nil)` assertion). Follow the repo's "one
  `Test*` per method, scenarios as subtests" rule: `TestCloseOrLogError` and
  `TestCloseOrJoin`, each with `t.Run` subtests. Use a `*bytes.Buffer` (or the new
  `SafeBuffer`) as the writer sink for `CloseOrLogError`. Use `require` from testify.
- **Acceptance Criteria:**
  - [ ] `TestCloseOrLogError` subtests: `error written to writer`, `nil close writes
    nothing`, `nil writer does not panic and still closes`, `nil closer is no-op`.
  - [ ] `error written to writer` asserts the buffer `Contains` the error message text.
  - [ ] `nil writer / nil closer` subtests assert no panic (use `require.NotPanics`).
  - [ ] `TestCloseOrJoin` subtests: `join into nil error`, `join into existing error`,
    `nil close leaves error untouched`, `nil errp does not panic`, `nil closer is
    no-op`. The `join into existing error` subtest asserts both the pre-existing and the
    close error are retrievable (e.g. `errors.Is`).
  - [ ] To exercise the `defer`-into-named-return path, include at least one subtest that
    wraps the helper in a small local closure with a named `(err error)` return and
    `defer CloseOrJoin(&err, …)`, asserting the returned `err`.
  - [ ] `CGO_ENABLED=0 go test -race -run 'TestCloseOrLogError|TestCloseOrJoin' .`
    passes.
- **Pitfalls & edge cases:**
  - Don't bypass the no-skipped-errors rule in tests: assert the close happened where it
    matters (e.g. a `fixedCloser` variant that records it was closed) rather than just
    trusting it.
  - For `join into existing error`, seed `err` *before* the deferred close fires; the
    cleanest pattern is a closure with a named return so the `defer` runs after the body
    sets `err`.
  - Keep subtests `t.Parallel()` only where there's no shared mutable buffer; the
    writer-sink subtests each use their own local buffer, so they can parallelize.
- **Complexity:** Medium

### Task 5: Tests for `SafeBuffer` in `buffer_test.go`
- **Description:** New `buffer_test.go`. `TestSafeBuffer` with subtests covering the
  basic read/write/string/bytes round trip, the `Bytes()` copy guarantee, the
  zero-value usability, and a dedicated concurrency subtest. The repo keeps
  `…ForRaceCondition`-style coverage for concurrency-sensitive types — give `SafeBuffer`
  a `concurrent writes are race-free` subtest that fans many goroutines at `Write` and
  is meaningful under `-race`.
- **Acceptance Criteria:**
  - [ ] `TestSafeBuffer` subtests: `write then string`, `write then read`,
    `NewBufferString seeds content`, `NewBuffer seeds content`, `zero value is usable`,
    `Bytes returns a copy`, `concurrent writes are race-free`.
  - [ ] `Bytes returns a copy` mutates the returned slice and/or does a follow-up
    `Write`, then asserts an earlier snapshot is unchanged.
  - [ ] `concurrent writes are race-free` launches N goroutines × M writes, waits on a
    `sync.WaitGroup`, and asserts total length == N×M×len(payload).
  - [ ] `CGO_ENABLED=0 go test -race -run TestSafeBuffer .` passes with no race report.
- **Pitfalls & edge cases:**
  - A concurrency test that doesn't actually overlap writes (e.g. accidental
    serialization via a shared lock in the *test*) proves nothing — let goroutines race
    on the buffer directly; the buffer's own mutex is what's under test.
  - The copy test is the one most likely to be written as a no-op (asserting equality of
    two views of the same backing array). Force divergence: capture `Bytes()`, then
    `Write` more, then assert the first capture's length/contents are unchanged.
- **Complexity:** Medium

   `closers.go`; do them in one session to share the single `import "errors"` change.
   Independent of Task 3, so could be parallelized with it if two contributors split the
   work, but the shared file makes serial-in-one-session the cleaner path.
2. **Task 3** (`SafeBuffer` / `buffer.go`) — independent new file; can run in parallel

3. **Task 4** (closers tests) — depends on Tasks 1 and 2.
4. **Task 5** (`SafeBuffer` tests) — depends on Task 3.

## Risks

- **Naming bikeshed / API permanence.** Once `CloseOrLogError`, `CloseOrJoin`, and
  `SafeBuffer` ship in a tagged minor, they're public API and renaming is a breaking
  change. The names should be settled in review *before* the tag, not after. (Mitigation:
  this plan commits to specific names with rationale; reviewer has final say, but the
  decision happens pre-merge.)
- **`SafeBuffer` scope creep.** A public buffer invites "just add `Len`/`Reset`/`Cap`"
  requests forever. Mitigation: ship the minimal surface; document that additions are
  demand-driven. Each future method must individually justify the mutex-guarding cost.
- **`Bytes()` copy is a behavioral surprise** for anyone expecting `bytes.Buffer`
  drop-in semantics (the stdlib aliases). Mitigation: prominent godoc; it's a deliberate
  safety divergence, not a bug.
- **Backward-compat regressions are low-risk but checkable.** All four existing
  `CloseOr*` functions are untouched; the only edit to an existing file is the additive
  `errors` import + two new functions in `closers.go`. Risk is essentially limited to a
  gofmt/import slip — caught by Task 6.

## Trade-offs

- **Excluded the ctx-aware variant (`CloseOrLogErrorInContext`).** Chose to leave it out
  rather than pull `context` into a deliberately tiny, dependency-light logging library
  for the benefit of exactly one consumer. The ctx-key convention (which context key
  holds the writer) is app-specific glue, not a library concern — baking one project's
  key scheme into the shared lib would either be wrong for the other two or force a
  config surface that dwarfs the helper. Recommendation: that consumer keeps a 5-line
  local wrapper that pulls its writer out of ctx and delegates to `CloseOrLogError`.
  Revisit only if ≥2 consumers converge on an identical ctx contract.
- **`CloseOrLogError` writer-first parameter order** vs. closer-first. Chose writer-first
  to mirror `fmt.Fprintf(w, …)` and keep the `io.Closer` trailing (consistent with the
  rest of the family where the closer is the last/only arg). The cost is a slight
  inconsistency with `CloseOrLog(c)` where the closer is first — but `CloseOrLog` has no
  writer to order against, so there's no real precedent being violated.
- **`SafeBuffer` as a named field, not embedded `bytes.Buffer`.** Embedding would be less
  code and auto-satisfy more interfaces, but it silently re-exports the entire
  unsynchronized method set — every one a race waiting to happen. Explicit delegation of
  a minimal method set is more typing now in exchange for a type that can't be misused.
- **Three additive helpers in one minor** vs. splitting `SafeBuffer` into its own
  release. Bundled: they're all small, related "things consumers reinvented," reviewing
  them together gives a coherent picture, and one tag is less churn than three. The
  `SafeBuffer` is the most separable (different file, different concern) — if review
  wants to defer it, Tasks 3 and 5 can be cut to a follow-up plan without touching Tasks
  1, 2, 4. The close helpers (1, 2, 4) are the non-negotiable core of this plan.
