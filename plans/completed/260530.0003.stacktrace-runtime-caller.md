# Optimize stack-trace capture: replace `debug.Stack()`+text-parsing with `runtime.Caller`

## Overview

`internal.LineTrace()` is on the hot path of every `NewTraceError`,
`NewTraceErrorf`, and `WrapTraceError` call. It currently materialises the
entire goroutine stack as a string via `runtime/debug.Stack()` (kilobytes of
allocation), scans it with `strings.LastIndex` for the literal substring
`"internal/stacktrace.go"`, splits the remainder on `\n`, then runs several
`path.Split`/`strings.LastIndex` passes to extract one method name and one
file:line. The bytes that survive are a tiny fraction of what was allocated.

This plan replaces `LineTrace` with a `runtime.Caller`-based implementation
that does no full-stack dump and produces a semantically-equivalent output
string in the form `"file.go:line (method)"`. `StackTrace()` is left as a thin
wrapper around `debug.Stack()` — it really does need the full goroutine dump,
and its callers (`NewStackTraceError` and friends) are not hot paths.

To keep the per-callsite skip count honest, `LineTrace` becomes
`LineTrace(skip int)` and each caller in `error.go` passes the depth that
points at the user code. The package is `internal`, so this signature change
is free.

## Assumptions

- The only consumers of `internal.LineTrace()` are the three constructors in
  `error.go` (`NewTraceError`, `NewTraceErrorf`, `WrapTraceError`) and the
  internal test in `internal/stacktrace_test.go`. Verified by `grep`; no
  external code can import `internal` (Go's `internal/` rule), so the
  signature change is safe.
- The only consumers of `internal.StackTrace()` are the three `*StackTrace*`
  constructors in `error.go`. The full goroutine text is what they want, so
  `StackTrace()` stays unchanged.
- Test pins for output format (used to define "semantically equivalent"):
  - `error_test.go:69` — `traceErr.Line()` contains `"error_test.go:"` and
    `"TestFullTrace"`.
  - `error_test.go:85,86` — contains `"error_test.go:"` and
    `"TestNewTraceErrorf"`.
  - `error_test.go:144,145` — contains `"error_test.go:"` and
    `"TestWrapTraceError"`.
  - `internal/stacktrace_test.go:10–15` — contains `"testing/testing.go:"`
    and `"tRunner"`.
  These are all `Contains`/substring checks, not exact-equality checks, so
  exact whitespace and order can vary slightly as long as the substrings are
  present in the same string.
- The new implementation targets Go's current stable toolchain (1.22+). No
  build tags, no version-specific paths.
- The format `"file.go:line (method)"` is preserved exactly: literal space
  before the open-paren, no trailing space.

## Tasks

### Task 1: Rewrite `internal.LineTrace` with `runtime.Caller` + `runtime.FuncForPC`, accept explicit `skip int`

- **Description:** Replace the body of `LineTrace` in
  `/Users/seilbekskindirov/Projects/seilbek/loginjector/internal/stacktrace.go`.
  Change the signature from `LineTrace() string` to `LineTrace(skip int)
  string`. The argument is forwarded as-is to `runtime.Caller(skip)` — the
  caller is responsible for picking the depth that points at the user's
  call site. Inside `LineTrace`:
  1. `pc, file, line, ok := runtime.Caller(skip)`. If `!ok`, return `""`
     (matches current "could not parse stack" behavior).
  2. Shorten `file` from absolute path to `package_dir/file.go` form. The
     existing parser keeps the immediate parent directory of the file
     (`testing/testing.go`, `internal/stacktrace.go`, `error_test.go` —
     when the parent is the module root, no parent dir is prefixed). Use
     `path.Base(path.Dir(file))` to grab the parent segment, then
     `path.Base(file)` for the filename, and join with `"/"`. Special-case:
     when `path.Dir(file)` is `.` or empty, emit just the filename. Match
     the current test assertions exactly: for files inside the loginjector
     module that live at the module root (like `error_test.go`), the
     existing implementation emits `"file_dir_basename/error_test.go"`
     (the parent dir under module root); the substring check
     `"error_test.go:"` does not care about the prefix, so both behaviors
     pass. For the stdlib case (`testing/testing.go`) the parent segment
     `"testing"` IS load-bearing. **Verify the chosen shortening against
     the current parser's output for at least these three cases before
     locking it in:** `testing/testing.go`, `error_test.go` (module-root
     file), `internal/stacktrace.go`.
  3. Resolve the function name with `fn := runtime.FuncForPC(pc); name :=
     fn.Name()`. `FuncForPC` returns the fully-qualified name (e.g.
     `github.com/prorochestvo/loginjector.TestFullTrace` or
     `testing.tRunner`). Strip the package prefix using
     `strings.LastIndex(name, ".")`. If the result still has a leading
     `(` from a method receiver block like `(*Type).Method`, strip up to
     the first `(` — the current parser does this with
     `strings.LastIndex(n, "(")` after the dot strip; preserve it.
  4. Return `file + ":" + strconv.Itoa(line) + " (" + method + ")"`.
- **Acceptance Criteria:**
  - [ ] Signature is `func LineTrace(skip int) string`.
  - [ ] Output for the caller-equivalent `skip` value is a string of the
        form `"<file>:<line> (<method>)"` where `<file>` ends in the
        filename and includes at most one parent directory segment.
  - [ ] Returns `""` when `runtime.Caller` reports `ok == false`.
  - [ ] No call to `debug.Stack()` from inside `LineTrace`.
  - [ ] No `strings.Split` on a stack-text payload inside `LineTrace`.
  - [ ] Existing assertions in `error_test.go` (lines 49, 69, 85–86,
        144–145, 178–179, 199–200) continue to pass once callsites are
        updated (Task 2). Specifically `assert.Contains(...,
        "error_test.go:")` and `assert.Contains(..., "TestFullTrace" /
        "TestNewTraceErrorf" / "TestWrapTraceError")` hold.
  - [ ] `go vet ./internal/` is clean.
  - [ ] `gofmt -l ./internal/stacktrace.go` prints nothing.
- **Pitfalls & edge cases:**
  - **Inlined functions.** Under aggressive inlining, `runtime.FuncForPC`
    can return the *outer* function name, not the inlined one. For
    `LineTrace` itself, prevent this by marking it `//go:noinline` (cheap)
    OR rely on the fact that `LineTrace` is non-trivial enough that the
    compiler won't inline it — adding `//go:noinline` is the defensive
    choice and costs nothing measurable. Recommend `//go:noinline`.
  - **Package-prefix stripping.** `runtime.FuncForPC().Name()` may return
    names containing dots inside the package path (e.g.
    `github.com/foo/bar.Method`). `strings.LastIndex(name, ".")` works
    because the *last* dot is always the function separator — module
    paths use `/` between segments, not `.` except in the registrar TLD
    (`github.com`, `gopkg.in`). Edge case: a function value name like
    `pkg.(*Type).Method` ends with `.Method`; last-dot split gives
    `Method`. Good. A closure or generic shape may emit names like
    `pkg.foo[...].func1`; last-dot split gives `func1` which is the
    closer-to-user-source name and matches what the current parser would
    have produced — acceptable.
  - **File path shortening.** The current parser keeps the parent
    directory because it operates on `debug.Stack()` text which prints
    *absolute* paths and the parser then drops everything above the
    grand-parent. Make sure the new implementation produces a string that
    *contains* `"error_test.go:"` for test files under the module root
    and `"testing/testing.go:"` for stdlib. Anything more aggressive
    (e.g. just `path.Base(file)`) breaks the stdlib substring test in
    `internal/stacktrace_test.go`.
  - **`skip` is 0-based.** `runtime.Caller(0)` returns the line inside
    `LineTrace` itself; `1` returns the immediate caller of `LineTrace`.
    Each callsite in Task 2 picks its own value.
- **Complexity:** Medium

  ```go
  // sketch — not to be pasted verbatim, the engineer should match exact
  // output to the test assertions and verify against the current parser.

  //go:noinline
  func LineTrace(skip int) string {
      pc, file, line, ok := runtime.Caller(skip)
      if !ok {
          return ""
      }

      // file shortening: keep "<parent>/<base>" when parent is non-trivial,
      // else just "<base>". This matches the current parser's behavior for
      // stdlib paths (testing/testing.go) and for module-internal paths
      // (error_test.go is fine either way because the assertion is a
      // substring match).
      base := path.Base(file)
      parent := path.Base(path.Dir(file))
      shortFile := base
      if parent != "" && parent != "." && parent != "/" {
          shortFile = parent + "/" + base
      }

      method := "unknown"
      if fn := runtime.FuncForPC(pc); fn != nil {
          name := fn.Name()
          if i := strings.LastIndex(name, "."); i >= 0 && i+1 < len(name) {
              name = name[i+1:]
          }
          if i := strings.Index(name, "("); i > 0 {
              name = name[:i]
          }
          if name != "" {
              method = name
          }
      }

      return shortFile + ":" + strconv.Itoa(line) + " (" + method + ")"
  }
  ```

### Task 4: Decide and document `StackTrace()` treatment — leave as-is

- **Description:** `StackTrace()` is consumed by `NewStackTraceError`,
  `NewStackTraceErrorf`, and `WrapStackTraceError` (lines 75–102 of
  `error.go`). Those constructors stash the result for later retrieval
  via `Stack()` — the consumer genuinely wants the full goroutine dump,
  formatted as text, multiline. `debug.Stack()` is the canonical way to
  produce that. Re-implementing via `runtime.Callers` +
  `runtime.CallersFrames` would let us format the dump ourselves (saving
  some allocation), but the win is small (these are not hot paths — you
  build a `StackTraceError` at the *moment of failure*, not in steady
  state) and the risk is real: the test at `error_test.go:206–211`
  asserts the stack `Contains(..., "TestNewStackTraceError")` and is
  multiline (`strings.Count(stack, "\n") > 1`). Any reformat must preserve
  those properties.

  **Decision:** leave `StackTrace()` unchanged. Document this in a code
  comment on `StackTrace()` so future readers know the optimization was
  considered and consciously deferred.
- **Acceptance Criteria:**
  - [ ] `StackTrace()` in `internal/stacktrace.go` remains a wrapper around
        `debug.Stack()`.
  - [ ] A short godoc note is added explaining that the function is kept
        on `debug.Stack()` deliberately (not a hot path, output format
        already matches consumers and tests).
  - [ ] No new callers of `StackTrace` introduced.
  - [ ] `TestNewStackTraceError` and all its subtests still pass.
- **Pitfalls & edge cases:**
  - The temptation to "while we're here, do it for `StackTrace` too" is

    `LineTrace`; `StackTrace` ships unchanged.
  - If a future plan revisits `StackTrace`, the test fixtures to honor

    package path of caller.

## Risks

- **Off-by-one `skip`.** The most likely failure mode. Mitigation:
  acceptance criteria for Task 2 are existing tests, which assert the
  caller frame by name — they will catch a wrong skip immediately.
- **Inlining changes between Go versions.** Mitigation: `//go:noinline`
  on `LineTrace` and on the three constructors that call it. Cheap and
  durable.
- **File-path format drift.** The current parser produces
  `package_dir/file.go`. If the new implementation produces just
  `file.go`, the `internal/stacktrace_test.go` assertion
  `Contains(..., "testing/testing.go:")` fails. Mitigation: explicit
  acceptance criterion in Task 1 to preserve the parent-dir prefix; spot
  check stdlib case before declaring done.
- **`runtime.FuncForPC` returning a wrapper name for generic/closure
  callsites.** Mitigation: documented as a known minor risk; the project
  has no current callers from inside generic instantiations, so the
  exposure is small. Revisit if/when consumers report unexpected method
  names in trace lines.
- **Behavior on `ok == false` from `runtime.Caller`.** Current code returns
  `""` in roughly the same shape (parser falls through to `return ""`).
  Mitigation: explicit acceptance criterion that `!ok` returns `""`.

## Trade-offs

- **Explicit `skip int` parameter vs hardcoded `runtime.Caller(2)`.**
  Chose the explicit parameter because it lets each callsite declare its
  own depth honestly. With a hardcoded skip, anyone adding a fourth
  trace-error constructor at a different call depth would silently get
  the wrong line. The cost is a slightly more verbose call site; the
  package is `internal`, so there's no public-API tax. Recommend this
  over the no-arg form.
- **Leave `StackTrace()` on `debug.Stack()` vs rewrite via
  `runtime.Callers`/`CallersFrames`.** Chose to leave it. The function is
  consumed by failure-path constructors that fire infrequently; the win
  would be modest and the risk of breaking the existing multiline output
  contract is real. Documented in Task 4 so this decision is recoverable
  from the source.
- **`//go:noinline` on `LineTrace` and on the three constructors vs trust
  the compiler.** Chose `//go:noinline`. It is a one-line annotation that
  pins the frame layout forever and removes a class of "this used to work
  before Go 1.X" bugs. The runtime cost of disabling inlining on these
  six functions is below noise — they all allocate.
- **Benchmark in same package vs separate `_test` file.** Chose separate
  `stacktrace_bench_test.go` so unit tests and benchmarks have distinct
  files. Matches the project's convention (no existing benchmarks force
  the issue, but separation is the lower-friction default).
