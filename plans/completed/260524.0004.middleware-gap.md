# Task Breakdown

## Overview

`github.com/prorochestvo/loginjector` ships an HTTP logging middleware,
`NewHttpPayloadHandler` (`middleware.go`), that captures the **full** request and
response — line, headers, request body, response body — and fans it into a `*Logger`.
A real consumer, `neuro_compass` (depends on loginjector **v1.0.5**), refused to use it
and hand-rolled its own one-line-summary middleware
(`internal/gateway/middleware/logger.go`). That is the strongest possible signal that
the library's middleware does not fit production needs.

This plan is **investigation-first**: Task 1 is a grounded diff/gap analysis between the
library's middleware and `neuro_compass`'s hand-rolled one, with each gap classified
**CONFIRMED** (proven by the neuro_compass diff) or **SPECULATIVE** (a real library
weakness, but not the thing that drove neuro_compass away). Tasks 2+ then plan
**additive, backward-compatible** improvements addressing the confirmed gaps.

**Hard constraint:** additive only. `NewHttpPayloadHandler`'s signature and default
behaviour must not change. New capability arrives via a new options-based constructor
plus exported functional options. Target a **new minor version** (v1.1.0). No behavioural
change for existing callers, ever.

Acceptance for the eventual implementation (all tasks): `CGO_ENABLED=0 go test -race ./...`
green, `go vet ./...` clean, `gofmt -l .` empty, godoc on every new export, tests covering
each new option and the interface pass-through.

## Assumptions

- `neuro_compass`'s `internal/gateway/middleware/logger.go` and `internal/logger.go`
  (both read for this plan) are representative of how the library is actually consumed —
  not a one-off experiment. Memory `[[consumer-boilerplate]]` notes three repos copy-paste
  a logger bootstrap, reinforcing that consumers route around library ergonomics.
- The library stays dependency-light (`stretchr/testify` test-only, `twinj/uuid`). No new
  runtime deps for this work — functional options and interface assertions are stdlib-only.
- "Backward compatible" means: existing import sites compile unchanged AND produce
  byte-identical log output and the same stdout summary they get today. The existing
  `SetHttpSummaryWriter` global (already merged) is preserved as-is.
- Consumer-defined `LogLevel` (no shipped constants) and the fan-out model are out of
  scope; this plan touches `middleware.go` only.
- `neuro_compass` is **READ-ONLY** context. Nothing in this plan modifies that repo. The
  goal is to make a future loginjector version that `neuro_compass` *could* adopt and
  delete its hand-rolled middleware.

## Tasks

### Task 1: Diff/gap analysis — library middleware vs. neuro_compass hand-roll (INVESTIGATION)

- **Description:** Capture the findings IN this plan (done below). Compare
  `loginjector.NewHttpPayloadHandler` / `interceptor` / `headerToString`
  (`middleware.go`) against `neuro_compass`'s `middleware.Logger` /
  `httpResponseWriter` (`internal/gateway/middleware/logger.go`), wired at
  `internal/gateway/router.go:98`. Classify every candidate gap as CONFIRMED or
  SPECULATIVE based on the actual neuro_compass code, not on theory.

- **What neuro_compass actually does** (`internal/gateway/middleware/logger.go`):
  - Signature: `func Logger(next http.Handler, logger io.Writer) (http.Handler, error)`.
    Takes an **`io.Writer`** and an **`http.Handler`**; returns an **`http.Handler`**.
  - Logs **one line only**: `"[%.3d] %s %s%s\n"` = status, method, path, optional
    `(user=<id>)`. The `user=` field comes from `ExtractUser(r.Context())`
    (`middleware/context.go:24`) — an app-domain value the library has no concept of.
  - Captures **no request body and no response body**. Its `httpResponseWriter` records
    `statusCode` and a `wroteHeader` guard — nothing else.
  - Uses the std `log.New(logger, "middleware ", log.Lmsgprefix)` for formatting; the
    log line format is the consumer's, not the library's.
  - `NewHttpPayloadHandler` (and the words "payload"/"HttpPayload") appear **nowhere**
    in the neuro_compass Go tree — verified by grep. It was evaluated and rejected, or
    never considered viable.

- **Gap classification:**

  **CONFIRMED (grounded in the neuro_compass diff — these explain the rejection):**

  - **(C1) Forced full request+response body capture, no opt-out / no size cap / no
    sampling.** `interceptor.Bytes()` (`middleware.go:147`) always emits trimmed request
    body (`i.payload`) and response body (`i.response`). `newInterceptor` unconditionally
    `io.TeeReader`s the request body into a buffer. neuro_compass logs **zero** bodies —
    it only wants status/method/path/user. For an API handling user data, buffering and
    persisting every request/response body is a heavyweight cost and a PII liability the
    consumer explicitly avoided. This is the single biggest divergence.

  - **(C2) No control over format / which fields are captured.** The library hard-codes
    the dump shape in `interceptor.Bytes()` and `headerToString`. neuro_compass wanted a
    compact custom line *including an app-domain field* (`user=<id>`) and a `"middleware "`
    prefix. The library offers no hook to choose fields, add context-derived fields, or
    change the line format. There is no seam where a consumer could inject `user=`.

  - **(C3) Signature shape mismatch: concrete `*Logger` + `http.HandlerFunc` instead of
    `io.Writer` + `http.Handler`.** Library:
    `NewHttpPayloadHandler(logger *Logger, level LogLevel, nextFunc http.HandlerFunc) (http.HandlerFunc, error)`.
    neuro_compass: `Logger(next http.Handler, logger io.Writer) (http.Handler, error)`.
    The library forces a concrete `*Logger` (the consumer would have to adapt) and works
    in `http.HandlerFunc`, while the consumer's chain is built from `http.Handler`
    (`router.go:85-104` composes `http.ServeMux`/`http.Handler`). Their hand-roll takes a
    plain `io.Writer`, which is trivially satisfiable and decoupled from `*Logger`.

  **SPECULATIVE (real library weaknesses, but NOT what drove neuro_compass — no evidence
  in their code):**

  - **(S1) `interceptor` does not preserve `http.Flusher` / `http.Hijacker` /
    `http.Pusher`.** `interceptor` embeds `http.ResponseWriter` and overrides
    `Write`/`WriteHeader`, so the wrapped writer **loses** any optional interfaces the
    underlying writer implemented. `NewHttpPayloadHandler` tries to `Flush()` the
    *original* `w` (`middleware.go:60`), but a downstream handler that type-asserts the
    *interceptor* for `http.Flusher` (SSE) or `http.Hijacker` (websocket) gets `ok ==
    false` and breaks. **Verdict for neuro_compass: not the cause** — grep found no
    `Flusher`/`Hijacker`/`Pusher`/`event-stream`/`websocket` usage in their gateway. BUT
    this is a genuine **correctness bug in the library** (P-worthy) that *will* bite any
    streaming consumer; it is worth fixing regardless. Compare: neuro_compass's own
    `httpResponseWriter` has the *same* limitation, so it is not a differentiator — it is
    a latent trap in both.

  - **(S2) Fixed, non-extensible redaction set.** `headerToString` (`middleware.go:177`)
    drops a hard-coded set: `Authorization`, `Proxy-Authorization`, `Cookie`,
    `Set-Cookie`. A consumer cannot add `X-Api-Key`, `X-Telegram-Init-Data`, etc.
    **Verdict for neuro_compass: not the cause** — they log no headers at all, so
    redaction extensibility is moot for them. Still a real gap for any consumer that
    *does* want body/header logging with custom secret headers (and ties directly to the
    C1 PII concern).

  - **(S3) Unconditional stdout summary.** Historically `NewHttpPayloadHandler` always
    `fmt.Fprintf`'d a one-line summary to stdout. **Already addressed** by the merged
    `SetHttpSummaryWriter` (`middleware.go:70`), so this is **no longer a gap** — note it
    for completeness and so reviewers don't re-litigate it. neuro_compass predates/ignored
    this anyway.

- **Acceptance Criteria:**
  - [ ] The CONFIRMED vs SPECULATIVE table above is recorded in this plan and each
        CONFIRMED gap cites the specific neuro_compass file/behaviour that proves it.
  - [ ] S1 is explicitly flagged as a library correctness bug (P-worthy) independent of
        neuro_compass.
  - [ ] S3 is recorded as already-resolved (no new work).
- **Pitfalls & edge cases:** Do not conflate "the library *could* do X better" with "this
  is *why* neuro_compass walked away." Only C1–C3 have evidence in their actual code; the
  rest are improvements justified on their own merits, not by the diff.
- **Complexity:** Medium (analysis only — no code).

### Task 2: Add `httpPayloadOption` functional-options scaffold + new constructor

- **Description:** Introduce an options-based constructor in `middleware.go` that is a
  strict superset of `NewHttpPayloadHandler`, leaving the original untouched. Proposed:
  `NewHttpPayloadHandlerWithOptions(logger *Logger, level LogLevel, nextFunc http.HandlerFunc, opts ...HttpPayloadOption) (http.HandlerFunc, error)`.
  Define an exported `HttpPayloadOption` type and an internal `httpPayloadConfig` struct
  carrying every tunable from Tasks 3–5. `NewHttpPayloadHandler` is refactored to call the
  new constructor with **zero options**, so the default config reproduces today's exact
  behaviour (full body capture, fixed redaction set, current line format, current summary
  via `httpSummaryWriter()`). This is the only allowed change to the existing function:
  delegation that is provably behaviour-preserving.
- **Acceptance Criteria:**
  - [ ] `NewHttpPayloadHandler(logger, level, next)` and
        `NewHttpPayloadHandlerWithOptions(logger, level, next)` (no opts) produce
        byte-identical logger output and identical stdout summary for the same request —
        asserted by a table test feeding a recorded request/response through both and
        comparing buffers.
  - [ ] `HttpPayloadOption` and the new constructor have godoc starting with the
        identifier name and ending with a period; docs state the default = legacy
        behaviour.
  - [ ] `go vet ./...` clean, `gofmt -l .` empty.
- **Pitfalls & edge cases:** The nil-logger guard (`middleware.go:24`) must live in the
  new constructor so both paths reject `nil` identically. Keep the existing
  `SetHttpSummaryWriter` global as the *default* summary sink; an option (Task 5) may
  override it per-handler without touching the global.
- **Complexity:** Medium

  **Code Example** (sketch — final names subject to review):

  ```go
  // HttpPayloadOption configures NewHttpPayloadHandlerWithOptions. Options are applied
  // in order; later options win. The zero set of options reproduces the exact behaviour
  // of NewHttpPayloadHandler.
  type HttpPayloadOption func(*httpPayloadConfig)

  type httpPayloadConfig struct {
      maxRequestBody  int           // <0 = unlimited (default), 0 = omit body
      maxResponseBody int           // <0 = unlimited (default), 0 = omit body
      redactHeaders   map[string]struct{} // case-insensitive; seeded with defaults
      summaryWriter   io.Writer     // nil = use package httpSummaryWriter()
  }

  func NewHttpPayloadHandlerWithOptions(logger *Logger, level LogLevel,
      nextFunc http.HandlerFunc, opts ...HttpPayloadOption) (http.HandlerFunc, error) {
      if logger == nil {
          return nil, fmt.Errorf("logger is nil")
      }
      cfg := defaultHttpPayloadConfig() // == legacy behaviour
      for _, o := range opts {
          o(&cfg)
      }
      // ... build interceptor from cfg ...
  }
  ```

### Task 3: `WithMaxBodyCapture` option — bound / disable body logging (addresses C1)

- **Description:** Add options to cap or disable request/response body capture, the #1
  confirmed gap. Proposed: `WithMaxRequestBody(n int)`, `WithMaxResponseBody(n int)`, and
  a convenience `WithoutBodies()` (sets both to 0). Semantics: `n < 0` = unlimited
  (default, legacy); `n == 0` = omit the body entirely from the dump; `n > 0` = capture at
  most `n` bytes and append a truncation marker (e.g. `…[truncated N bytes]`). Wire the
  request cap into the `io.TeeReader` path in `newInterceptor` (bound the tee'd buffer),
  the response cap into `interceptor.Write`, and respect both in `interceptor.Bytes()`.
- **Acceptance Criteria:**
  - [ ] `TestNewHttpPayloadHandlerWithOptions` subtests: `default_captures_full_body`,
        `request_body_truncated_at_limit`, `response_body_truncated_at_limit`,
        `without_bodies_omits_both`, `truncation_marker_present`,
        `zero_limit_still_logs_head_and_headers`.
  - [ ] A `…ForRaceCondition`-style test (per CLAUDE.md convention for the fan-out / file
        handlers) drives concurrent requests through a handler with a body cap under
        `-race` and shows no data race on the bounded buffer.
  - [ ] Truncation **must not** corrupt the proxied response: the downstream client still
        receives the full, untruncated body. Only the *logged copy* is bounded. Asserted
        by comparing the `httptest.ResponseRecorder` body to the original payload.
  - [ ] godoc on each option states the `<0 / 0 / >0` semantics.
- **Pitfalls & edge cases:**
  - **Do not truncate what the client sees.** The cap applies to the logging buffer only;
    `interceptor.Write` must always forward the full `b` to the wrapped writer
    (`middleware.go:136`). Easy to get wrong by truncating before the real write.
  - The request-body tee must stop *buffering* past the cap but must keep *reading*
    through to the real `Closer`, or downstream handlers that read the body get a short
    read. Use a counting writer that discards past the limit, not a hard `io.LimitReader`
    on the body itself.
  - `interceptor.Bytes()` length pre-calc (`middleware.go:160`) must account for omitted /
    truncated bodies or the `make([]byte, 0, l)` cap is wrong (correctness-neutral but
    sloppy — fix it).
- **Complexity:** Hard

### Task 4: `WithRedactHeaders` option — extensible redaction set (addresses S2, supports C1)

- **Description:** Let callers add header names to the redaction set used by
  `headerToString`. Proposed: `WithRedactHeaders(names ...string)`. The default set
  (`Authorization`, `Proxy-Authorization`, `Cookie`, `Set-Cookie`) is always present and
  **cannot be removed** (fail-safe — never make it *easier* to leak secrets). Matching is
  case-insensitive (canonicalise via `http.CanonicalHeaderKey`). `headerToString` changes
  from a free function reading a hard-coded list to one that consults the per-handler
  config's redaction set.
- **Acceptance Criteria:**
  - [ ] Subtests: `default_set_still_redacted`, `custom_header_redacted`,
        `case_insensitive_match`, `default_set_cannot_be_removed`.
  - [ ] Existing default-path output is unchanged (Task 2 byte-identity test still green).
  - [ ] godoc states the default set is always-on and additive-only.
- **Pitfalls & edge cases:** `headerToString` is currently a package-level function shared
  by request and response headers; threading per-handler config in means either making it
  a method/closure or passing the set explicitly — pick the form that keeps the default
  path allocation-free. Canonicalise both the configured names and the iterated keys, or
  case mismatches silently leak (`x-api-key` vs `X-Api-Key`).
- **Complexity:** Medium

### Task 5: `WithSummaryWriter` per-handler summary override (tidies S3)

- **Description:** Allow a per-handler summary destination without mutating the global set
  by `SetHttpSummaryWriter`. Proposed: `WithSummaryWriter(w io.Writer)`; `io.Discard`
  silences the summary for that handler only. When the option is unset, fall back to the
  package `httpSummaryWriter()` (preserving the global default and existing behaviour).
- **Acceptance Criteria:**
  - [ ] Subtests: `unset_uses_global_default`, `option_overrides_global`,
        `discard_silences_summary`, `global_unchanged_after_handler_built` (asserts the
        option did not mutate `httpSummaryOutput`).
  - [ ] godoc clarifies precedence: per-handler option > global `SetHttpSummaryWriter` >
        `os.Stdout` default.
- **Pitfalls & edge cases:** Don't reintroduce the global mutation S3 was fixing — the
  option must scope to the single handler. Read the global under its existing
  `httpSummaryMu` RLock only when the option is unset.
- **Complexity:** Easy

### Task 6: Preserve optional `http.ResponseWriter` interfaces in `interceptor` (fixes S1 — correctness)

- **Description:** Make the `interceptor` transparently expose `http.Flusher`,
  `http.Hijacker`, and `http.Pusher` when the underlying `http.ResponseWriter` implements
  them, so streaming/websocket/HTTP-2-push handlers downstream still work when wrapped.
  This is a **standalone correctness fix** flagged P-worthy in Task 1; it is NOT what drove
  neuro_compass (they have no streaming), but it is a latent bug for any streaming
  consumer. Implement explicit pass-through methods that type-assert
  `i.ResponseWriter` and either delegate or return the documented "not supported" error
  (`http.Hijacker` returns `(nil, nil, fmt.Errorf(...))` when unsupported). Note: a
  `Flush()` pass-through also makes the post-`nextFunc` `f.Flush()` on the raw `w`
  (`middleware.go:60`) redundant for callers that flush through the interceptor — keep the
  existing line for backward compatibility but verify no double-flush regression.
- **Acceptance Criteria:**
  - [ ] Subtests: `flusher_passthrough` (asserts `interceptor` satisfies `http.Flusher`
        and forwards `Flush`), `hijacker_passthrough`, `pusher_passthrough`,
        `unsupported_writer_reports_not_supported` (wrap a bare writer; assertions return
        `ok == false` / hijack returns an error, never a panic).
  - [ ] Compile-time interface checks per CLAUDE.md where it makes sense (the interceptor
        does *not* statically implement these — it conditionally does — so the tests assert
        runtime behaviour rather than `var _ http.Flusher = ...`).
  - [ ] Existing default-path test (Task 2) still byte-identical — adding pass-through
        methods must not change the dump output.
- **Pitfalls & edge cases:**
  - The classic "wrapping ResponseWriter loses interfaces" trap. Because the set of
    optional interfaces is open-ended, a method-by-method pass-through is the pragmatic
    choice over the 8-way `httpsnoop`-style combinatorial wrapper (which would also add
    surface area / a dependency). Document the chosen three and that exotic interfaces are
    not bridged.
  - `Hijack()` after capture is fundamentally incompatible with response-body buffering —
    once hijacked, the interceptor can no longer see the response. Document that hijacked
    connections are logged with whatever was captured pre-hijack (request side intact,
    response side empty). Do not pretend to capture a hijacked stream.
  - Guard against double-flush / writing headers after hijack.
- **Complexity:** Hard

- **Acceptance Criteria:**
  - [ ] `CGO_ENABLED=0 go test -race ./...` includes any new `Example` functions and they

  - [ ] README middleware section documents every new option with its default.
  - [ ] CLAUDE.md architecture note updated for the interface pass-through (S1 fix).

## Execution Order

1. **Task 1** — gap analysis (done in this plan; gates everything else).
2. **Task 2** — options scaffold + delegating constructor (foundation; everything below
   hangs off `httpPayloadConfig`).
3. **Task 3** (C1, body caps) and **Task 4** (S2, redaction) — independent of each other,
   **parallelizable** once Task 2 lands. Task 3 is the highest-value confirmed gap; do it
   first if serialising.
4. **Task 5** (summary override) — independent, **parallelizable** with 3/4.
5. **Task 6** (S1 interface pass-through) — independent of 3/4/5 (touches `interceptor`
   struct methods, not the config dump), can run in **parallel**, but its byte-identity
   guard depends on Task 2 existing.
6. **Task 7** — docs; **last**, after the option set is final.

The full three-lens reviewer fan-out (per CLAUDE.md Agent Pipeline) applies on first
review of the implemented plan. Flag Task 6 to **Lens A (correctness)** and Task 3's
"don't truncate the client's bytes" invariant to **Lens A + Lens B (security/ops)**.

## Risks

- **Behavioural-drift risk (highest):** the whole plan rests on `NewHttpPayloadHandler`
  delegating to the new constructor *without* changing output. Mitigation: the Task 2
  byte-identity test is a gate for every subsequent task; if it ever goes red, a "harmless"
  refactor has broken backward compat.
- **Body-cap correctness:** truncating the *logged* copy while forwarding the *full* bytes
  to the client is subtle (Task 3). A bug here either corrupts responses (severe) or leaks
  full bodies despite a configured cap (PII). Mitigated by explicit invariant tests, but
  this is the fragile area.
- **Interface pass-through is open-ended (Task 6):** bridging three interfaces leaves
  exotic ones (`io.ReaderFrom`, `http.CloseNotifier`) unbridged. Accepted scope; documented.
  Risk that a future streaming consumer needs a fourth — revisit then, additively.
- **Validation gap:** this plan is grounded in *one* consumer (neuro_compass). The other
  two copy-paste consumers from `[[consumer-boilerplate]]` may have different needs (e.g.
  they *do* want body logging). The options design is permissive enough to serve both, but
  confirm against a second consumer before declaring the gap closed.

## Trade-offs

- **New constructor over variadic-on-existing:** chose
  `NewHttpPayloadHandlerWithOptions(...)` over adding `...HttpPayloadOption` to the
  existing `NewHttpPayloadHandler`. Both are technically backward-compatible (Go variadics
  are append-only), but a separate constructor keeps the original signature pristine in
  godoc, makes the "default = legacy" contract obvious, and avoids tempting future authors
  to sneak behaviour changes into the canonical name. Cost: two entry points to document.
- **Method-by-method interface pass-through over a combinatorial wrapper:** chose explicit
  Flusher/Hijacker/Pusher methods over an 8-way generated wrapper (or pulling in
  `felixge/httpsnoop`). Trade-off: doesn't bridge *every* optional interface, but stays
  dependency-light (CLAUDE.md constraint) and covers the interfaces that actually matter
  for SSE/websocket/H2-push. Revisit additively if a real consumer needs more.
- **Default redaction set is immutable:** chose "callers may *add* but never *remove*
  defaults" over a fully replaceable set. Slightly less flexible, but a logging library
  should never make leaking `Authorization` a one-liner. Security beats ergonomics here.
- **Scope held to `middleware.go`:** deliberately did *not* fold the C3 signature mismatch
  (`*Logger`/`http.HandlerFunc` vs `io.Writer`/`http.Handler`) into this plan as a new
  `io.Writer`-based constructor. The library's middleware is intentionally `*Logger`-aware
  (it uses `WriteLog(level, ...)` for the fan-out). An `io.Writer` adapter is a reasonable
  *future* additive constructor but is a separate concern; bundling it would balloon scope
  and risk the backward-compat gate. Flagged for a follow-up plan.
