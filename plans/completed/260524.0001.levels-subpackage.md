# Task Breakdown

## Overview

Three downstream consumers each carry a near-identical hand-copied `internal/logger.go` that redeclares the same 6-rung severity ladder (`LogLevelDebug..LogLevelCritical` = 1..6) plus a `ParseLogLevel`. The core `loginjector` package deliberately ships no level constants — that "LogLevel is consumer-defined" stance is a feature the maintainer wants kept, so we will not pollute the root.

Instead, ship the common ladder as an **optional sub-package** `github.com/prorochestvo/loginjector/levels`. It defines exported constants of type `loginjector.LogLevel` (`Debug`..`Critical` = 1..6, byte-identical to what consumers already use, so they can delete their copies without renumbering or touching call sites), a `Parse(string) loginjector.LogLevel` (defaults to `Info`), and a `Name(loginjector.LogLevel) string` renderer. The core stays untouched: opting in is a single import; ignoring the sub-package leaves the unopinionated core exactly as it is.

This is additive, fully backward compatible, and targets a new **minor** version (e.g. v1.x.0). The sub-package imports the root; the root must never import `levels` (no import cycle).

## Assumptions

- Module path is `github.com/prorochestvo/loginjector` (confirmed in `go.mod`), Go 1.22.2. The sub-package import path is therefore `github.com/prorochestvo/loginjector/levels`, no `go.mod` edits required.
- Directory layout is a single new package directory: `levels/levels.go` (implementation) + `levels/levels_test.go` (tests). No further nesting.
- The canonical ladder consumers use is, ascending severity: `Debug=1, Info=2, Warning=3, Error=4, Severe=5, Critical=6`. Values start at **1**, not 0, matching the existing copies so the drop-in is renumber-free. (Discussed in Trade-offs.)
- `Parse` is case-insensitive, trims surrounding whitespace, and falls back to `Info` for any unrecognized input (mirrors the consumers' `ParseLogLevel` default). The exact accepted spelling set — and whether to add `warn`/`err` aliases — is resolved in Task 2 / Trade-offs.
- The consumers' existing quirk is that only `"warning"` (not `"warn"`) parses; `"err"` is likewise not accepted. We must decide whether to preserve that quirk verbatim or widen it. Default recommendation below: keep canonical names byte-for-byte AND accept `warn`/`err` as extra inbound aliases (parse-only, never emitted by `Name`).
- No constant of type `LogLevel` is added to the root package in this work; the root API surface is frozen for this plan.

## Tasks

### Task 1: Create the `levels` package with the constant ladder
- **Description:** Add `levels/levels.go` declaring `package levels`, importing `github.com/prorochestvo/loginjector`, and defining six exported constants of type `loginjector.LogLevel`. Use a typed `const (...)` block with explicit values to make the 1..6 mapping unmistakable and audit-proof against silent `iota` drift:
  - `Debug    loginjector.LogLevel = 1`
  - `Info     loginjector.LogLevel = 2`
  - `Warning  loginjector.LogLevel = 3`
  - `Error    loginjector.LogLevel = 4`
  - `Severe   loginjector.LogLevel = 5`
  - `Critical loginjector.LogLevel = 6`
  Add a package-level godoc comment on `package levels` explaining this is an *optional* preset of the consumer-defined `loginjector.LogLevel`, that the core ships no constants by design, and that values match the historical consumer ladder so existing copies can be deleted as-is. Every constant gets a godoc comment starting with its name and ending with a period.
- **Acceptance Criteria:**
  - [ ] `levels/levels.go` compiles: `go build ./...` clean.
  - [ ] Constants are exactly `Debug=1, Info=2, Warning=3, Error=4, Severe=5, Critical=6`, each of static type `loginjector.LogLevel`.
  - [ ] Package-level godoc plus a godoc comment on every exported constant (starts with the identifier, ends with a period), per CLAUDE.md.
  - [ ] `gofmt -l .` reports nothing for the new file.
- **Pitfalls & edge cases:** Do NOT use bare `iota` starting at 0 — it would silently produce 0..5 and break the drop-in promise; if `iota` is used at all it must start the block at `1`. The constants must be typed `loginjector.LogLevel`, not untyped ints, so `logger.SetMinLevel(levels.Info)` and `logger.WriteLog(levels.Error, ...)` type-check without conversions. Keep the constant declaration ascending so severity ordering reads top-to-bottom and matches the `level >= minimumLogLevel` gate in `logger.go`.
- **Complexity:** Easy

### Task 2: Add `Parse(string) loginjector.LogLevel` and `Name(loginjector.LogLevel) string`
- **Description:** In `levels/levels.go`, add two exported functions that round-trip the ladder.
  - `Parse(s string) loginjector.LogLevel`: lowercase + `strings.TrimSpace` the input, switch on it, return the matching constant; default (unknown / empty) returns `Info`. This replaces the consumers' `ParseLogLevel`. Recommended accepted set: the six canonical names (`debug`, `info`, `warning`, `error`, `severe`, `critical`) PLUS inbound-only aliases `warn` -> `Warning` and `err` -> `Error`. The alias decision is the one open question — see Trade-offs; whichever way it lands, document it in the godoc.
  - `Name(l loginjector.LogLevel) string`: map each known constant to its canonical lowercase name; for any value outside 1..6 return a stable sentinel (recommend `"unknown"`). `Name` emits only canonical names, never aliases, so `Name(Parse("warn")) == "warning"` by design.
  - Decide and document the casing contract: `Name` returns lowercase canonical names; `Parse` is case-insensitive. This keeps the round-trip total and predictable.
- **Acceptance Criteria:**
  - [ ] `Parse` is case-insensitive and whitespace-tolerant (`Parse(" INFO ") == Info`).
  - [ ] `Parse` of any unrecognized or empty string returns `Info` (documented default).
  - [ ] `Name` returns the canonical lowercase name for each of the six constants and a documented sentinel (`"unknown"`) for out-of-range values.
  - [ ] Round-trip holds for canonical names: for each constant `c`, `Parse(Name(c)) == c`.
  - [ ] If aliases are kept: `Parse("warn") == Warning`, `Parse("err") == Error`, and `Name` still emits only `"warning"`/`"error"`.
  - [ ] Both functions carry godoc comments documenting the default-to-`Info` behavior, the casing contract, and the alias decision.
  - [ ] No error is discarded with `_`; functions are pure (no I/O), so this is trivially satisfied.
- **Pitfalls & edge cases:** Empty string must hit the `Info` default, not a zero `LogLevel(0)` — `0` is below `Debug` and would silently never gate any handler. `Name` must guard out-of-range input (consumers can pass arbitrary `LogLevel` ints) instead of indexing a slice and panicking. Keep `Parse` and `Name` mutually consistent: every name `Name` can emit must be a name `Parse` accepts. Do not let `Name` emit alias spellings or the round-trip assertion in tests becomes ambiguous.
- **Complexity:** Easy

### Task 3: Tests for `levels` (`levels/levels_test.go`)
- **Description:** Add `levels/levels_test.go` (`package levels` or `levels_test` — prefer `levels_test` for black-box import-path validation) using `github.com/stretchr/testify`. Follow the repo rule: one `Test*` per function, scenarios as `t.Run` subtests. Parallelize subtests (the package is pure, no shared mutable state).
  - `TestParse` subtests: `canonical_names` (table over all six), `case_insensitive`, `whitespace_trimmed`, `unknown_defaults_to_info`, `empty_defaults_to_info`, and (if kept) `aliases_warn_and_err`.
  - `TestName` subtests: `each_constant_renders_canonical`, `out_of_range_returns_sentinel` (e.g. `Name(0)`, `Name(99)`), `name_never_emits_aliases`.
  - `TestRoundTrip` subtest: for every constant, assert `Parse(Name(c)) == c`.
  - `TestConstantValues`: assert the literal mapping `Debug==1 .. Critical==6` so any future renumbering trips a test (this is the drop-in contract, guard it explicitly).
- **Acceptance Criteria:**
  - [ ] `CGO_ENABLED=0 go test -race ./...` is green (whole module, including the new package).
  - [ ] `CGO_ENABLED=0 go test -race ./levels/` is green in isolation.
  - [ ] Tests assert constant values are exactly 1..6 (`TestConstantValues`).
  - [ ] Tests cover `Parse` round-trip, the `Info` default for unknown/empty input, and `Name` out-of-range handling.
  - [ ] `go vet ./...` clean; `gofmt -l .` empty.
- **Pitfalls & edge cases:** Use `levels_test` external package so the test exercises the real import path and exported surface (catches accidental unexported identifiers). When comparing `LogLevel` values with testify, compare the typed values directly (`require.Equal(t, levels.Info, levels.Parse("xxx"))`) — don't cast to `int` and lose the type-check intent. If you skip the alias subtest, remove it rather than leaving it `t.Skip`-ped (no dead scaffolding).
- **Complexity:** Easy

Recommended: a brief section in `README` (or a `levels`-scoped doc comment already covered in Task 1) showing `import ".../levels"` and `loginjector.NewLogger(levels.Info, ...)`, plus a one-line statement that the core remains constant-free and this is opt-in. Do NOT edit `CLAUDE.md`'s manifesto wording or add constants to the root. Then run the full verification gate.
- **Acceptance Criteria:**
  - [ ] Root package source is unchanged (no new constants, no new imports in any root `*.go`); `git diff` touches only `levels/`, `plans/`, and optionally `README`.
  - [ ] `go list -deps ./... | grep loginjector` confirms `levels` depends on root and root does NOT depend on `levels` (no cycle — Go would refuse to compile a cycle anyway, but assert the direction explicitly).
  - [ ] Full gate green: `CGO_ENABLED=0 go test -race ./...`, `go vet ./...`, `gofmt -l .` empty.
  - [ ] Every exported identifier in `levels` has a godoc comment (manual check or `go doc github.com/prorochestvo/loginjector/levels`).
- **Pitfalls & edge cases:** Resist scope creep — do not "while I'm here" add `Info`-style constants to the root, do not rewrite the consumers (out of scope for this plan; their migration is a separate follow-up). Verify nothing in the root accidentally imports `levels` for "convenience" — that would invert the dependency and create a cycle the moment someone references a core symbol from `levels`.
- **Complexity:** Easy

## Execution Order

1. **Task 1** — constants (foundation; everything else references them).
2. **Task 2** — `Parse` / `Name` (depends on the constants).
3. **Task 3** — tests (depends on 1 + 2; the value/round-trip assertions need the real functions).
4. **Task 4** — doc note + final verification sweep (last; confirms the additive/no-cycle invariants hold).

Tasks 1 and 2 land in the same file and are naturally done together; they are split here only for clear acceptance boundaries. No parallelism worth flagging — this is a one-file package.

## Risks

- **Drift between this preset and consumer copies during migration.** If a consumer's local ladder ever diverged (e.g. someone slipped a 7th level or reordered), a naive swap to `levels` would change behavior. Mitigation: `TestConstantValues` pins 1..6; consumer migration (separate plan) must diff their local constants against `levels` before deleting.
- **Future renumbering temptation.** Someone may later "tidy" the ladder to start at 0 or insert a level. That silently breaks every consumer that persisted numeric levels (configs, DB rows). Mitigation: `TestConstantValues` is the tripwire; the godoc states the values are a compatibility contract, not an internal detail.
- **Alias scope creep.** Accepting `warn`/`err` is convenient but expands the parse contract forever (you can't easily remove an accepted input later). Mitigation: keep `Name` emitting only canonical names so the *output* contract stays narrow; aliases are inbound-only and explicitly documented.

## Trade-offs

- **Sub-package vs. root constants:** Chose a sub-package over adding constants to `loginjector` to preserve the "LogLevel is consumer-defined" manifesto. Cost: consumers add one import line. Benefit: the core stays unopinionated and the preset is purely opt-in. This is the entire point of the task, so the trade is accepted by design.
- **Values 1..6 vs. starting at 0:** Chose **1..6** to match the existing consumer copies byte-for-byte, making the migration a delete-and-import with zero renumbering and zero risk to any persisted numeric levels. Starting at 0 would be marginally more idiomatic for fresh code (and lets the zero value mean "lowest") but would silently shift every persisted level by one and break the drop-in promise — not worth it. The non-zero start also means a zero-value `LogLevel` is intentionally *below* `Debug`, which is fine: it just gates nothing, consistent with `level >= minimumLogLevel`.
- **Alias question (`warn`/`err`):** Open decision, recommend **accept them as inbound-only aliases**. The consumers' historical `ParseLogLevel` accepted only `"warning"`/`"error"`, so strictly preserving the quirk means rejecting `warn`/`err`. But operators routinely type `warn`, and `Parse` already defaults unknown input to `Info` — so today `warn` silently becomes `Info`, the worst outcome (a warning misclassified as info). Accepting `warn`->`Warning` and `err`->`Error` fixes a latent footgun while keeping `Name` output canonical. If the maintainer prefers strict historical parity, drop the aliases and the alias subtest; the rest of the plan is unaffected. User has final say.
- **`Name` returns `"unknown"` for out-of-range vs. erroring:** Chose a sentinel string over an `(string, error)` signature to keep `Name` ergonomic for logging-format use (it's called on the hot path of rendering a level label, where an error return is just friction). Out-of-range is a programmer/data error, not a recoverable condition; a stable sentinel is the pragmatic call.
