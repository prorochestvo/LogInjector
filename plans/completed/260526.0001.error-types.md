# 007 — Richer error types (close the consumer-reimplementation gaps)

## Implementation Notes (as shipped)

Decisions taken during the engineer + 3-lens-review cycle that the original plan
did not pre-specify:

- **`NewTraceErrorf` uses `fmt.Errorf` (not `Sprintf`)** so a `%w` verb wraps the
  argument; consequence: `Unwrap()` is always non-nil (returns the internal
  `fmt.Errorf` node, terminating with nil when no `%w` was used). `errors.Is`/
  `errors.As` traverse through the two-hop chain. `WrapTraceError` keeps the
  one-hop semantics (`Unwrap()` returns the supplied cause directly). Both
  contracts are documented in godoc; reviewers (lens A/C) flagged the original
  godoc as misleading, hence the rewrite.
- **`WrapTraceError` and `WrapStackTraceError` render their message via
  `fmt.Sprintf`** — godoc now explicitly warns that `%w` in the format string is
  not supported there (would produce `%!w(...)`); the explicit `cause` argument is
  the wrapping mechanism. Asymmetric with `NewTraceErrorf` by design.
- **`traceErrorWithCause.Error()` was refactored** from a switch over the four
  (line/msg/cause) combinations to a parts-conditional-join shape to fix the
  `"line: : cause"` double-separator that lens A flagged for the empty-message
  case. A subtest now asserts the absence of any `": :"` substring.
- **`WrapHttpError.Error()` falls back to `"HTTP <code>"`** when
  `http.StatusText` returns `""` (unknown code), so it never produces an empty
  string. This diverges from `NewHttpError(code).Error()` (which still returns
  `""` for unknown codes — the base handler is unchanged); godoc calls out the
  divergence so a switch from the old to the new constructor cannot silently
  alter output.
- **`NewStackTraceError().Error()` returns `""`** by design (per AC) — godoc
  prominently warns that this value contributes no text to `errors.Join`/`%w`
  chains, and recommends `NewStackTraceErrorf` when a non-empty error string is
  needed in the chain.
- **Compile-time `var _ Iface = (*T)(nil)` assertions added to the EXISTING
  `traceError` and `httpError` structs** alongside the new ones (the new types
  carry them; the originals didn't). Strictly additive declarations — no body
  change to the existing structs or methods. Verified `git diff` shows zero
  changes to the existing `Error()`/`Line()`/`StatusCode()` bodies.
- **`cachedRuntime()` uses a package-level `sync.Once`** (not `sync.OnceValue`)
  for consistency with the rest of the package's idioms; race-test hammers 20
  goroutines and asserts value identity (the heavier "pointer equality" claim in
  the original comment was incorrect and got rewritten).
- **`PaywallError` / `ErrNotFound` remain excluded** (app-specific business
  rules, not library material — per plan Trade-offs).

Backward-compatibility gate: the existing `TraceError`/`HttpError` interfaces,
`NewTraceError`/`NewHttpError` constructors, and `traceError`/`httpError` struct
bodies + methods are byte-for-byte unchanged (relocated to the
public-on-top/internals-bottom layout). `internal/stacktrace.go` is untouched.

## Overview

Three downstream projects keep redefining their own error types in a local
`internal/errors.go` because `loginjector`'s `error.go` is too thin. The library
ships exactly two error types today (`error.go`):

- `TraceError` / `NewTraceError()` — carries a single `Line()` from
  `internal.LineTrace()`. It takes **no message** (you cannot attach context) and
  does **not** implement `Unwrap` (it cannot carry an underlying cause). That is the
  ergonomic dead-end that drove the reimplementation.
- `HttpError` / `NewHttpError(code)` — `StatusCode()` + an `Error()` derived from
  `http.StatusText`. It also cannot wrap a cause.

The consumer-side inventory (already collected — do not re-explore) shows what's
actually missing in the wild:

| Consumer type        | What it is / why it exists                                                              | Library action                                  |
|----------------------|----------------------------------------------------------------------------------------|-------------------------------------------------|
| `HttpCodeError`      | a near-rename of `HttpError`, but able to carry an underlying error                     | add a **cause-carrying constructor** for `HttpError`, do not duplicate the type |
| `StackTraceError`    | full `debug.Stack()` + an `init()`-cached block of OS/arch/Go-version details           | add a new **`StackTraceError`** built on `internal.StackTrace()` |
| `PublicError`        | splits a user-safe public message from the internal error (show-client vs. log)         | add a new **`PublicError`** with `Unwrap` to the internal cause |
| `PaywallError`       | app-specific business rule                                                              | **excluded** (not library material — see Trade-offs) |
| `ErrNotFound`        | app-specific sentinel                                                                   | **excluded** (not library material — see Trade-offs) |

This plan additively closes the generalizable gaps so consumers can delete their
copies. We **add new types and constructors**; we do **not** edit the existing
`TraceError` / `HttpError` interfaces or change the existing `New*` signatures.

This is a **minor** release: backward compatible, no breaking changes.

## Assumptions

1. **Additive-only, backward compatible.** `TraceError`, `HttpError`, `NewTraceError()`,
   and `NewHttpError(code)` keep their exact current shape and behavior. Existing
   callers and any external implementers of these interfaces continue to compile and
   pass. This is non-negotiable (HARD CONSTRAINT from the task).
2. **Interfaces cannot grow methods without breaking implementers**, so all new
   capability (messages, wrapping, full stacks, public/internal split) ships as **new
   exported types and/or new constructors**, never as new methods bolted onto
   `TraceError` / `HttpError`.
3. **Go 1.22.2** (per `go.mod`): `errors.Is`, `errors.As`, `errors.Join`,
   `errors.Unwrap`, and `fmt.Errorf("%w", …)` are all available. We rely on the
   standard `Unwrap() error` convention for single-cause wrapping.
4. **`internal.StackTrace()` is currently dead code** (only `internal.LineTrace()` is
   called, from `NewTraceError`). The new `StackTraceError` is its first real
   consumer — no change to `internal/stacktrace.go` is required or planned.
5. New code lives in **`error.go`** (root package `loginjector`); new tests live in
   **`error_test.go`**, matching the existing one-`Test*`-per-method + subtests
   convention. No new files unless a section grows unwieldy.
6. **Stack capture must happen in the constructor**, not lazily, so the trace reflects
   the call site that created the error (same contract as today's `NewTraceError`).
7. OS/arch/Go-version metadata is **constant for the lifetime of the process**, so it
   is computed **once** (package-level, captured at first use) and shared — not
   recomputed per error. (Matches the consumer's `init()`-cached approach; see Task 2.)
8. The line-prefix dependency: `internal.LineTrace()` keys off the literal substring
   `internal/stacktrace.go` in the raw stack. New error types that want a single line
   must go through `internal.LineTrace()` (not re-implement the parse), so this
   coupling stays in one place.

### Task 1: Add additive message + wrapping constructors for `TraceError`

- **Description (what / why / how):**
  Today `NewTraceError()` (`error.go:15`) captures only a stack line and carries no
  message and no cause. Add **two new constructors** that produce a richer trace error
  *while still satisfying the existing `TraceError` interface*, so existing
  `errors.As(err, &traceErr)` call sites keep working:
  - `NewTraceErrorf(format string, args ...any) TraceError` — captures the trace line
    (via `internal.LineTrace()`, same as today) **and** an attached formatted message.
    If any `args` contains an `error` and the format uses `%w`, the result must also
    `Unwrap` to it (standard `fmt.Errorf` semantics).
  - `WrapTraceError(cause error, format string, args ...any) TraceError` — captures the
    trace line, attaches a message, and wraps `cause` so `errors.Unwrap`,
    `errors.Is(err, cause)`, and `errors.As` traverse to it.

  Implement these on a **new unexported struct** (e.g. `traceErrorWithCause`) that
  embeds the line, a message, and an optional `cause error`, exposing `Line()`,
  `Error()` (message + line, and the cause's text when present), and `Unwrap() error`.
  Do **not** add `Unwrap` to the existing `traceError` struct unless it returns `nil`
  — and since it has no cause, simply leave `traceError` untouched.

  Rationale for keeping the line via `internal.LineTrace()`: it already strips frames
  down to `file.go:line (method)` and centralizes the brittle `internal/stacktrace.go`
  substring parsing (Assumption 8). Reuse it; do not re-walk the stack here.

- **Acceptance Criteria:**
  - [ ] `NewTraceErrorf` and `WrapTraceError` exist, are exported, return `TraceError`,
        and have godoc comments starting with the identifier name and ending with a period.
  - [ ] `var _ TraceError = (*traceErrorWithCause)(nil)` compile-time assertion present.
  - [ ] `NewTraceError()`, `traceError`, and the `TraceError` interface are **byte-for-byte
        unchanged** (verify via `git diff` on those lines).
  - [ ] `Line()` on the new type contains the **caller's** `file.go:line (method)`, not
        the constructor's internal frame (i.e. the same call-site contract the existing
        `TestFullTrace` asserts).
  - [ ] `Error()` includes both the attached message and the trace line; when a cause is
        present, the cause's message is reachable through the error string.
  - [ ] `errors.Unwrap(WrapTraceError(cause, "x"))` returns `cause`; `errors.Is` and
        `errors.As` traverse to `cause`.
  - [ ] A `TraceError` produced by either new constructor still satisfies
        `errors.As(joined, &traceErr)` after `errors.Join`, mirroring `TestFullTrace`.

- **Pitfalls & edge cases:**
  - **Frame depth.** `internal.LineTrace()` finds the **last** occurrence of
    `internal/stacktrace.go` and walks a fixed number of lines forward to find the
    caller frame. If you add helper indirection between the public constructor and
    `internal.LineTrace()`, the resolved line shifts to the wrong frame. Call
    `internal.LineTrace()` **directly inside the exported constructor**, at the same
    nesting depth as the existing `NewTraceError`. Verify with a test that asserts the
    line points at `error_test.go` and the test function name (see Task 5).
  - **`%w` vs explicit cause.** `NewTraceErrorf` with `%w` and `WrapTraceError` with an
    explicit `cause` are two paths to the same outcome. Decide one canonical internal
    representation: easiest is to have `NewTraceErrorf` build the message via
    `fmt.Errorf(format, args...)` and store *that* as the cause (so `%w` wrapping falls
    out for free), while `WrapTraceError` stores the explicit cause and a separately
    formatted message. Document which wins if a caller passes both a `%w` arg and uses
    `WrapTraceError`.
  - **Nil cause.** `WrapTraceError(nil, …)` must not panic and `Unwrap()` must return
    `nil` (so `errors.Is`/`errors.As` terminate cleanly).
  - **Empty trace.** `internal.LineTrace()` returns `""` if it can't find its anchor;
    `Error()` must still produce a useful message in that case (don't emit a bare
    leading `: ` separator with nothing before it).
  - Do not let `Error()` print the cause twice (once via your own concatenation, once
    via `fmt.Errorf("%w")`).

- **Complexity:** Medium

### Task 2: Add `StackTraceError` — full stack + cached OS/runtime details

- **Description (what / why / how):**
  Add a heavier companion to `TraceError` that carries the **full** stack (via
  `internal.StackTrace()`, which is currently unused — Assumption 4) instead of a
  single line, plus a one-time-captured block of OS / arch / Go-version details.
  This is the library home for the consumers' `StackTraceError`.

  Surface:
  - A new exported interface, e.g.
    ```
    // StackTraceError captures a full stack trace and runtime environment details.
    type StackTraceError interface {
        Stack() string         // full debug.Stack() output
        Runtime() string       // cached OS/arch/Go-version block
        error
    }
    ```
    *(Decide between an interface and a concrete exported struct in Trade-offs; the
    interface form mirrors the existing `TraceError`/`HttpError` style.)*
  - Constructors paralleling Task 1:
    - `NewStackTraceError() StackTraceError` — full stack, no message, no cause
      (the heavy analogue of `NewTraceError()`).
    - `NewStackTraceErrorf(format string, args ...any) StackTraceError`
    - `WrapStackTraceError(cause error, format string, args ...any) StackTraceError`
      with `Unwrap() error`.

  Runtime details are **captured once** at package init (or via a `sync.Once`-guarded
  package var) from `runtime.GOOS`, `runtime.GOARCH`, and `runtime.Version()` —
  Assumption 7. They do not change during a process, so per-error recomputation is
  pure waste; the consumer already cached them in `init()`.

- **Acceptance Criteria:**
  - [ ] `StackTraceError` interface (or exported struct) exists with godoc; `Stack()`
        returns the full multi-line trace from `internal.StackTrace()`, not the
        single-line `LineTrace` form.
  - [ ] `Runtime()` returns a string containing `runtime.GOOS`, `runtime.GOARCH`, and
        `runtime.Version()`, and the value is **computed once** (assert via test that
        two errors share the identical runtime string, and ideally that the value is a
        package-level constant/var, not freshly built per call).
  - [ ] `var _ StackTraceError = (*…)(nil)` compile-time assertion present.
  - [ ] `NewStackTraceError()`, `NewStackTraceErrorf`, `WrapStackTraceError` exist,
        exported, godoc'd.
  - [ ] `errors.Unwrap` / `errors.Is` / `errors.As` traverse to the wrapped cause for
        the wrapping constructors; `Unwrap()` returns `nil` for the non-wrapping ones.
  - [ ] `internal/stacktrace.go` is **unchanged**.

- **Pitfalls & edge cases:**
  - **`debug.Stack()` is large and grows with goroutine depth.** Capturing it on every
    error is heavier than `LineTrace`; document that `StackTraceError` is the
    "expensive, full-context" type and `TraceError` is the cheap one. Don't make
    `StackTraceError` the default anywhere.
  - **Stack contains library frames.** `internal.StackTrace()` returns the raw
    `debug.Stack()` including the `internal` and `loginjector` frames at the top. That's
    acceptable for a *full* stack (unlike `LineTrace`, we are not trying to point at one
    caller frame), but tests should assert the trace contains the **test function name**
    so we know it captured the caller's stack, not just library noise.
  - **`init()` ordering / `sync.Once`.** If you use a package-level `var x = buildRuntime()`
    it runs at init with no ordering hazard (no cross-package init dependency here). A
    `sync.Once` is also fine and lazier. Pick one; don't do both.
  - **Multiline `Error()` formatting.** A full stack inside `Error()` produces a huge,
    newline-laden string. Decide whether `Error()` includes the full stack or just the
    message (+ a hint to call `Stack()`); a giant `Error()` string can wreck log lines
    and `errors.Join` output. Recommend: `Error()` = message (+ cause), `Stack()` =
    the heavy payload on demand. State this in the godoc.

- **Complexity:** Medium

### Task 3: Add `PublicError` — public/internal message split with `Unwrap`

- **Description (what / why / how):**
  Add the generalizable "what you show the client vs. what you log" type. This is the
  most broadly useful of the consumer types and the strongest candidate for library
  inclusion.

  Surface:
  ```
  // PublicError separates a user-safe public message from the internal cause.
  // Error() returns the internal detail for logging; PublicMessage() returns the
  // text that is safe to expose to clients.
  type PublicError interface {
      PublicMessage() string
      error
  }

  // NewPublicError builds a PublicError from a client-safe message and an internal cause.
  func NewPublicError(public string, cause error) PublicError
  ```
  - `Error()` exposes the **internal** detail (for logs) — typically `cause.Error()`,
    or the public message if `cause` is nil.
  - `PublicMessage()` returns **only** the safe text.
  - `Unwrap() error` returns `cause` so `errors.Is`/`errors.As` reach the real error
    (e.g. you can `errors.As` down to an `HttpError` underneath a `PublicError`).

  Consider an additional convenience that composes with HTTP (Task 4): a `PublicError`
  whose cause is an `HttpError` lets a handler pick the status from the cause and the
  body from `PublicMessage()`. Keep that as documentation/usage, not extra surface,
  unless trivially clean.

- **Acceptance Criteria:**
  - [ ] `PublicError` interface + `NewPublicError(public string, cause error)` exist,
        exported, godoc'd.
  - [ ] `PublicMessage()` returns exactly the public string and **never** leaks the
        internal cause's text.
  - [ ] `Error()` (for logging) **does** include the internal cause detail.
  - [ ] `var _ PublicError = (*…)(nil)` compile-time assertion present.
  - [ ] `errors.Unwrap(NewPublicError("x", cause)) == cause`; `errors.Is`/`errors.As`
        traverse to `cause`.
  - [ ] A test proves the public and internal messages are distinct and that a secret
        token placed in the internal cause does **not** appear in `PublicMessage()`
        (mirrors the `KEY_1/2/3` redaction-style assertions in the existing tests).

- **Pitfalls & edge cases:**
  - **Leakage is the whole point.** The single most important test asserts the internal
    text (a fake secret) is absent from `PublicMessage()`. Get this wrong and the type
    is worse than useless.
  - **Nil cause.** `NewPublicError("msg", nil)` must not panic; `Error()` should fall
    back to the public message, and `Unwrap()` returns `nil`.
  - **Empty public message.** Decide behavior — probably allow it (caller's choice) but
    don't accidentally make `PublicMessage()` return the internal text as a "fallback",
    which would re-leak. Empty stays empty.
  - **`errors.Is` precedence.** Because `Unwrap` points at the cause, `errors.Is(pubErr,
    someSentinel)` works through it — good. Just ensure you don't also define a
    surprising `Is` method.

- **Complexity:** Easy

### Task 4: Reconcile `HttpError` — add a cause-carrying constructor (no duplication)

- **Description (what / why / how):**
  The consumer `HttpCodeError` is essentially `HttpError` plus an underlying error. Do
  **not** duplicate the type. Instead add a new constructor that produces a value
  satisfying the **existing** `HttpError` interface but also implementing `Unwrap`:
  - `WrapHttpError(code int, cause error) HttpError` — `StatusCode()` returns `code`,
    `Error()` returns `http.StatusText(code)` plus the cause detail (for logs), and
    `Unwrap() error` returns `cause`.

  Implement on a **new unexported struct** (e.g. `httpErrorWithCause`) so the existing
  `httpError` (`error.go:43`) and `NewHttpError` stay untouched. Because the return
  type is the existing `HttpError` interface, every current `errors.As(err, &httpErr)`
  site (see `TestHttpError`) keeps working unchanged, and now also gets `errors.Is`
  reachability to the cause.

  Optionally document the `PublicError`-over-`HttpError` composition from Task 3 as the
  recommended pattern for "client sees a clean message + status, logs see the cause".

- **Acceptance Criteria:**
  - [ ] `WrapHttpError(code int, cause error) HttpError` exists, exported, godoc'd.
  - [ ] `var _ HttpError = (*httpErrorWithCause)(nil)` compile-time assertion present.
  - [ ] `httpError`, `NewHttpError`, and the `HttpError` interface are **byte-for-byte
        unchanged**.
  - [ ] `StatusCode()` returns the supplied code; `Error()` contains
        `http.StatusText(code)` and the cause detail.
  - [ ] `errors.Unwrap(WrapHttpError(500, cause)) == cause`; `errors.As(err, &httpErr)`
        still finds the `HttpError`, and `errors.Is(err, cause)` is true.
  - [ ] The existing `TestHttpError` still passes verbatim.

- **Pitfalls & edge cases:**
  - **Nil cause.** `WrapHttpError(code, nil)` must behave like `NewHttpError(code)` for
    `Error()` (just the status text, no trailing separator) and `Unwrap()` returns nil.
  - **Status text for unknown codes.** `http.StatusText` returns `""` for unknown codes;
    `Error()` must not become an empty/garbage string when the code is bogus and the
    cause is also nil. Decide a sane fallback (e.g. include the numeric code).
  - **Don't widen the interface.** It's tempting to add a `Cause()` accessor to
    `HttpError`; that would break external implementers. Use `Unwrap` (free via
    `errors` package) instead of a new interface method.

- **Complexity:** Easy

## Risks

- **Frame-depth fragility (R1, high-impact/medium-likelihood).** `internal.LineTrace()`
  resolves the caller by counting frames from the `internal/stacktrace.go` anchor. Any
  helper indirection in the new trace constructors shifts the reported line to the wrong
  frame. *Mitigation:* call `internal.LineTrace()` directly inside each exported
  constructor at the same depth as `NewTraceError`; assert file+function in tests.
- **Accidental old-behavior change (R2, high-impact/low-likelihood).** Refactoring
  `error.go` while adding types could perturb `httpError`/`traceError` `Error()` output
  and break `TestHttpError`/`TestFullTrace` (and external callers). *Mitigation:* leave
  the existing structs/constructors/interfaces byte-for-byte unchanged (explicit AC in
  Tasks 1 & 4); confirm with `git diff`.
- **Public-message leakage (R3, high-impact/medium-likelihood).** A sloppy `PublicError`
  that falls back to the cause's text in `PublicMessage()` re-leaks internals — the
  exact failure the type is meant to prevent. *Mitigation:* dedicated leakage test
  (Task 5); empty public message stays empty, never falls back.
- **`debug.Stack()` cost / log bloat (R4, medium-impact/medium-likelihood).** Full
  stacks are large; if `StackTraceError.Error()` embeds the whole stack it bloats every
  log line and `errors.Join` output. *Mitigation:* `Error()` = message+cause only;
  full stack via `Stack()` on demand; document it.
- **Interface-growth temptation (R5, high-impact/low-likelihood).** Adding `Cause()`/
  `Unwrap()` *to the existing interfaces* would break external implementers. *Mitigation:*
  new types + the free `Unwrap` convention only; never edit `TraceError`/`HttpError`.
- **Scope creep into app-specific types (R6, low-impact).** Pressure to also ship
  `PaywallError`/`ErrNotFound`. *Mitigation:* explicitly excluded (Trade-offs).

## Trade-offs

- **New types/constructors over editing existing interfaces (chosen).** Go interfaces
  are satisfied structurally; adding a method to `TraceError`/`HttpError` would silently
  break any external type implementing them, and changing `NewTraceError()`'s signature
  breaks every caller. The whole point of the redesign is to *stop* consumers from
  forking, so a breaking release would be self-defeating. We pay a small surface-area
  cost (more constructors: `NewTraceErrorf`, `WrapTraceError`, `WrapHttpError`, the
  `StackTrace*` family, `NewPublicError`) to keep the release a clean **minor**. Worth it.
- **`Unwrap` convention over a `Cause()` accessor (chosen).** Standard-library
  `errors.Is`/`errors.As`/`errors.Unwrap` already traverse via `Unwrap() error` with no
  new interface method. A `Cause()` method would either widen the existing interface
  (breaking) or fragment the API. `Unwrap` is idiomatic and free.
- **Runtime details cached once vs per-error (chosen: cached).** OS/arch/Go-version are
  invariant for the process (Assumption 7); recomputing per error is pure overhead. The
  consumer already cached them in `init()`. Trade-off: a package-level var/`sync.Once`
  is process-global state, but it's immutable, so there's no concurrency hazard. We lose
  nothing and skip per-error `runtime.*` calls.
- **`StackTraceError.Error()` = message only, full stack on demand (chosen).** Putting
  the full `debug.Stack()` in `Error()` makes every log line and joined-error string
  enormous. Keeping `Error()` lean and exposing the heavy payload through `Stack()` keeps
  the type usable in normal logging while still carrying full context when explicitly
  asked. Cost: callers must know to call `Stack()` — documented in godoc.
- **Reconcile `HttpError` (add `WrapHttpError`) vs. duplicate as `HttpCodeError`
  (chosen: reconcile).** Shipping a second near-identical HTTP error type would split
  the API and confuse `errors.As` targets (which interface do you assert?). Adding a
  cause-carrying constructor that returns the *existing* `HttpError` interface keeps a
  single target type and a single mental model. Cost: the new variant carries an
  `Unwrap` that the old one lacks — acceptable asymmetry, documented.
- **`StackTraceError` as interface vs. concrete struct (chosen: interface).** Mirrors
  the existing `TraceError`/`HttpError` house style and lets `Stack()`/`Runtime()` be
  the documented contract without exposing fields. Cost: slightly more boilerplate than
  an exported struct; consistency wins.
- **Excluding `PaywallError`/`ErrNotFound` (chosen).** These encode *application
  business rules* (a paywall policy, a domain "not found" sentinel), not generic
  logging/error-shape concerns. `loginjector` is a dependency-light **logging library**;
  baking in domain semantics would (a) impose one app's vocabulary on every consumer and
  (b) bloat the surface with types most consumers don't want. `ErrNotFound` in
  particular is a one-line `errors.New` sentinel any app should own. Consumers keep these
  locally; the library only absorbs the *generalizable* shapes (trace, full-stack,
  public/internal split, http-with-cause).
