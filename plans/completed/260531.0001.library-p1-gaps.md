# Library P1 gaps: panic-recover middleware, console-vs-file printer split, opt-in runtime details

> **Completed 2026-05-31.** Three additive features shipped — `httptap.NewRecoverHandler`
> (Gap A), `LeveledHandler` interface + `WithMinLevel` helper + `WithPrinterMinLevel`
> option on `NewFileLogger` (Gap B), and `SetRuntimeDetailsProvider` process-wide hook
> with panic-safe fallback (Gap C). Pipeline: architect plan → engineer implementation
> → three-lens parallel review (correctness / security / performance) → targeted fix
> pass for P1 + P2 findings → green verification. Zero new module dependencies. All
> existing public signatures and tests preserved byte-for-byte.
>
> Notable review-driven hardening:
> - Gap C: provider panic inside `sync.Once.Do` is now recovered (fallback to base
>   `os/arch/go` descriptor) so a misbehaving provider cannot kill `Runtime()` for
>   the process lifetime. `runtimeDetails.once` became `*sync.Once` so the test-only
>   reset helper performs an atomic-pointer swap under the existing mutex rather
>   than the undefined-behavior value reassignment.
> - Gap A: `flushTo` no longer emits a spurious `WriteHeader(200)` for handlers that
>   wrote nothing — net/http's natural implicit-200 path is preserved. New
>   `panic_after_write_and_flush_buffered_body_not_on_wire` subtest pins the
>   buffering contract under the Flush+panic edge case.
> - Gap B: `WithPrinterMinLevel` no longer silently no-ops when the requested
>   printer level equals the logger's `minLevel` — the `WithMinLevel` wrap is now
>   applied unconditionally when the option is set.
> - All `httptap` option tests folded into `TestNewRecoverHandler` as subtests per
>   CLAUDE.md convention.

## Overview

Three independent P1 gaps were surfaced by auditing seven downstream consumers under
`~/Projects/seilbek/` and `~/Projects/developments/`. In each gap, multiple consumers
reinvented the same wheel because the library does not ship it. None of the three gaps
share code — they touch three different files and can be sequenced or parallelised at
the engineer's discretion.

**Gap A — `httptap.NewRecoverHandler(logger, level, next, opts...)`** (six consumer
copies). A `defer recover()` middleware that recovers any panic, logs it with a stack
trace at a configurable level, and responds with HTTP 500 + a configurable JSON body.
Lives in the existing `httptap/` sub-package because it composes with
`NewPayloadHandler` and `NewAccessHandler` and matches their style.

**Gap B — `WithPrinterMinLevel(level)` option on `NewFileLogger`, backed by a new
`LeveledHandler` interface in `Logger` core** (five consumer copies). Consumers ship
a `NewLogger(folder, name, printerMinLevel)` where the *console printer* has a
separate threshold from the *file* handler. Today's `NewFileLogger` gates both with
a single `minLevel`. The fix extends `Logger` with optional per-handler thresholds
via a new `LeveledHandler` interface and a `WithMinLevel(level, handler) io.Writer`
helper; `NewFileLogger` then composes the option by wrapping its console printer in
`WithMinLevel`. The existing `NewLogger`/`NewFileLogger` signatures and behaviour are
untouched — both `LeveledHandler` and `WithMinLevel` are purely additive.

**Gap C — opt-in runtime-details provider hook on `StackTraceError`** (two consumer
copies, byte-identical). Two consumers (`fx_rate_monitor`, `com_lingocrm_api`) ship a
`StackTraceError` whose `OSDetails()` returns gopsutil-sourced CPU/RAM plus PID/PPID
text. Loginjector's `StackTraceError.Runtime()` ships only `os/arch/go`. The hard
constraint is "no gopsutil in `go.mod`" (CLAUDE.md mandates dependency-light), so the
fix is a *hook* that the library owns and a `runtimex` provider that lives elsewhere
(out of scope for this plan). After comparing the two design alternatives, this plan
recommends a **process-wide `SetRuntimeDetailsProvider`** to match the existing
`cachedRuntime()` precedent. Argued in Task 5.

## Assumptions

- Library stays pure Go, `CGO_ENABLED=0`, no new direct dependencies. The only direct
  deps remain `stretchr/testify` (tests) and `twinj/uuid` (hook IDs).
- Every existing public symbol (`NewFileLogger`, `NewStackTraceError`,
  `NewStackTraceErrorf`, `WrapStackTraceError`, `cachedRuntime` semantics, every
  `httptap` constructor) keeps its current signature and current behaviour
  byte-for-byte. The whole point is **additive** for every gap.
- The `runtimex` sub-package that hosts the gopsutil-sourced provider implementation
  is **out of scope**. This plan only adds the hook in `error.go`. The follow-up plan
  will add the provider package and wire it up.
- Gap A's recover middleware reuses existing root-package error types
  (`loginjector.NewStackTraceError`, `loginjector.NewHttpError`,
  `loginjector.NewPublicError`) — no fresh `debug.Stack()` call. The middleware writes
  the response body in JSON; the body shape and `Content-Type` are configurable.
- The existing `httptap` optional-interface wrapper (`wrapWithOptionalInterfaces` in
  `httptap/rw.go`, the eight `*RW` composite types, the `hijackMarker` marker
  interface) is used as-is for Gap A. The recover handler buffers the response body
  before forwarding, so SSE/WebSocket/HTTP-2-push handlers downstream **lose
  streaming semantics** under the recover wrapper — that is acceptable for a recover
  middleware and documented in godoc. Consumers wanting both streaming and recover
  should wrap their non-streaming routes only.
- "post-`WriteHeader` panic" semantics: once any byte of the response status line has
  been written to the wire, the middleware **cannot** rewrite it. The plan handles
  this by buffering the response in memory and flushing only after `next` returns
  cleanly; if `next` panics before the buffered flush, no bytes have hit the wire and
  the recover middleware writes the 500 response. This is a deliberate trade-off
  (memory cost vs. correctness), spelled out in Task 1 and the Trade-offs section.
- The seven consumer audit was performed by the user; this plan does not re-audit. The
  shapes encoded in Task acceptance criteria are taken from the user's description and
  cross-checked against `error.go` / `handler.go` / `httptap/options.go` to ensure
  they fit the existing style.
- Tests follow the established convention: one `TestType_Method` function per
  exported method with `t.Run("scenario", …)` subtests; concurrency-sensitive paths
  carry a `…_ForRaceCondition` subtest run under `CGO_ENABLED=0 go test -race`;
  compile-time `var _ Iface = (*mock)(nil)` assertions guard mocks.

## Tasks

### Task 1: Implement `httptap.NewRecoverHandler` core (panic recovery + JSON 500 response)

- **Description:** Add a new file `httptap/recover.go` containing the new exported
  constructor. The handler composes with `NewPayloadHandler` / `NewAccessHandler` —
  it takes an `http.HandlerFunc` and returns an `http.HandlerFunc`, so callers can
  chain `recover( payload( access( next ) ) )` (or any other order) trivially.

  Signature:
  ```go
  func NewRecoverHandler(
      logger *loginjector.Logger,
      level loginjector.LogLevel,
      next http.HandlerFunc,
      opts ...RecoverOption,
  ) (http.HandlerFunc, error)
  ```

  Behaviour (zero-options baseline, before Task 2 wires the options DSL):
  - Validate `logger != nil` and `next != nil`; return `(nil, fmt.Errorf("logger is
    nil"))` / `(nil, fmt.Errorf("next is nil"))` to match the styling of
    `NewPayloadHandlerWithOptions` (`httptap/payload.go:121`).
  - Per request:
    1. Wrap the response writer in a buffering wrapper, `recoverResponseWriter`,
       that captures `statusCode`, header map, body bytes, and a `wroteHeader` flag.
       The wrapper does *not* forward `WriteHeader` / `Write` to the underlying
       writer immediately; it accumulates into a per-request `bytes.Buffer` and a
       cloned `http.Header`. **This is the only way to "swap" the response when
       `next` panics after calling `WriteHeader`.**
    2. Pass the buffering wrapper through `wrapWithOptionalInterfaces` so downstream
       handlers still see `http.Flusher` / `http.Hijacker` / `http.Pusher` when the
       underlying writer has them. **Caveat:** the buffering wrapper makes `Flush()`
       a no-op (nothing to flush — bytes are buffered, not in `os.Stdout`), and
       hijacking through it works but means the buffered bytes are discarded. Both
       are documented and the godoc tells the caller not to use the recover wrapper
       around SSE / WebSocket routes.
    3. `defer recover()` block:
       - If `recover()` returns nil and the connection was not hijacked, copy the
         buffered headers + status + body to the real `http.ResponseWriter`. Done.
       - If `recover()` returns nil and the connection *was* hijacked
         (`recoverResponseWriter.hijacked == true`), skip the copy — the caller owns
         the conn.
       - If `recover()` returns a non-nil value:
         a. Discard the buffered body (it is now garbage from the panicked handler).
         b. Build the trace error:
            ```go
            ste := loginjector.NewStackTraceError()
            // join: ste (for the stack + runtime), an HttpError(500), a PublicError,
            // and a synthetic "panic: <value>" via fmt.Errorf so the panic value text
            // shows up in the log line.
            joined := errors.Join(
                fmt.Errorf("panic: %v", panicValue),
                ste,
                loginjector.NewHttpError(http.StatusInternalServerError),
                loginjector.NewPublicError(cfg.publicMessage, nil),
            )
            ```
         c. Log it: `logger.WriteLog(level, []byte(joined.Error() + "\n" +
            ste.Stack()))`. The stack goes on a second line so single-line log
            consumers still get the joined message on line one. `errors.Join`'s
            default `Error()` already puts each joined error on its own line, so the
            full payload is greppable. Log errors from `WriteLog` are swallowed via
            `println(err.Error())` to match the existing payload/access handler
            style (we do not have a better sink — the logger *is* the sink).
         d. If `recoverResponseWriter.wroteHeader` is true (panic-after-WriteHeader):
            we cannot rewrite the status because we have not actually written it to
            the wire yet — it is still in the buffer. So we **can** safely write the
            500 response, because nothing has hit the wire. Reset the buffered state
            and write the 500.
         e. If `recoverResponseWriter.hijacked` is true (panic-after-Hijack): we
            cannot respond at all. The caller owns the conn; the panic is logged
            and we return. Document this loudly.
         f. Otherwise (no Hijack), write the 500 response to the real
            `http.ResponseWriter` directly (bypassing the buffer wrapper, which is
            already discarded): set `Content-Type` from `cfg.contentType`
            (default `application/json; charset=utf-8`), set `X-Content-Type-Options:
            nosniff` (defensive), `WriteHeader(500)`, write `cfg.fallbackBody`.
       - If `next` panicked with `http.ErrAbortHandler` (the stdlib's sentinel for
         "abort without logging"), re-panic immediately after the conn cleanup. Do
         NOT log it; do NOT write a 500 body. This matches `net/http`'s own server
         behaviour (`server.go` recovers `ErrAbortHandler` silently). Failing to
         do this would spam logs for every aborted client.
    4. Call `next(wrapped, r)`. The deferred recover-and-flush runs after.

  Default config baked into this task (overridable by Task 2):
  - `cfg.publicMessage = "something went wrong"`
  - `cfg.contentType = "application/json; charset=utf-8"`
  - `cfg.fallbackBody = []byte(`{"error":"something went wrong"}`)`

  Concrete struct shape for the buffering wrapper (at the bottom of the file, per
  the "public surface on top, internals at bottom" memory rule):
  ```go
  type recoverResponseWriter struct {
      http.ResponseWriter
      headers     http.Header
      body        bytes.Buffer
      statusCode  int
      wroteHeader bool
      hijacked    bool
  }
  ```
  - `Header() http.Header` — lazily clone the real writer's headers on first call,
    then return the local map. (Same trick `httptest.ResponseRecorder` uses.)
  - `WriteHeader(code int)` — first call wins, records `code` and `wroteHeader =
    true`. Subsequent calls are no-ops.
  - `Write(b []byte) (int, error)` — if `!wroteHeader`, calls
    `WriteHeader(http.StatusOK)` first; then appends `b` to the body buffer; returns
    `(len(b), nil)`.
  - `markHijacked()` — sets `hijacked = true`. Satisfies the existing
    `hijackMarker` interface in `httptap/rw.go:13` so `wrapWithOptionalInterfaces`
    accepts it.
  - Compile-time assertions:
    ```go
    var _ http.ResponseWriter = (*recoverResponseWriter)(nil)
    var _ hijackMarker         = (*recoverResponseWriter)(nil)
    ```
  - Helper `(rrw *recoverResponseWriter) flushTo(w http.ResponseWriter) error` —
    used on the no-panic path to copy headers, status, and body to the real writer.
    Handles the "implicit 200" case: if `!wroteHeader`, call
    `w.WriteHeader(http.StatusOK)` before `w.Write(rrw.body.Bytes())`.

  Godoc for `NewRecoverHandler` (verbatim — keep it short, no marketing):
  ```go
  // NewRecoverHandler returns an HTTP middleware that recovers any panic in next,
  // logs the panic with a stack trace via logger at level, and responds with HTTP
  // 500 plus a JSON body.
  //
  // Behaviour:
  //   - On a clean return from next, the response is flushed to the underlying
  //     ResponseWriter as if no middleware were present.
  //   - On a panic, the buffered response is discarded and a fresh 500 response
  //     is written with Content-Type "application/json; charset=utf-8" and body
  //     {"error":"something went wrong"}. Both are overridable via options.
  //   - http.ErrAbortHandler is re-panicked silently, matching net/http's own
  //     server recovery. No log line, no 500 body.
  //   - If next hijacks the connection before panicking, the panic is logged but
  //     no response can be written — the caller owns the conn.
  //
  // The handler buffers the entire response body in memory before forwarding it.
  // This is necessary so a panic AFTER a WriteHeader call can still be replaced
  // with a 500 response. Do not place this middleware in front of SSE, WebSocket,
  // or HTTP/2-push handlers: streaming is lost. Wrap only routes that produce
  // bounded response bodies.
  //
  // The stack trace logged uses loginjector.NewStackTraceError(); it contains
  // absolute build-host file paths and MUST NOT be returned to clients. Only the
  // configured fallback body reaches the client.
  ```

- **Pitfalls & edge cases:**
  - **Buffer-the-response trade-off.** This is the whole crux. If we forwarded
    `WriteHeader` to the real writer eagerly, a post-`WriteHeader` panic would leave
    the client with a `200 OK` followed by garbage — the worst possible outcome
    (browser shows half a page, or hangs). Buffering costs memory and breaks
    streaming. The godoc tells the caller. The engineer MUST NOT "optimise" by
    forwarding eagerly — doing so silently re-introduces the bug.
  - **`http.ErrAbortHandler` is `==`-compared (sentinel value), not type-asserted.**
    Stdlib does `if err == http.ErrAbortHandler` in `server.go`. Match that.
  - **`recover()` returns `interface{}`, which may be an error, a string, or
    anything.** `fmt.Errorf("panic: %v", panicValue)` is the right way to render it
    — `%v` handles all concrete types including `nil`-ish wrappers without a type
    assertion crash.
  - **Re-panic via `panic(panicValue)`, not `panic(err)`.** Re-wrapping
    `ErrAbortHandler` in a new error breaks the stdlib's `==` check upstream and
    the server will then log it instead of silently dropping it. Plain re-panic
    only.
  - **The recover wrapper claims the buffer in `defer`. `next` may launch a
    goroutine that writes to the original `w` after `next` returns.** Out of scope:
    that is a misuse, the stdlib http server has the same limitation, and the
    middleware cannot fix it. Do NOT try to handle this — leave the buffer logic
    single-goroutine.
  - **`Header()` lazy-clone.** Calling `Header()` on the buffer wrapper must NOT
    mutate the underlying writer's header map. Clone on first call. After flush,
    the cloned headers overwrite the underlying writer's; this matches what a
    direct write would have done.
  - **Hijacker pass-through path.** The composite `*RW` from
    `wrapWithOptionalInterfaces` calls `base.markHijacked()` before delegating to
    the underlying `Hijack()`. After hijack, the buffer is dead — we set
    `hijacked = true`, do not flush, do not write 500 on later panic. The
    `markHijacked()` method on `recoverResponseWriter` is the only thing wiring
    this together; do not forget it.
  - **`logger.WriteLog` is non-blocking with respect to the request finishing
    only when there is more than one sink** — see `logger.go:155`. With a single
    handler the write is inline. The recover middleware's deferred block runs
    *before* the request completes from `net/http`'s perspective (it's inside the
    handler goroutine), so a slow log sink blocks the response by the time it
    takes to log. This is acceptable for a recover path (panics are rare); if a
    caller wants async logging they can attach a buffered/async writer at the
    `Logger` layer.

- **Complexity:** Hard

- **Code Example:**
  ```go
  package httptap

  import (
      "bytes"
      "errors"
      "fmt"
      "net/http"

      loginjector "github.com/prorochestvo/loginjector"
  )

  type recoverConfig struct {
      publicMessage string
      contentType   string
      fallbackBody  []byte
  }

  func defaultRecoverConfig() recoverConfig {
      return recoverConfig{
          publicMessage: "something went wrong",
          contentType:   "application/json; charset=utf-8",
          fallbackBody:  []byte(`{"error":"something went wrong"}`),
      }
  }

  // RecoverOption is the functional-options type; constructors land in Task 2.
  type RecoverOption func(*recoverConfig)

  func NewRecoverHandler(
      logger *loginjector.Logger,
      level loginjector.LogLevel,
      next http.HandlerFunc,
      opts ...RecoverOption,
  ) (http.HandlerFunc, error) {
      if logger == nil { return nil, fmt.Errorf("logger is nil") }
      if next == nil   { return nil, fmt.Errorf("next is nil") }

      cfg := defaultRecoverConfig()
      for _, o := range opts { o(&cfg) }

      return func(w http.ResponseWriter, r *http.Request) {
          rrw := &recoverResponseWriter{ResponseWriter: w}
          wrapped := wrapWithOptionalInterfaces(rrw, w)

          defer func() {
              pv := recover()
              if pv != nil {
                  if pv == http.ErrAbortHandler {
                      panic(pv) // re-panic silently per stdlib convention
                  }
                  ste := loginjector.NewStackTraceError()
                  joined := errors.Join(
                      fmt.Errorf("panic: %v", pv),
                      ste,
                      loginjector.NewHttpError(http.StatusInternalServerError),
                      loginjector.NewPublicError(cfg.publicMessage, nil),
                  )
                  if _, err := logger.WriteLog(level, []byte(joined.Error()+"\n"+ste.Stack())); err != nil {
                      println(err.Error()) // matches existing httptap convention
                  }
                  if rrw.hijacked { return }
                  // buffer absorbed everything; the wire is still clean.
                  h := w.Header()
                  h.Set("Content-Type", cfg.contentType)
                  h.Set("X-Content-Type-Options", "nosniff")
                  w.WriteHeader(http.StatusInternalServerError)
                  _, _ = w.Write(cfg.fallbackBody)
                  return
              }
              // clean path: flush buffered response to real writer.
              if rrw.hijacked { return }
              if err := rrw.flushTo(w); err != nil {
                  println(err.Error())
              }
          }()

          next(wrapped, r)
      }, nil
  }

  // recoverResponseWriter buffers headers, status, and body so a panic after
  // WriteHeader can still be replaced with a 500 response.
  type recoverResponseWriter struct {
      http.ResponseWriter
      headers     http.Header
      body        bytes.Buffer
      statusCode  int
      wroteHeader bool
      hijacked    bool
  }

  func (rrw *recoverResponseWriter) Header() http.Header {
      if rrw.headers == nil { rrw.headers = rrw.ResponseWriter.Header().Clone() }
      return rrw.headers
  }

  func (rrw *recoverResponseWriter) WriteHeader(code int) {
      if rrw.wroteHeader { return }
      rrw.statusCode = code
      rrw.wroteHeader = true
  }

  func (rrw *recoverResponseWriter) Write(b []byte) (int, error) {
      if !rrw.wroteHeader { rrw.WriteHeader(http.StatusOK) }
      return rrw.body.Write(b)
  }

  func (rrw *recoverResponseWriter) markHijacked() { rrw.hijacked = true }

  func (rrw *recoverResponseWriter) flushTo(w http.ResponseWriter) error {
      if rrw.headers != nil {
          for k, v := range rrw.headers {
              w.Header()[k] = v
          }
      }
      if rrw.wroteHeader {
          w.WriteHeader(rrw.statusCode)
      } else {
          w.WriteHeader(http.StatusOK)
      }
      _, err := w.Write(rrw.body.Bytes())
      return err
  }

  var _ http.ResponseWriter = (*recoverResponseWriter)(nil)
  var _ hijackMarker         = (*recoverResponseWriter)(nil)
  ```

### Task 2: Add `RecoverOption` constructors (options DSL)

- **Description:** Add the option constructors in the same file (`httptap/recover.go`).
  Follow the existing payload-options style verbatim — see `httptap/options.go:53-95`
  for `WithMaxRequestBody`, `WithRedactHeaders`, `WithSummaryWriter` for the pattern.

  Constructors to add:
  ```go
  // WithFallbackResponse overrides the response body written on panic recovery.
  // The body is sent verbatim. Default is `{"error":"something went wrong"}`.
  // Pair with WithFallbackContentType if the body is not JSON.
  func WithFallbackResponse(body []byte) RecoverOption {
      return func(c *recoverConfig) { c.fallbackBody = body }
  }

  // WithFallbackContentType overrides the Content-Type header set on the panic
  // response. Default is "application/json; charset=utf-8".
  func WithFallbackContentType(ct string) RecoverOption {
      return func(c *recoverConfig) { c.contentType = ct }
  }

  // WithPublicMessage overrides the message embedded in the PublicError that is
  // joined into the logged error chain. It does NOT change the response body —
  // use WithFallbackResponse for that. Default is "something went wrong".
  func WithPublicMessage(msg string) RecoverOption {
      return func(c *recoverConfig) { c.publicMessage = msg }
  }
  ```

  **Three options is the entire surface.** Resist scope creep. The user's brief
  flagged "design conservatively, don't gold-plate". An IP-extracting option, a
  request-ID option, a custom logger formatter — all explicitly out of scope. Any
  consumer wanting more wraps `NewRecoverHandler` themselves.

  Per-option docstrings follow the same style as `httptap/options.go`: a single
  paragraph stating the effect, the default, and any cross-option interaction.

- **Acceptance Criteria:**
  - [ ] Three exported option constructors: `WithFallbackResponse`,
    `WithFallbackContentType`, `WithPublicMessage`. No others.
  - [ ] Each option mutates exactly one field on `recoverConfig` and returns a
    `RecoverOption`.
  - [ ] Default values match Task 1 baseline: body =
    `{"error":"something went wrong"}`, content-type =
    `application/json; charset=utf-8`, public message = `"something went wrong"`.
  - [ ] Passing the same option twice: last write wins (free with the functional
    options pattern; no extra logic needed). Test this in Task 3.
  - [ ] `go vet ./...`, `gofmt -l .` clean.

- **Pitfalls & edge cases:**
  - **`WithFallbackResponse([]byte{})`** (empty body) must be allowed — some
    consumers want a bare 500 with no body. Do NOT add a "non-empty" guard.
  - **`WithFallbackContentType("")`** sets `Content-Type` to the empty string,
    which the stdlib server will sniff. That is the caller's choice; do NOT
    silently substitute a default. The default applies only when the option is
    not passed at all.
  - **`WithPublicMessage("")`** is allowed — the joined error chain just gets an
    empty `PublicMessage()`. The response body is unaffected.
  - Resist adding `WithRecoverHandler(func(panicValue interface{}) []byte)`-style
    callbacks. Once a caller can compute the body from the panic value, you have
    leaked the panic value to the response and re-opened the exact information
    disclosure problem `PublicError` exists to prevent. If a future caller really
    needs this, they can wrap `NewRecoverHandler` and intercept their own panic.

- **Complexity:** Easy

### Task 4: Add `LeveledHandler` interface, `WithMinLevel` helper, and `WithPrinterMinLevel` option

- **Description:** Extend `Logger` core with per-handler minimum levels via a new
  optional interface, then expose a `WithPrinterMinLevel` option on `NewFileLogger`
  that composes on top of it. Today `NewFileLogger(folder, prefix, minLevel,
  opts...)` (see `filelogger.go:79`) gates *every* handler — file rotation AND the
  timestamped stdout printer — at `minLevel`. Five consumer projects ship a
  `NewLogger(folder, name, printerMinLevel)` that wires up a separate threshold for
  the printer so the console shows only important messages while the file captures
  everything.

  This task makes the change in three layers, all additive:

  **Layer 1 — `LeveledHandler` interface (`logger.go`).** Declare a new optional
  interface near the existing `Logger` / `LogLevel` declarations (top of the file,
  per the "public surface on top" memory rule):

  ```go
  // LeveledHandler is an optional interface a handler may implement to declare a
  // minimum log level distinct from the parent Logger's minimumLogLevel. When a
  // handler implements LeveledHandler, Logger.WriteLog uses its MinLevel() instead
  // of Logger.minimumLogLevel for the per-handler gate. Handlers that do not
  // implement LeveledHandler keep the historical behaviour — they fire when
  // level >= Logger.minimumLogLevel.
  //
  // MinLevel must be safe to call concurrently and must return the same value
  // for the lifetime of the handler — Logger caches no result.
  type LeveledHandler interface {
      io.Writer
      MinLevel() LogLevel
  }
  ```

  **Layer 2 — `WithMinLevel` helper (`handler.go`).** Add a small wrapper that
  promotes any `io.Writer` to a `LeveledHandler`. Lives in `handler.go` next to the
  other handler factories, because it is a handler-shape utility:

  ```go
  // WithMinLevel wraps handler so that Logger treats it as a LeveledHandler with
  // the given minimum level. The returned writer forwards Write calls to handler
  // unchanged; the only added behaviour is the MinLevel() method that Logger.WriteLog
  // reads via interface assertion. Use this to give one handler a threshold
  // distinct from the parent Logger's minimumLogLevel — for example, to send only
  // high-severity messages to the console while sending everything to a file:
  //
  //   file    := loginjector.CyclicOverwritingFilesHandler(...)
  //   console := loginjector.WithMinLevel(levels.Warning, loginjector.TimestampedPrintHandler())
  //   logger, _ := loginjector.NewLogger(levels.Info, file, console)
  //
  // The returned writer is itself thread-safe via Logger's existing ensureThreadSafe
  // wrapping at registration. Calling WithMinLevel on an already-leveled writer
  // replaces the inner MinLevel().
  func WithMinLevel(level LogLevel, handler io.Writer) io.Writer {
      return &leveledWriter{inner: handler, level: level}
  }

  // leveledWriter is the concrete LeveledHandler implementation behind WithMinLevel.
  type leveledWriter struct {
      inner io.Writer
      level LogLevel
  }

  func (lw *leveledWriter) Write(p []byte) (int, error) { return lw.inner.Write(p) }
  func (lw *leveledWriter) MinLevel() LogLevel          { return lw.level }

  var _ LeveledHandler = (*leveledWriter)(nil)
  ```

  **Layer 3 — `Logger.WriteLog` per-handler gate (`logger.go`).** Today's gate is a
  single boolean `handlersActive := level >= l.minimumLogLevel` (`logger.go:147`),
  used as one switch over the full handler slice. Replace with a per-handler check:

  ```go
  // before (logger.go:146-153):
  // handlers fire only at or above the minimum level.
  handlersActive := level >= l.minimumLogLevel
  if handlersActive {
      sinks = append(sinks, l.handlers...)
  } else if len(sinks) == 0 {
      // below minimum with no matching hooks: nothing to do.
      return 0, nil
  }

  // after:
  // handlers fire when level >= their per-handler threshold. The threshold is
  // h.MinLevel() if h implements LeveledHandler, else l.minimumLogLevel.
  anyHandlerActive := false
  for _, h := range l.handlers {
      threshold := l.minimumLogLevel
      if lh, ok := unwrapLeveled(h); ok {
          threshold = lh.MinLevel()
      }
      if level >= threshold {
          sinks = append(sinks, h)
          anyHandlerActive = true
      }
  }
  if !anyHandlerActive && len(sinks) == 0 {
      // below minimum with no matching hooks and no leveled handlers active.
      return 0, nil
  }
  ```

  The `unwrapLeveled` helper handles the interaction with `ensureThreadSafe`:
  `NewLogger` wraps every handler in a `*writer{ h: w.Write }` (`logger.go:36-41`),
  which would hide the `LeveledHandler` interface on the inner sink. Two reasonable
  fixes; the diff sketch above uses option (B), which is the simpler one to
  retrofit:

  - **(A) Check before wrapping.** `NewLogger` (and `Hook`, by symmetry) records
    the original handler's `MinLevel()` *before* `ensureThreadSafe` and stores it
    on the resulting `*writer` (extend `writer` with `level LogLevel; leveled bool`).
    `unwrapLeveled` then inspects the `*writer`. **Drawback:** mutates the `writer`
    struct shape, which is used by `Hook` too.

  - **(B, recommended) Propagate through the wrapper.** Extend `*writer` so it
    forwards `MinLevel()` if the inner sink implements `LeveledHandler`:

    ```go
    // in logger.go, near the writer struct:
    func (w *writer) MinLevel() LogLevel {
        if lh, ok := w.original.(LeveledHandler); ok {
            return lh.MinLevel()
        }
        return 0 // never read — unwrapLeveled checks the assertion first.
    }
    ```

    This requires `*writer` to retain a reference to the original `io.Writer`,
    not just `h: w.Write` (the current shape — see `logger.go:40`). Change
    `ensureThreadSafe` to:

    ```go
    func ensureThreadSafe(w io.Writer) io.Writer {
        if existing, ok := w.(*writer); ok {
            return existing
        }
        return &writer{original: w, h: w.Write}
    }
    ```

    And `unwrapLeveled` is then:

    ```go
    // unwrapLeveled returns the LeveledHandler view of h, peeling the
    // ensureThreadSafe *writer wrapper if present. Returns (nil, false) when h
    // is not leveled.
    func unwrapLeveled(h io.Writer) (LeveledHandler, bool) {
        if lh, ok := h.(LeveledHandler); ok {
            // both bare LeveledHandlers and *writer values land here, because
            // *writer.MinLevel() is defined above.
            return lh, true
        }
        return nil, false
    }
    ```

    The check works for both `*writer` (carrying a leveled inner sink) and a bare
    `LeveledHandler` passed in unwrapped (defensive: callers who pre-wrap their
    own thread-safe leveled sink). The `*writer.MinLevel()` only returns a real
    value when the inner sink is leveled; the assertion in `unwrapLeveled` could
    in theory match a non-leveled `*writer`, so add a small guard:

    ```go
    func unwrapLeveled(h io.Writer) (LeveledHandler, bool) {
        if w, ok := h.(*writer); ok {
            if lh, ok := w.original.(LeveledHandler); ok {
                return lh, true
            }
            return nil, false
        }
        if lh, ok := h.(LeveledHandler); ok {
            return lh, true
        }
        return nil, false
    }
    ```

    With this version, `*writer` no longer needs a `MinLevel()` method at all —
    `unwrapLeveled` peels the wrapper explicitly. Simpler. Go with this.

  **Layer 4 — `WithPrinterMinLevel` option (`filelogger.go`).** With Layers 1-3 in
  place, the option is a thin composition:

  ```go
  // WithPrinterMinLevel overrides the threshold at which the console printer
  // emits messages, decoupling it from the file-handler threshold (minLevel).
  // When not set, the printer fires at minLevel (current behaviour). When set,
  // the console printer fires at level >= this value while the file handler
  // continues to fire at level >= minLevel.
  //
  // Has no effect when combined with WithoutPrinter — WithoutPrinter wins.
  //
  // Implemented by wrapping the printer handler in WithMinLevel before adding
  // it to the Logger handler list; the per-handler gate in Logger.WriteLog
  // does the rest.
  func WithPrinterMinLevel(level LogLevel) FileLoggerOption {
      return func(c *fileLoggerConfig) {
          c.printerMinLevel = level
          c.printerMinLevelSet = true
      }
  }
  ```

  Plus, in the `NewFileLogger` body where the printer is appended today
  (`filelogger.go:122-124`):

  ```go
  if cfg.printer {
      p := TimestampedPrintHandler(cfg.printerOpts...)
      if cfg.printerMinLevelSet && cfg.printerMinLevel != minLevel {
          p = WithMinLevel(cfg.printerMinLevel, p)
      }
      handlers = append(handlers, p)
  }
  ```

  Fields added to `fileLoggerConfig` (`filelogger.go:140`):

  ```go
  printerMinLevel    LogLevel
  printerMinLevelSet bool       // disambiguates explicit-zero from unset
  ```

  When `printerMinLevelSet == false`, or when `printerMinLevel == minLevel`, the
  printer is appended without the `WithMinLevel` wrap and the legacy code path
  is byte-identical.

### Task 6: Add process-wide `SetRuntimeDetailsProvider` hook on `StackTraceError`

- **Description:** Modify `error.go` so that `StackTraceError.Runtime()` can be
  augmented by an opt-in process-wide provider function. The provider is set at
  startup; once set, its return value is **appended** to (not replacing) the
  existing `cachedRuntime()` string.

  **Recommended design (recap from Overview): process-wide hook.** The
  alternatives:

  1. **Provider function via option on each constructor**
     (`WithRuntimeDetailsProvider(fn) StackTraceErrorOption`). Cost: every
     `NewStackTraceError*` callsite must wire the provider. The consumer pattern
     observed in `fx_rate_monitor` and `com_lingocrm_api` is "every panic
     handler builds a `StackTraceError` and wants the gopsutil details"; making
     them remember to pass the option at every callsite is annoying.

  2. **Process-wide `SetRuntimeDetailsProvider(fn)`** (recommended). Set once at
     startup; every subsequent `NewStackTraceError*` call picks it up. Cost:
     global state — harder to test, must reset between test runs, racy if mutated
     after constructors run.

  Why (2) wins: the existing `cachedRuntime()` (`error.go:306`) is **already
  process-wide global state** with the same lifecycle pattern. The whole
  `runtimeDetails` struct is a `sync.Once`. Adding a sibling provider hook keeps
  the design symmetric. Testability is solved by a paired
  `resetRuntimeDetailsProvider()` test helper (unexported, lives in
  `error_test.go`). Mutation-after-construction risk is documented as "call
  `SetRuntimeDetailsProvider` once at startup, before any error is constructed";
  this matches how `log.SetOutput`, `runtime/debug.SetGCPercent`, and most
  stdlib process-wide hooks document themselves.

  Concrete additions to `error.go`:

  ```go
  // SetRuntimeDetailsProvider installs a process-wide provider that supplies
  // additional runtime detail text appended to every StackTraceError's
  // Runtime() output. The provider is called at most once per process; its
  // return value is cached for the lifetime of the process, mirroring the
  // caching behaviour of the built-in os/arch/Go-version runtime string.
  //
  // Call this at startup, before any StackTraceError is constructed. Calling
  // it after a StackTraceError has been constructed has NO effect, because the
  // runtime string is cached on first use (see Runtime() docs). Calling it
  // more than once: the last call before the cache is populated wins; calls
  // after the cache is populated are no-ops.
  //
  // The provider must be safe to call from any goroutine. Loginjector calls
  // it under the same sync.Once that guards the built-in runtime string, so
  // the provider sees no concurrent invocation; once cached, Runtime() never
  // calls it again.
  //
  // Pass nil to clear a previously-set provider (useful in tests).
  func SetRuntimeDetailsProvider(fn func() string) {
      runtimeDetailsExtraMu.Lock()
      runtimeDetailsExtraProvider = fn
      runtimeDetailsExtraMu.Unlock()
  }

  var (
      runtimeDetailsExtraMu       sync.Mutex
      runtimeDetailsExtraProvider func() string
  )
  ```

  Modify `cachedRuntime()` (`error.go:313`):
  ```go
  func cachedRuntime() string {
      runtimeDetails.once.Do(func() {
          base := fmt.Sprintf("os=%s arch=%s go=%s",
              runtime.GOOS, runtime.GOARCH, runtime.Version())
          // grab the provider under the mutex once, release before calling it.
          runtimeDetailsExtraMu.Lock()
          provider := runtimeDetailsExtraProvider
          runtimeDetailsExtraMu.Unlock()
          if provider != nil {
              extra := provider()
              if extra != "" {
                  base = base + " " + extra
              }
          }
          runtimeDetails.value = base
      })
      return runtimeDetails.value
  }
  ```

  **Critical:** the existing `Runtime()` godoc on `StackTraceError`
  (`error.go:67`) currently says "computed once on first use and cached for the
  process lifetime — every StackTraceError value shares the same string." This
  contract holds — the only change is *what* the once-computed value is. Update
  the godoc to explicitly mention the provider:
  ```
  // Runtime returns a string describing the OS, architecture, and Go version,
  // computed once on first use and cached for the process lifetime — every
  // StackTraceError value shares the same string. If
  // SetRuntimeDetailsProvider was called with a non-nil function before any
  // StackTraceError was constructed, the provider's output is appended to the
  // built-in os/arch/go descriptor.
  ```

  **Out of scope:** the `runtimex` sub-package that hosts `GopsutilProvider()` is
  a separate follow-up plan. This task just exposes the hook.

- **Acceptance Criteria:**
  - [ ] New exported function in `error.go`:
    `func SetRuntimeDetailsProvider(fn func() string)`. Godoc matches the
    verbatim block above.
  - [ ] New unexported package-level state: `runtimeDetailsExtraMu sync.Mutex`,
    `runtimeDetailsExtraProvider func() string`.
  - [ ] Modified `cachedRuntime()` reads the provider under the mutex inside the
    `sync.Once`, appends its non-empty output to the base string with a single
    space separator. Empty / nil provider: behaviour unchanged from current.
  - [ ] `StackTraceError.Runtime()` godoc updated to mention the provider.
  - [ ] When no provider is set: every existing test in `error_test.go` passes
    unchanged. The returned runtime string is byte-identical to the current
    output.
  - [ ] When a provider is set BEFORE any `NewStackTraceError*` call:
    `Runtime()` returns `"os=… arch=… go=… <provider-output>"`.
  - [ ] When `SetRuntimeDetailsProvider(nil)` is called AFTER a provider was set
    but BEFORE the cache is populated: `Runtime()` returns the base string
    only.
  - [ ] When the provider is set AFTER the first `NewStackTraceError` call: the
    set is a no-op (the cache has been populated); document explicitly in the
    godoc.
  - [ ] `go vet ./...`, `gofmt -l .` clean.

- **Pitfalls & edge cases:**
  - **`sync.Once` + mutex.** The pattern `runtimeDetailsExtraMu.Lock(); provider
    := runtimeDetailsExtraProvider; runtimeDetailsExtraMu.Unlock()` grabs the
    provider under lock and releases the lock *before* calling the provider.
    The provider must be free to call back into anything (including the logger)
    without deadlocking. Do NOT hold the mutex during the provider call.
  - **Test reset.** Tests that set a provider must reset it after. Add an
    unexported test helper in `error_test.go`:
    ```go
    // resetRuntimeCacheForTest clears the once-computed runtime string AND the
    // provider so subsequent tests get a clean slate. Test-only; do not call
    // from production code.
    func resetRuntimeCacheForTest() {
        runtimeDetails.once = sync.Once{}
        runtimeDetails.value = ""
        runtimeDetailsExtraMu.Lock()
        runtimeDetailsExtraProvider = nil
        runtimeDetailsExtraMu.Unlock()
    }
    ```
    Call this in `t.Cleanup(...)` of every provider-using test. **The reset
    helper is the per-test ergonomics fix the user flagged.** Without it,
    tests that set a provider would pollute subsequent tests in the same `go
    test` run because the `sync.Once` only fires once per process and
    test-process is shared across all tests in the package.
  - **Race: setting after first construction.** Documented as "no effect" — but
    a paranoid test should verify this. Subtest:
    `provider_set_after_first_call_is_ignored`. Construct one
    `NewStackTraceError`; assert `Runtime()` returns the base string; then set
    a provider; construct a second `NewStackTraceError`; assert its `Runtime()`
    STILL returns the base string (no provider output). This documents the
    contract in code.
  - **Provider returns empty string.** Treat as "no extra info"; append nothing
    (not even the trailing space). Spelled out in the implementation.
  - **Provider panics.** Not handled — let it propagate. A misbehaving provider
    is a programming error; loginjector is not the place to swallow it.
  - **Compatibility window with consumers.** Existing consumers
    (`fx_rate_monitor`, `com_lingocrm_api`) ship their own `StackTraceError`
    type. They will migrate to loginjector's at their own pace; the hook is
    additive and does nothing until someone calls
    `SetRuntimeDetailsProvider`.

- **Complexity:** Medium

## Risks

- **Gap A's buffering wrapper breaks streaming handlers silently.** The risk
  is that a consumer adds `NewRecoverHandler` to a route that streams
  responses (SSE, large file downloads), notices the route hangs / times out /
  buffers in memory until OOM, and blames the library. Mitigation: the godoc
  for `NewRecoverHandler` explicitly says "do not use around SSE / WebSocket /
  HTTP-2 push routes". The library cannot prevent misuse statically. A
  future option `WithStreamingPassthrough()` could opt out of buffering, but
  then the post-`WriteHeader` panic case re-opens — explicitly out of scope.

- **Gap A's `http.ErrAbortHandler` handling.** If the engineer re-panics with
  `fmt.Errorf(...)` instead of the raw `panicValue`, the stdlib server's
  `==`-comparison fails and the conn is logged instead of silently dropped.
  Mitigation: tests for this case (Task 3 subtest
  `panic_with_ErrAbortHandler`). Reviewer must verify the re-panic uses the
  raw value.

- **Gap B's `ensureThreadSafe` wrapper silently hides `LeveledHandler`.** If
  the engineer forgets to retain `original io.Writer` on `*writer` (or skips
  the `unwrapLeveled` helper that peels the wrapper), the interface check
  matches `*writer` directly, falls through to `l.minimumLogLevel`, and the
  option silently does nothing. The user gets the legacy behaviour with no
  error and no clue. Mitigation: Task 5's
  `leveled_handler_through_thread_safe_wrapper` subtest is mandatory; it is
  the only assertion that catches this. Reviewer must trace
  `WithMinLevel` -> `NewLogger` -> `ensureThreadSafe` -> `unwrapLeveled` -> 
  gate and confirm the wrapper is peeled.

- **Gap B's per-handler gate changes a hot path.** `Logger.WriteLog` is
  called on every log message. Replacing a single boolean (`handlersActive
  := level >= l.minimumLogLevel`) with a per-handler interface assertion +
  comparison runs the assertion N times per call (N = number of handlers,
  typically 1-3). Type assertions on concrete pointer-receiver types are
  effectively free in Go, but the change is still measurable. Mitigation:
  Task 8 records before/after benchmark numbers; a meaningful regression
  (>5%) on the existing `logger_test.go` benchmarks triggers a re-design
  conversation, not a merge.

- **Gap B's `LeveledHandler.MinLevel()` is read every call, not cached.**
  This is by design — the contract specifies the value must be stable over
  the handler lifetime, so the cost is essentially a struct-field read.
  But a misbehaving consumer that returns a different value on each call
  produces undefined behaviour across the gate. The library does not detect
  or defend against this; the godoc states the contract clearly. Mitigation:
  documentation only.

- **Gap C's process-wide hook + test isolation.** If `resetRuntimeCacheForTest`
  is not called in `t.Cleanup`, a test that sets a provider pollutes
  subsequent tests in the same `go test` run. Mitigation: every
  provider-using test calls the reset helper. Code-review gate.

- **Gap C's "set after first call is ignored" is surprising.** Consumers
  might expect `SetRuntimeDetailsProvider` to take effect immediately for
  the next error. It does not — the `sync.Once` has already fired. Mitigation:
  spelled out in the godoc, asserted by a test
  (`provider_set_after_first_call_is_ignored`). The alternative (un-cache the
  runtime string on every call) is a 10× performance hit on the hot error
  path; not worth it.

- **Allocation regressions across all three gaps.** Task 1 buffers every
  response; Task 4 adds a per-handler interface assertion + level compare
  inside `WriteLog`'s hot path; Task 6 adds one extra function call inside
  `sync.Once`. Each is a measurable cost; in aggregate they could move
  benchmark numbers. Mitigation: Task 8 records before/after numbers in the
  final commit message. A meaningful regression triggers a re-design
  conversation, not a merge.

- **Tests under `-race` reveal pre-existing latent bugs.** Adding new tests
  often surfaces races the existing suite did not exercise. If Task 8's
  `-race` run fails on code NOT touched by this plan, scope creep: file a
  separate bug and ship the gaps regardless. Do NOT silently fix
  unrelated bugs in this plan's commit.

## Trade-offs

- **Gap A: buffer the entire response vs. forward eagerly.** Buffering costs
  memory per request and breaks streaming. Eager forwarding loses the ability
  to convert a post-`WriteHeader` panic into a clean 500. We chose buffering
  because (a) the "half a page then garbage" failure mode is the worst kind
  of bug to discover in production, (b) streaming routes can opt out by
  simply not wrapping them, (c) the memory cost is bounded by the response
  size — which is, by definition, bounded for non-streaming routes. The
  alternative (write a sentinel "X-Panic-Reset: 1" trailer header and let
  the client retry) is laughable in 2026. **User-confirmed:** the streaming
  trade-off is accepted; the godoc-warning mitigation (see Task 1 acceptance
  criteria) is the documented contract.

- **Gap A: three options or more.** Five was tempting (add
  `WithRecoverLogger`, `WithStackCapture(func() string)`, …). Three is the
  minimum that lets a consumer change the response body, the content type,
  and the public message. Anything beyond can wrap `NewRecoverHandler` in
  ten lines.

- **Gap B: Shape 1 (optional `LeveledHandler` interface) vs. Shape 2 (paired
  struct + new constructor).** Shape 1 declares `LeveledHandler { io.Writer;
  MinLevel() LogLevel }`; `Logger.WriteLog` checks each handler — if it
  implements the interface, use `MinLevel()` instead of `Logger.minimumLogLevel`
  for the gate; otherwise use the global. Shape 2 declares a paired struct
  `LeveledHandler { MinLevel LogLevel; Writer io.Writer }` and a new
  constructor `NewLoggerWithLeveledHandlers(min, ...LeveledHandler)`.

  We chose Shape 1 for three reasons:
  (a) **Idiomatic Go.** Optional interfaces detected via type assertion are
      the stdlib's own pattern — `io.WriterTo`, `http.Flusher`, `http.Hijacker`,
      `http.Pusher`. This codebase already uses the pattern heavily in
      `httptap/rw.go` for the eight `*RW` composite types. Consumers reading
      `Logger`'s code will recognise it immediately.
  (b) **Zero API breakage.** `NewLogger(minLevel, handlers...)` stays the
      single constructor. Existing handler factories (`PrintHandler`,
      `CyclicOverwritingFilesHandler`, etc.) keep their `io.Writer` return
      type — they simply do not implement the new interface, so behaviour is
      unchanged. Shape 2 requires a sibling constructor with overlapping
      responsibility; consumers who want a mix of leveled and unleveled
      handlers under Shape 2 must wrap all of them, including the unleveled
      ones (extra ceremony for the common case).
  (c) **Composes cleanly with the `WithMinLevel(level, handler)` helper.**
      Shape 1 + `WithMinLevel` lets a consumer opt in per-handler without
      learning a new constructor:
      `NewLogger(Info, file, WithMinLevel(Warning, console))`.
      Shape 2 forces the consumer to learn both a constructor AND a struct
      type. The discovery cost is also lower for Shape 1: anyone scanning
      `handler.go` for handler factories sees `WithMinLevel` listed alongside
      `PrintHandler`, `TelegramHandler`, etc.

  Shape 1's only real cost is the implicit/explicit semantics split: a
  handler's threshold may or may not exist depending on whether it implements
  the interface. This is acceptable because the default (no interface =
  global threshold) is the obvious and existing behaviour, and `WithMinLevel`
  makes the explicit case visible at the callsite.

- **Gap B: interface-detection happens before the thread-safe wrapper hides
  it.** `NewLogger`'s `ensureThreadSafe` wraps every handler in `*writer{
  h: w.Write }`. A naive `LeveledHandler` check on the wrapped value would
  always miss the interface. Two ways to fix: (a) detect the interface
  *before* wrapping and stash the level on the wrapper, (b) propagate
  through the wrapper by adding `original io.Writer` to `*writer` and
  having `unwrapLeveled` peel it. We chose (b) because it keeps the
  detection logic in one place (`WriteLog`) and avoids spreading
  level-awareness through the registration path. Cost: one extra field on
  `*writer` and a tiny `unwrapLeveled` helper.

- **Gap B: console-printer-only API vs. fully-generic per-handler exposure.**
  The library now ships *both*: `WithMinLevel` (generic, available to any
  handler) AND `WithPrinterMinLevel` (specific to `NewFileLogger`'s console
  printer). The former is the foundation; the latter is the convenience layer
  for the observed consumer pattern. Consumers using `NewLogger` directly can
  reach for `WithMinLevel` on any handler; consumers using `NewFileLogger`
  get the named option that matches their copy-paste pattern. Earlier drafts
  considered shipping only the narrow option; we widened the scope after
  re-reading the consumer evidence — once the core supports it, exposing the
  generic helper is essentially free.

- **Gap C: process-wide hook vs. per-constructor option.** Per-constructor
  option is more testable and avoids global state. Process-wide hook
  matches the existing `cachedRuntime()` pattern and frees consumers from
  wiring the provider at every callsite. We chose process-wide because the
  existing code is already process-wide for the same data, and the
  consumer evidence is "every panic constructor wants the gopsutil details"
  — the per-callsite annoyance is real. **User-confirmed:** the
  process-wide `SetRuntimeDetailsProvider` + `sync.Once` design is the
  chosen approach.

- **Gap C: append vs. replace.** Appending the provider's output to
  `os=… arch=… go=…` keeps the base descriptor reliable. A consumer that
  wants to fully replace can do so by setting `LogLevel` constants to
  unambiguously identify their environment in their own way. We picked
  append because the base data is cheap, useful, and never the wrong thing
  to log.

- **Gap C: `SetRuntimeDetailsProvider` vs. `RegisterRuntimeDetailsProvider`
  with cumulative providers.** Cumulative providers (multiple `Register`
  calls chain) would let two libraries each contribute runtime detail
  without stepping on each other. We picked single-set because (a) the
  consumer pattern is "one application owns its environment description",
  (b) cumulative providers require ordering, deduplication, and a way to
  remove a registration — far more API surface than the gap demands, (c) a
  consumer that wants composition can pass `func() string { return a() + " "
  + b() }` themselves.

- **Library-level vs. sub-package.** Gap A goes into `httptap/` (HTTP-shaped,
  matches sibling handlers). Gap B is a root-package option (`filelogger.go`
  lives at the root). Gap C is a root-package hook (`error.go` lives at the
  root). The `runtimex` sub-package that hosts the gopsutil provider — out
  of scope here — will follow the `httptap` / `levels` sub-package pattern:
  separate directory, separate import, dep-light root unaffected.
