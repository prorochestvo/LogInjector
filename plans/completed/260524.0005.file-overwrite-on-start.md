# Task Breakdown

## Implementation Notes (as shipped)

One deliberate divergence from Task 1's acceptance criteria: the construction-time
truncate open uses `os.O_WRONLY|os.O_TRUNC` **without `O_CREATE`** (the AC text said
`O_WRONLY|O_CREATE|O_TRUNC`). Reason: with `O_CREATE` the factory created an empty
index-1 file at construction — a side effect the base handler does not have, which
contradicted this plan's own "no-op when the file is absent" contract (Task 1 pitfalls,
Task 2 subtest). Dropping `O_CREATE` makes the handler truly "identical to the base
except truncation": a missing file or folder is a no-op at construction (creation
happens on the first write, as in the base), an existing file is truncated, and any
other error (e.g. permissions) is stashed and surfaced on the first `Write` — exactly
mirroring the base handler's `seedErr` pattern. The three-lens review flagged the
`O_CREATE` side effect (Lens A P1 / Lens B godoc); this resolves it. All other ACs met.

## Overview

Restore an "overwrite-on-start / fresh-log-per-run" capability to the file handlers
as an **additive, backward-compatible** feature, shipped as **v1.1.0**.

Context (verified, not assumed): both `CyclicOverwritingFilesHandler` and
`FileByFormatHandler` (`handler.go`) open files with
`os.O_WRONLY|os.O_CREATE|os.O_APPEND` and **never** truncate. `git log --all -S
O_TRUNC` is empty — a truncate mode never existed in tracked history (deep history was
lost in a repo reinitialization), so this is a *new* feature, not a regression to
revert. The handler name `CyclicOverwritingFilesHandler` is misleading: it appends to
the current index file, rotates on size, and ring-prunes the oldest files — it never
"overwrites" a file's contents.

Approach: add a new sibling factory that, **once at construction**, truncates the
current target index file (index 1) so a fresh run starts with an empty file, then
behaves identically to the existing handler (seed size from disk → in this case zero
after truncation → append → rotate → prune). The existing factory signature and
behavior stay byte-for-byte unchanged. A godoc clarification corrects the
"Overwriting" naming confusion without renaming anything (a rename would break the
public API).

Scope is the cyclic handler. `FileByFormatHandler` is addressed only as a documented
non-goal (see Trade-offs / Task 5).

## Assumptions

- **Chosen semantics for "fresh log on each run" = option (a):** truncate ONLY the
  current target index file (index 1) exactly once, at construction time, via a single
  `O_TRUNC` open. Rotation and pruning then continue exactly as in the existing
  handler. The two alternatives — (b) clear all `prefix.*.log` in the folder and
  restart at index 1, and (c) start a brand-new index beyond existing files — are
  documented as rejected alternatives in Trade-offs, not implemented.
- Index always starts at 1 on construction (this is the existing, documented behavior —
  the handler does not scan the folder to resume at the highest index). The truncate
  therefore targets exactly the same file the existing constructor would seed from:
  `prefix.00000001.log`.
- "Truncate at construction" means: if `prefix.00000001.log` exists, empty it now; if it
  does not exist, do nothing (it will be created on first write, already empty). After
  truncation the seeded size counter is 0.
- This is a new minor version (v1.1.0). Current released tag is `v1.0.5`; the only
  change is additive (new exported symbol + godoc), so SemVer minor is correct.
- No new third-party dependencies. Stays `CGO_ENABLED=0`, deps unchanged
  (`stretchr/testify`, `twinj/uuid`).
- The truncate uses `defaultFilePermissions` (0640) and `defaultFileExtension`
  ("log"), matching every other file path in this package.

## Tasks

### Task 1: Add `CyclicOverwritingFilesHandlerWithReset` factory

- **Description:** In `handler.go`, add a new exported factory next to
  `CyclicOverwritingFilesHandler` (currently lines 86–159). Same parameter list and
  return type:
  `CyclicOverwritingFilesHandlerWithReset(folder, fileNamePrefix string, maxFileCapacity uint32, maxFilesInFolder int) io.Writer`.
  Its only behavioral difference from the existing factory: **once, at construction
  time**, it truncates the index-1 target file (`fileNamePrefix.00000001.log`) so the
  run starts with an empty file, then proceeds with the existing seed/append/rotate/
  prune logic (where the seeded size is now 0 because the file was just emptied). The
  existing `CyclicOverwritingFilesHandler` must remain **unchanged** (do not edit it).

  Implementation note (why a sibling factory, not a shared private helper edit): keep
  the existing factory's body untouched to guarantee byte-for-byte behavioral identity
  and a trivial review diff. The new factory may share the index/filename/`writer`
  construction shape, but the truncation step is the only addition and must run exactly
  once before the `writer` closure is returned. Do **not** add an `O_TRUNC` open inside
  the per-write `h` closure — that would erase the file on every write.

- **Acceptance Criteria:**
  - [ ] New exported function `CyclicOverwritingFilesHandlerWithReset` exists with the
        signature above and returns an `io.Writer`.
  - [ ] At construction it opens `path.Join(folder, fileNamePrefix+".00000001."+defaultFileExtension)`
        with `os.O_WRONLY|os.O_CREATE|os.O_TRUNC` (perms `defaultFilePermissions`),
        then closes it immediately — no write performed at construction.
  - [ ] After construction and before any `Write`, a pre-existing index-1 file with
        content is empty (size 0).
  - [ ] After truncation the in-memory size counter is 0 (the first write appends from
        an empty file).
  - [ ] Subsequent writes append, rotate at `maxFileCapacity`, and prune via
        `verifyFiles` identically to the existing handler.
  - [ ] `CyclicOverwritingFilesHandler` source is unchanged (diff shows only additions).
  - [ ] A construction-time truncate error is surfaced (not silently dropped) on the
        first `Write` return value, mirroring the existing `seedErr` fail-soft pattern
        (stash at construction, join into the first `Write`'s error, then clear).

- **Pitfalls & edge cases:**
  - Truncating inside the per-write closure would zero the file on every message —
    truncate must happen exactly once, before the closure is returned.
  - If the folder does not exist, `O_TRUNC|O_CREATE` open will fail; decide between
    fail-soft (stash the error, surface on first write — consistent with the existing
    `seedErr` handling) vs. silently ignoring `os.IsNotExist`. Recommended: treat a
    missing parent directory the same way the existing handler treats a missing file —
    do not create directories here (the existing handler doesn't either; first write
    would fail to open and surface the error). Stash any non-`IsNotExist`-style open
    error and join it into the first write.
  - Do not call `verifyFiles` at construction — reset truncates the current file only;
    pruning stays write-driven, identical to the existing handler.
  - `O_APPEND` and `O_TRUNC` together is nonsensical — the truncation open is a
    *separate, one-shot* open (`O_WRONLY|O_CREATE|O_TRUNC`), distinct from the
    per-write append open inside `h`.
  - Respect `defaultFilePermissions` (0640) and `defaultFileExtension` — do not
    hard-code 0666 or "log" (an old comment in CLAUDE.md says 0666; the constant is
    actually 0640 — trust the constant).

- **Complexity:** Medium

- **Code Example:** (shape only — illustrates the one-shot truncate before returning
  the writer; the per-write closure body is the same as the existing handler's)

  ```go
  func CyclicOverwritingFilesHandlerWithReset(folder, fileNamePrefix string, maxFileCapacity uint32, maxFilesInFolder int) io.Writer {
      index := 1
      fileName := fileNamePrefix + "." + fmt.Sprintf("%0.8X", index) + "." + defaultFileExtension

      // overwrite-on-start: truncate the current target file exactly once at
      // construction so each run begins with an empty index-1 file. seeded size is 0.
      var resetErr error
      f, err := os.OpenFile(path.Join(folder, fileName), os.O_WRONLY|os.O_CREATE|os.O_TRUNC, defaultFilePermissions)
      if err != nil {
          resetErr = err // fail-soft: surface on first Write, like seedErr
      } else if cErr := f.Close(); cErr != nil {
          resetErr = cErr
      }

      var fileSize uint64 // starts at 0 after truncation

      w := &writer{
          h: func(msg []byte) (int, error) {
              err := resetErr
              resetErr = nil
              // ... identical append / rotate / prune body as CyclicOverwritingFilesHandler ...
              _ = err
              return 0, nil // placeholder — real body mirrors existing handler
          },
      }
      return w
  }
  ```

### Task 2: Test — overwrite mode truncates the existing file at construction

- **Description:** Add a new top-level test
  `TestCyclicOverwritingFilesHandlerWithReset` in `handler_test.go`, structured as
  one function with `t.Run(...)` subtests (per the repo's "one `Test*` per method,
  scenarios as subtests" rule). Reuse the existing `extractFilesOrFail` helper. The
  load-bearing subtest pre-creates the index-1 file with content, constructs the
  handler in reset mode, and asserts the file is empty/started fresh **before** the
  first write appends.

- **Acceptance Criteria:**
  - [ ] Subtest `truncates existing file at construction`: pre-create
        `prefix.00000001.log` with non-empty content; construct via
        `CyclicOverwritingFilesHandlerWithReset`; **before any Write**, assert the file
        on disk is empty (size 0). Then perform a write and assert the file contains
        only the newly written line (e.g. `"first\n"`), proving the old content is gone
        and the new content was appended fresh.
  - [ ] Subtest `no existing file is a no-op`: construct in an empty folder; assert no
        files exist before the first write (truncate of a non-existent file must not
        spuriously create non-empty state) and that the first write creates
        `prefix.00000001.log` with just that line.
  - [ ] Subtest `rotation and prune still apply`: mirror the existing
        `TestCyclicOverwritingFilesHandler` capacity/prune assertions on the reset
        handler (small `maxFileCapacity`, several writes) to prove reset composes with
        rotation and `verifyFiles` exactly like the base handler.
  - [ ] Subtest `seeded size resets to zero`: pre-create index-1 file at exactly
        `maxFileCapacity` bytes, construct in reset mode, then a single small write must
        **not** trigger rotation (because truncation reset the seeded size to 0) — i.e.
        index stays at 1, no `prefix.00000002.log` is produced. This is the inverse of
        `TestCyclicOverwritingFilesHandler_SeedFromDisk` and is the key composition
        proof.
  - [ ] All assertions use `require`/`assert` from testify; compile-time interface
        check is not needed (no new mock type).

- **Pitfalls & edge cases:**
  - Use `t.TempDir()` and `defaultFilePermissions`/`defaultFileExtension` (don't
    hard-code "log"/0640) so the test matches the package's own filename construction
    (`fmt.Sprintf("%s.%08X.%s", prefix, 1, defaultFileExtension)` — note `%08X`, the
    same width the handler uses via `%0.8X`).
  - The "before first write" assertion must read the file directly with `os.Stat` /
    `os.ReadFile` — `extractFilesOrFail` returns content keyed by name and is fine for
    the post-write checks, but the pre-write empty-file check needs a direct stat
    because an empty file still exists on disk.
  - `bytes.TrimSpace` is applied to each message by the handler, so expected content is
    `"first\n"`, not `" first \n"` — match the existing tests' expectations.

- **Complexity:** Medium

## Risks

- **Truncation in the wrong place** (inside the per-write closure) silently wipes logs
  on every message. Mitigation: explicit acceptance criterion that the truncate open is
  one-shot at construction, plus the `seeded size resets to zero` subtest which would
  catch repeated truncation indirectly (rotation timing would be wrong).
- **Operator surprise / data loss expectation:** a user who points the reset handler at
  a folder with valuable index-1 content loses it on restart by design. Mitigation:
  blunt godoc — "truncates the current file on construction; existing content in
  index-1 is destroyed." This is the feature, not a bug, but it must be unmissable.
- **Name confusion compounding:** adding `…WithReset` to an already-misnamed
  `…Overwriting` handler risks more confusion ("isn't overwriting already in the
  name?"). Mitigation: the Task 4 godoc explicitly explains the base handler does *not*
  overwrite, and reset is the one that actually truncates.
- **Backward compatibility:** zero risk to existing callers — the existing factory is
  untouched and no signature changes. The new symbol is purely additive (SemVer minor).

## Trade-offs

- **Semantics: chose (a) truncate-current-file over (b) clear-all and (c) new-index.**
  - (a) truncate index-1 at construction — chosen. It is the smallest, most local
    change; composes cleanly with the existing index-starts-at-1 behavior and the
    stat-based size seeding (seeded size simply becomes 0); rotation and prune are
    unaffected. "Fresh log this run" maps naturally to "the file this run writes to
    starts empty."
  - (b) remove/clear all `prefix.*.log` and restart at index 1 — rejected for now. More
    destructive (nukes prior runs' rotated history), and overlaps with what
    `maxFilesInFolder` pruning already does over time. Heavier blast radius for a
    marginal benefit. Documented as a possible future `…WithPurge` variant.
  - (c) start a brand-new index beyond existing files — rejected. Requires the
    folder-scanning resume logic the handler explicitly does *not* have today (per its
    own godoc), so it's a larger change; and it grows file count rather than reusing,
    fighting the ring-prune model. Better suited to a separate "resume at highest index"
    feature.
- **API shape: chose a sibling factory over functional options / config struct.**
  - Sibling factory `CyclicOverwritingFilesHandlerWithReset(...)` — chosen. Matches the
    package's existing idiom (every handler is a plain factory function), keeps the
    existing factory byte-for-byte unchanged, is trivially discoverable via godoc, and
    needs no new exported `Option` or `Config` type. Lowest cognitive and review cost.
  - Variadic functional options on a *new* factory — rejected: introduces an exported
    `Option` type and machinery for a single boolean knob; over-engineered for one flag
    and out of step with the rest of the package (no other handler uses options).
  - Config struct constructor — rejected for the same reason: heavier surface for one
    boolean; would also tempt a migration of the existing factory, which we must not
    touch.
- **Rename vs. clarify:** chose godoc clarification over renaming
  `CyclicOverwritingFilesHandler`. A rename is the "correct" fix for the misleading
  name but breaks every existing import — unacceptable for a minor version. Godoc is the
  honest, non-breaking compromise.
- **`FileByFormatHandler` left out:** its per-write generated filename makes
  "truncate the construction-time target" ill-defined; a date-stamp generator already
  yields fresh files per period. Adding reset there needs distinct semantics and is
  deferred to its own plan rather than bolted on here.
