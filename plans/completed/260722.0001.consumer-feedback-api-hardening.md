# Task Breakdown â€” consumer-feedback API hardening (from beacon v1.0.7 adoption)

## Overview

Consumer-driven design plan. While migrating **beacon** to loginjector v1.0.7 (cheap
`NewTraceError` on every expected error, `NewStackTraceError` only on process-level panic
recovery), a three-lens review flagged loginjector API footguns and gaps. On closer reading
of the source, several are **deliberate, well-documented design** â€” not bugs. This plan
keeps only what a real consumer (beacon) actually needs, plus a few additive hardenings for
the next consumer.

**Nothing here blocked beacon** â€” it shipped on v1.0.7 unchanged. This plan is the library
side; implement it when convenient and beacon will wire to the new API afterward.

**Maintainer decision (this revision):** implement **all** tasks 1â€“7; task 8 stays a
documented **defer** (see its section for the rationale). Every task below has been
re-verified against the actual source and rescoped where the original wording was
undeliverable or ambiguous, so the engineer can execute without re-deciding.

Source: beacon `refactor(errors): migrate to loginjector v1.0.7 trace discipline`
(`plans/completed/260721.0001.stacktrace-trace-discipline-migration.md` in the beacon repo).

## How to read this plan

Each task states: the problem, the current library behavior (with `file:line`), whether
**beacon actually needs it**, the proposed API, and â€” where beacon needs it â€” a concrete
**beacon usage example written as if the API already exists** (the target the library must
satisfy). Beacon's own tree is intentionally NOT wired to these yet; it stays on v1.0.7.
All beacon usage examples in this revision are written to compile against the proposed
signatures.

### Task 1 â€” `WithFileMode(os.FileMode)` for RotatingFileHandler (beacon needs this)
- **Description**: `RotatingFileHandler` hardcodes `defaultFilePermissions = 0640`
  (`handler.go:694`), applied at the live-file open (`handler.go:314`) and the gzip `O_EXCL`
  temp create (`handler.go:892`). beacon's log files contain `chat_id` values (PII-adjacent
  under beacon's data policy) and should be `0600` on a shared host; a consumer cannot
  currently tighten the mode. Add a `WithFileMode` option that threads a `cfg.fileMode`
  through both create sites.
- **Proposed API**:
  ```go
  // WithFileMode overrides the permission bits used when the handler CREATES a log
  // file â€” the live prefix.<8 hex>.log (or prefix.log under WithStableCurrentName) and
  // the gzip .log.gz temp under WithCompress. The default is 0640.
  //
  // It applies only to files this handler creates; the process umask still masks the
  // bits (the effective mode is only ever more restrictive than requested). It does NOT
  // re-chmod a file that already exists on disk â€” a live file left at 0640 by a prior
  // run keeps 0640 because it is opened O_APPEND, not created. Pass WithFreshStart if you
  // need the mode to apply to a from-scratch ring.
  func WithFileMode(mode os.FileMode) RotatingFileOption
  ```
- **Implementation (pin these decisions):**
  1. Add `fileMode os.FileMode` to `rotatingFileConfig` (`handler.go:553`).
  2. **Seed the default in the config literal**, because a bare zero-value `os.FileMode`
     is `0` (an unusable mode). At `handler.go:206`:
     ```go
     cfg := rotatingFileConfig{
         maxFileSize: 5 << 20,
         maxFiles:    7,
         fileMode:    defaultFilePermissions, // zero-value os.FileMode is 0 (invalid); seed it
     }
     ```
  3. Use `cfg.fileMode` in place of the `defaultFilePermissions` literal at the live-file
     open (`handler.go:314`). For the gzip temp create (`handler.go:892`), `compressFile`
     currently takes `(folder, base string)` â€” thread the mode in (add a `mode os.FileMode`
     parameter, or read it from a captured `cfg`). Its comment about the `0640 O_EXCL` temp
     must be updated to reflect the configurable mode.
- **Explicitly out of scope**: `FileByFormatHandler` (`handler.go:365`) shares the same
  `defaultFilePermissions` const at its own create site but takes **no options**. It is
  intentionally left on the hardcoded `0640`; do not add a mode knob there in this task.
- **How beacon uses it** (as if ready â€” `internal/logger.go`):
  ```go
  fHNDL := loginjector.RotatingFileHandler(folder, name,
      loginjector.WithMaxFileSize(defaultFileSize),
      loginjector.WithMaxFiles(7),
      loginjector.WithFileMode(0o600), // logs carry chat_id; keep them owner-only
  )
  ```
- **Acceptance criteria**:
  - Created live file has the requested mode (assert against `os.Stat().Mode().Perm()`,
    accounting for umask on the CI host â€” set a permissive umask in the test or assert the
    masked expectation).
  - A rotated `.gz` created under `WithCompress` + `WithFileMode` has the requested mode.
  - With no `WithFileMode`, the default stays `0640` (existing behavior byte-identical).
  - godoc states the created-files-only + umask semantics as above.
- **Pitfalls**: umask masks the bits â€” a test asserting an exact `0600` must control the
  umask or assert `mode & ^umask`. The live file is `O_APPEND`-opened, so a pre-existing
  file from a prior run is NOT re-chmod'd; the test must create a fresh file (or use a temp
  dir / `WithFreshStart`) to observe the requested mode.
- **Breaking**: no (additive). **Complexity**: Small.

### Task 2 â€” `StackRedacted()` on StackTraceError (beacon: optional, log cleanliness)
- **Description**: `Stack()` embeds absolute build-host paths (`error.go:62-64`,
  `internal/stacktrace.go:24-25`); the only guardrail is a godoc warning. Add a
  `StackRedacted()` accessor that returns the stack with each frame's absolute source path
  shortened, reusing **the exact shortening `LineTrace` already applies** â€” no new module-root
  detector.
- **Pinned trimming rule**: `LineTrace` shortens an absolute file path to `<parent>/<base>`
  (`internal/stacktrace.go:68-76`) â€” stdlib frames become `testing/testing.go`, module files
  get their immediate parent-dir prefix. `StackRedacted()` applies that same reduction to
  every `file:line` frame line in the `Stack()` text. Extract the shortening into one shared
  internal helper so the `LineTrace` path and the stack-redaction path cannot drift.
- **Proposed API**: a package-level function, NOT a method on the exported `StackTraceError`
  interface â€” adding a method to an unsealed exported interface breaks any external type that
  implements it, which is forbidden in this public library. A new function is purely additive.
  ```go
  // StackRedacted returns e.Stack() with each frame's absolute source path shortened to
  // "<parent>/<base>" â€” the identical reduction LineTrace applies (e.g.
  // "/home/ci/proj/main.go:10" becomes "proj/main.go:10"). The line numbers, goroutine
  // header, and frame ordering are preserved; only the leading directory prefix of each
  // frame path is dropped. This removes build-host absolute path prefixes, making the
  // output safe to surface where Stack() is not.
  func StackRedacted(e StackTraceError) string
  ```
  Note: this is deliberately NOT `-trimpath` parity. `-trimpath` keeps the full
  module-relative path; the `LineTrace` reduction collapses to parent/base. The godoc above
  promises exactly what the reuse delivers.
- **Implementation (concrete)** â€” in `internal/stacktrace.go`:
  ```go
  // shortenFramePath reduces an absolute source path to "<parent>/<base>".
  // Extracted from LineTrace (lines 71-76) so both paths share one rule.
  func shortenFramePath(file string) string {
      base := path.Base(file)
      parent := path.Base(path.Dir(file))
      if parent != "" && parent != "." && parent != "/" {
          return parent + "/" + base
      }
      return base
  }

  // RedactStackPaths rewrites each tab-indented frame line of a StackTrace() dump
  // ("\t<abs-path>:<line> +0x..") so <abs-path> is shortened via shortenFramePath.
  func RedactStackPaths(stack string) string {
      lines := strings.Split(stack, "\n")
      for i, ln := range lines {
          if !strings.HasPrefix(ln, "\t") {
              continue // only frame path lines are tab-indented
          }
          body := ln[1:]
          suffix := ""
          if sp := strings.IndexByte(body, ' '); sp >= 0 { // strip trailing " +0x.."
              body, suffix = body[:sp], body[sp:]
          }
          if colon := strings.LastIndexByte(body, ':'); colon >= 0 { // "<path>:<line>"
              lines[i] = "\t" + shortenFramePath(body[:colon]) + body[colon:] + suffix
          }
      }
      return strings.Join(lines, "\n")
  }
  ```
  Rewrite `LineTrace` (`internal/stacktrace.go:71-76`) to call `shortenFramePath` so the two
  paths use one implementation. The package-level `StackRedacted(e StackTraceError)` returns
  `internal.RedactStackPaths(e.Stack())`.
- **How beacon uses it** (as if ready â€” `cmd/collector/main.go` recover):
  ```go
  stackErr := loginjector.NewStackTraceError()
  log.Printf("execution: runner panic recovered: %v\n%s\n%s",
      rec, loginjector.StackRedacted(stackErr), stackErr.Runtime())
  ```
- **Acceptance criteria**:
  - A redacted stack contains no leading build-host absolute prefix (no `/home/...` /
    `/Users/...` / build-root segment) on any frame line.
  - It still contains `<parent>/<base>:<line>` frames and the `+0x..` offsets.
  - `LineTrace`'s existing output is unchanged (its tests still pass) â€” the extraction is
    behavior-preserving.
- **Pitfalls**: the goroutine header and `func(...)` lines are not tab-indented and must be
  left untouched â€” only rewrite tab-indented frame lines. A path containing a space is
  theoretically possible but does not occur in `runtime.Stack` output; the space-split is
  safe for real dumps.
- **Breaking**: no. **Complexity**: Small (down from Medium: the trimming rule is now pinned
  to LineTrace reuse, no design decision left open).

### Task 3 â€” sanitize provider output shape in Runtime() (beacon: indirect)
- **Description**: This is **output-shape robustness, not a security fix.** The provider set
  via `SetRuntimeDetailsProvider` is trusted, process-owned code, invoked at most once under
  the `sync.Once` in `cachedRuntime` (`error.go:388-419`) â€” there is no attacker-controlled
  path and no log-injection threat model. The real problem is cosmetic/structural: the
  provider result is appended verbatim as `base + " " + extra` (`error.go:410-411`), so a
  multi-line provider (beacon's provider returns a multi-line gopsutil block) makes
  `Runtime()` render as a mashed first line followed by stray newlines that break the
  single-line contract other sinks assume.
- **Proposed change**: sanitize the provider result to keep `Runtime()` single-line â€”
  collapse newlines/tabs to spaces and strip other control characters â€” **inside the
  `once.Do` closure, before the append** at `error.go:410-411`. Because this runs inside the
  `once.Do` it is single-threaded; no locking concerns.
  ```go
  // sanitizeRuntimeExtra keeps Runtime() single-line: CR/LF/TAB become spaces,
  // other control chars are dropped. Runs inside the once.Do (single-threaded).
  func sanitizeRuntimeExtra(s string) string {
      return strings.Map(func(r rune) rune {
          switch {
          case r == '\n' || r == '\r' || r == '\t':
              return ' '
          case r < 0x20 || r == 0x7f:
              return -1 // drop other control chars
          default:
              return r
          }
      }, s)
  }
  ```
  At the append site:
  ```go
  if extra := provider(); extra != "" {
      base = base + " " + sanitizeRuntimeExtra(extra)
  }
  ```
  Update the `SetRuntimeDetailsProvider` godoc (`error.go:357-358`) from "avoid control
  characters" to state that CR/LF/TAB are collapsed to spaces and other control characters
  are stripped so `Runtime()` stays single-line.
- **How beacon uses it**: beacon keeps returning rich detail from its provider; the library
  guarantees the appended output can't break the log line. (beacon may also single-line its
  own provider â€” see the beacon-side follow-up.)
- **Acceptance criteria**:
  - A provider returning `"a\nb\x1b[31mc"` yields a single-line `Runtime()` with no newline
    and no ESC byte (`\x1b`) in the appended segment.
  - A single-line provider result is byte-identical to today (only control chars change).
- **Pitfalls**: don't strip legitimate printable Unicode â€” only `< 0x20`, `0x7f`, and the
  whitespace-to-space mapping. Runs of spaces are acceptable; no need to collapse them.
- **Breaking**: no (output-shape hardening). **Complexity**: Small.

### Task 4 â€” redact the bot token from TelegramHandler errors (P1, security; other consumers)
- **Description**: **Confirmed real leak.** `TelegramHandler` builds
  `url = ".../bot<TOKEN>/sendDocument"` at `handler.go:23`. When the request build
  (`handler.go:56-59`) or `client.Do` (`handler.go:62-65`) fails, the returned `*url.Error`'s
  `Error()` embeds the full URL â€” **including the token** â€” which then lands in logs and log
  files. `botToken` is the only token-bearing value in the closure. beacon does not use
  `TelegramHandler` (it has its own `tbotclient`), so **no beacon impact**, but it is a real
  token leak for any consumer that does. This is the highest-value item in the plan.
- **Proposed change**: redact the token from the **surfaced error string** at the two
  URL-bearing return sites (`handler.go:58` and `handler.go:64`). Do not alter the other
  multipart-part error returns â€” they never carry the URL.
  ```go
  // build once alongside url (handler.go:23):
  redactToken := func(s string) string {
      // guard: empty token â†’ nothing to redact; a very short token would over-match
      // unrelated substrings, and no real Telegram token is that short, so skip it.
      if len(botToken) < 8 {
          return s
      }
      return strings.ReplaceAll(s, botToken, "***")
  }
  ```
  At the two sites:
  ```go
  request, err := http.NewRequest("POST", url, payload)
  if err != nil {
      return 0, fmt.Errorf("could not create HTTP request: %v", redactToken(err.Error()))
  }
  ...
  r, err := (&http.Client{Timeout: time.Second * 20}).Do(request)
  if err != nil {
      return 0, fmt.Errorf("could not send HTTP request: %v", redactToken(err.Error()))
  }
  ```
- **Caveats to encode** (pinned):
  - **Guard empty / very-short `botToken`** (the `len(botToken) < 8` gate) so redaction never
    over-matches unrelated substrings; a token that short is not a valid Telegram token.
  - **Redact only the surfaced string.** These sites already flatten with `%v` (not `%w`), so
    there is no `errors.Is`/`As` chain to preserve or break; redacting `err.Error()` before
    formatting is safe and does not degrade traversal that isn't there.
  - The URL after redaction reads `.../bot***/sendDocument` â€” the literal `bot` prefix stays,
    only the secret is masked.
- **Acceptance criteria**:
  - A forced transport error (e.g. point the handler at an unroutable/closed address, or
    inject a failing round-tripper) returns an error whose string does **not** contain the
    token, and does contain `***`.
  - The happy path and non-URL error messages are unchanged.
- **Pitfalls**: the `Timeout`ed default client makes a real network failure easy to force in
  a test (connect to a reserved TEST-NET address or a closed local port); keep the test fast
  by using a short-timeout client seam or an unroutable address so it fails deterministically.
- **Breaking**: no. **Complexity**: Small.

### Task 5 â€” runtime-details freeze probe (beacon: weak â€” RESCOPED)
- **Why the original spec was undeliverable**: the plan proposed
  `RuntimeDetailsProviderSet() bool` promising "a provider is registered AND the string has
  not yet been frozen â€” i.e. a provider set now would still take effect." That is not
  honestly deliverable:
  - `sync.Once` exposes no "has it fired" query.
  - `runtimeDetails.value` is written **without the mutex** (`error.go:416`), so any external
    read of it data-races a concurrent `Do`.
  - the "would still take effect" guarantee is inherently TOCTOU â€” a freeze can happen between
    the check and a subsequent `SetRuntimeDetailsProvider`.
  - the beacon example called `SetRuntimeDetailsProvider()` with **no argument**, which does
    not compile (the signature is `SetRuntimeDetailsProvider(fn func() string)`).
- **Rescoped, honestly-deliverable design**: add a **freeze-state probe** backed by an
  `atomic.Bool` set once at the top of the `once.Do` closure. It reports one true fact â€”
  whether the runtime-details string has already been computed and cached â€” and drops the
  concurrency / "would take effect" guarantee entirely. (Chosen over the "is a provider
  currently registered" boolean because beacon's actual intent is to catch *late*
  registration, which freeze-state answers and registration-state does not.)
- **Proposed API**:
  ```go
  // RuntimeDetailsFrozen reports whether the process-wide runtime-details string has
  // already been computed and cached (frozen). Once it returns true, a subsequent
  // SetRuntimeDetailsProvider call is a no-op â€” the provider will never run and its
  // detail will not appear in any StackTraceError's Runtime().
  //
  // This is an advisory diagnostic, not a synchronization primitive: it reports the
  // freeze state at the moment of the call. It gives no guarantee that a provider set
  // immediately after a false result will take effect (a concurrent StackTraceError
  // construction may freeze the string in between). Use it at startup to warn that a
  // provider was registered too late, not to gate registration.
  func RuntimeDetailsFrozen() bool
  ```
- **Required data-race-free implementation** (`error.go`):
  - Add `import "sync/atomic"` (not currently imported).
  - Add a package-level `var runtimeDetailsFrozen atomic.Bool`.
  - As the **first statement inside the `once.Do` closure** in `cachedRuntime`
    (`error.go:395`), `runtimeDetailsFrozen.Store(true)`.
  - `func RuntimeDetailsFrozen() bool { return runtimeDetailsFrozen.Load() }`.
  - The atomic is the only cross-goroutine channel here; it never reads the mutex-unguarded
    `runtimeDetails.value`, so there is no data race. Note the test-reset seam that swaps
    `runtimeDetails.once` (`error.go:377-384`) must also reset `runtimeDetailsFrozen` to
    `false` to keep the probe consistent across a reset â€” wire that into the existing reset
    helper if one exists, or document it beside the atomic.
- **How beacon uses it** (as if ready â€” `internal/logger.go`, at startup):
  ```go
  loginjector.SetRuntimeDetailsProvider(func() string {
      return hostRuntimeDetails() // beacon's single-lined gopsutil block
  })
  if loginjector.RuntimeDetailsFrozen() {
      log.Printf("warning: runtime-details provider registered after first stack trace; detail will be missing")
  }
  ```
- **Acceptance criteria**:
  - `RuntimeDetailsFrozen()` returns `false` before any `StackTraceError` is constructed and
    `true` after the first `NewStackTraceError()`/`...f`/`Wrap...`.
  - The `-race` suite is green with a concurrent constructor + probe test.
  - godoc states the advisory (non-synchronizing, freeze-only) semantics â€” no "would take
    effect" language.
- **Pitfalls**: do not try to report "provider still effective" â€” that is the TOCTOU claim we
  are explicitly dropping. Keep the probe to the single observable fact (frozen yes/no).
- **Breaking**: no. **Complexity**: Small.

### Task 6 â€” `WithMaxAgeDays(int)` convenience
- **Description**: `WithMaxAge(time.Duration)` invites the `WithMaxAge(14)` == 14ns footgun;
  the godoc already warns about it (`handler.go:146-148`). Add a days-unit wrapper. Not
  consumed by beacon. **Note for the maintainer**: this doubles the age-option vocabulary and
  sets a precedent (someone will ask for `WithMaxAgeHours` next). Since the decision is to
  ship all tasks, keep it strictly minimal â€” a one-line delegation, no new config field, no
  new prune path.
- **Proposed API**:
  ```go
  // WithMaxAgeDays is WithMaxAge(time.Duration(days) * 24 * time.Hour) with an
  // unmistakable unit â€” it prunes rotated backups older than the given number of days.
  // See WithMaxAge for the full pruning semantics, mtime caveats, and the zero-disables
  // rule; this is purely a units-safe wrapper over it.
  func WithMaxAgeDays(days int) RotatingFileOption {
      return WithMaxAge(time.Duration(days) * 24 * time.Hour)
  }
  ```
- **Acceptance criteria**:
  - `WithMaxAgeDays(14)` produces the same `cfg.maxAge` as `WithMaxAge(14 * 24 * time.Hour)`
    (assert via a rotation test, or expose through an existing test seam).
  - `WithMaxAgeDays(0)` disables age pruning (delegates to the zero-disables path).
  - godoc cross-references `WithMaxAge` and does not restate the pruning mechanics.
- **Pitfalls**: keep it a pure delegate â€” do not add a second config field or a parallel
  prune branch; the existing `maxAge` path already handles everything.
- **Breaking**: no. **Complexity**: Small (trivial).

### Task 7 â€” `NewTraceErrorSkip(skip int)` + caller-frame regression test
- **Description**: `NewTraceError` hardcodes `internal.LineTrace(2)` (`error.go:21`). A future
  wrapper inserted between the public constructor and `LineTrace`, or a consumer wrapping the
  constructor, silently shifts every trace line to the wrong frame with no signal. Two
  deliverables: (a) export `NewTraceErrorSkip(skip int)` as the **generalized primitive that
  `NewTraceError` delegates to** â€” so it is not dead surface but the actual implementation
  seam; (b) a regression test that pins `NewTraceError()`'s resolved frame so an internal
  refactor that shifts the skip depth fails CI.
- **Proposed API + delegation**:
  ```go
  // NewTraceError creates a TraceError whose Line() is the caller's file:line (method).
  func NewTraceError() TraceError {
      return NewTraceErrorSkip(1) // +1 for this delegation hop
  }

  // NewTraceErrorSkip creates a TraceError whose Line() is resolved skip frames above
  // the caller of NewTraceErrorSkip. skip=0 is the direct caller; skip=1 is that
  // caller's caller. Use it when wrapping NewTraceError in a helper so the line still
  // points at your caller rather than the wrapper.
  func NewTraceErrorSkip(skip int) TraceError {
      return &traceError{line: internal.LineTrace(skip + 2)}
  }
  ```
  Skip arithmetic (verified against the source): `LineTrace` frame 0 is itself, 1 is
  `NewTraceErrorSkip`, 2 is `NewTraceErrorSkip`'s direct caller â€” so `LineTrace(skip+2)` with
  `skip=0` resolves the direct caller. `NewTraceError` adds one hop, so it passes `skip=1`,
  yielding `LineTrace(3)` = the consumer that called `NewTraceError` â€” identical to today's
  `LineTrace(2)` direct call. Behavior is preserved.
- **Required regression test** (encode these caveats):
  - Capture the expected line **dynamically** with `runtime.Caller(0)` on the same source
    line as the `NewTraceError()` call â€” do NOT hardcode a literal line number (it rots on
    every edit above it):
    ```go
    te := loginjector.NewTraceError(); _, file, line, _ := runtime.Caller(0)
    // te.Line() must reference filepath.Base(file) and strconv.Itoa(line)
    ```
  - Assert only the **base filename + line + method** substring â€” NOT the parent-dir prefix.
    `LineTrace` prepends `<parent>/`, and on CI the parent is the checkout directory name,
    which is environment-dependent and brittle. Assert `Contains(te.Line(), base+":"+line)`
    and that the method name is present; do not assert the leading directory.
  - Add a companion assertion that `NewTraceErrorSkip(0)` from the test resolves the test's
    own frame, documenting the primitive's contract.
- **Acceptance criteria**:
  - `NewTraceError().Line()` contains this test's base filename and the captured line number;
    an internal refactor that changes the skip depth breaks the test.
  - `NewTraceError` and `NewTraceErrorSkip` produce identical output for the equivalent frame.
- **Pitfalls**: `runtime.Caller(0)` and the `NewTraceError()` call must be on the **same
  source line** for the captured line to match the resolved line. Method-name assertion must
  tolerate the package-stripped form `LineTrace` produces (`(TestXxx)` style), not a fully
  qualified name.
- **Breaking**: no (additive export + behavior-preserving refactor). **Complexity**: Small.

### Task 8 â€” lazy stack capture (DEFER â€” not implemented this round)
- **Description**: `NewStackTraceError()` runs `internal.StackTrace()` eagerly (`error.go:83`).
  For a recover path that discards the error, a lazy variant (capture PCs via
  `runtime.Callers`, format on `Stack()`) would save the formatting cost.
- **Decision â€” deferred, and this holds even under "ship all":** `StackTrace()` is already a
  single-allocation 8 KiB buffer path (`internal/stacktrace.go:26-46`); the only caller in
  scope is beacon's single cold panic-recovery site, which always consumes the stack it
  captures. A lazy variant adds PC-slice retention and a format-on-read code path for a saving
  that never materializes on the one real call site. This is a genuine YAGNI defer, not
  deferred-by-omission â€” revisit only if a hot or discard-heavy full-stack usage appears.
- **Complexity**: Medium. **Verdict**: defer.

## Explicit non-changes (documented design â€” do NOT "fix") â€” re-verified

- **`NewStackTraceError().Error()` returns `""`** â€” deliberate and documented
  (`error.go:76-81`; `stackTraceError.Error()` returns `""` when `msg==""` and `cause==nil`,
  `error.go:245-256`). Compact `Error()`, full stack via `Stack()`;
  `NewStackTraceErrorf`/`WrapStackTraceError` give a non-empty chain string. **beacon does not
  need a change** â€” it calls `.Stack()`/`.Runtime()` directly and never relies on `Error()` in
  a `Join`. Reversing this is a breaking change against explicit design and stays out of
