# Task Breakdown

## Overview

The HTTP middleware (`NewHttpPayloadHandler`, `NewHttpPayloadHandlerWithOptions`, `NewHttpAccessHandler`, the options DSL, the summary-writer global, the interceptor, the eight `*RW` interface-passthrough wrappers, and the redaction helpers) currently lives at the module root and forces every consumer of the `loginjector` core (a fan-out logger) to pull in `net/http`, `net`, `bufio`, and an entire HTTP-tap subsystem they may not need. Move it to a sibling sub-package `github.com/prorochestvo/loginjector/httptap`, matching the stdlib `net/http/httputil`/`httptest`/`httptrace` naming pattern. After the move the root package logs; `httptap` taps HTTP traffic into a root `*Logger`. Import direction is `httptap` → `loginjector` (the sub-package needs `*Logger` and `LogLevel`); the root never imports `httptap`.

Per the locked design decisions: this is a deliberate **breaking change in 1.x**. No deprecation shim lives in the root (any such shim would re-import `httptap` and create an import cycle). All `Http`-prefixed identifiers lose the prefix on the move (stutter removal: `httptap.NewPayloadHandler` reads, `httptap.NewHttpPayloadHandler` stutters). The break is announced in a new `CHANGELOG.md`; the README and CLAUDE.md examples are rewritten to point at the new API. The user is the sole consumer of this library and has explicitly accepted the cost.

Public API rename map (the **only** signature-level changes; behaviour and log output stay byte-identical):

| Old (root)                         | New (`httptap`)                                     |
| ---------------------------------- | --------------------------------------------------- |
| `NewHttpPayloadHandler`            | `httptap.NewPayloadHandler`                         |
| `NewHttpPayloadHandlerWithOptions` | `httptap.NewPayloadHandlerWithOptions`              |
| `NewHttpAccessHandler`             | `httptap.NewAccessHandler`                          |
| `HttpPayloadOption`                | `httptap.PayloadOption`                             |
| `WithMaxRequestBody` etc.          | `httptap.WithMaxRequestBody` etc. (move, no rename) |
| `SetHttpSummaryWriter`             | `httptap.SetSummaryWriter` (see Task 8 / open question) |

Final file layout inside `httptap/`:

```
httptap/
  doc.go            # package comment + brief example
  payload.go        # NewPayloadHandler, NewPayloadHandlerWithOptions, options DSL, payloadConfig
  access.go         # NewAccessHandler + accessResponseWriter
  interceptor.go    # interceptor + cappedWriter + buildBodySnippet (all unexported)
  rw.go             # hijackMarker + 8 *RW wrappers + wrapWithOptionalInterfaces (all unexported)
  redact.go         # headerToStringWithRedact + defaultRedactHeaders (unexported)
  summary.go        # SetSummaryWriter + summary-writer global (Task 8: may be folded into payload.go)
  *_test.go         # tests split along the same axes
```

## Assumptions

- Module path is `github.com/prorochestvo/loginjector` (confirmed in `go.mod`); the new package's import path is therefore `github.com/prorochestvo/loginjector/httptap`. No `go.mod` edits required.
- Grep confirms the only Go files referencing the moved symbols (`Http*Handler`, `httpSummaryWriter`, `httpSummaryOutput`, `hijackMarker`, `wrapWithOptionalInterfaces`, `interceptor`, `httpPayloadConfig`, `cappedWriter`, `buildBodySnippet`, `headerToStringWithRedact`, the `With*` options) are `middleware.go`, `middleware_access.go`, `middleware_test.go`, `middleware_access_test.go`. No other root files import these, so the move is local: delete from root and re-add under `httptap/`.
- `httptap → loginjector` is the only legal import direction. `loginjector → httptap` would cycle (httptap needs `*Logger` and `LogLevel` from root). No shim layer in root is therefore possible — and none is wanted (locked decision #3).
- `levels/` keeps its current name (different naming axis: it ships value constants, not a transport tap). It is unaffected.
- `internal/` stays for stack-trace parsing; do not move it, do not touch it.
- Tests that exercise private internals (interceptor, cappedWriter, buildBodySnippet, `*RW` wrappers, the package-global summary writer) stay in `package httptap` (white-box). Tests that exercise only the public surface MAY move to `package httptap_test` (black-box) if the engineer judges it cleaner; if in doubt, keep all moved tests in `package httptap` to minimise refactor surface area on the first pass.
- `CHANGELOG.md` does not yet exist in this repo; this plan creates it.
- This change targets the next breaking minor inside `1.x` (the user has accepted the in-1.x break trade-off rather than a v2 module rename).
- `middleware.go` currently retains a legacy type-alias block at the bottom (`flusherInterceptor = flusherRW`, …) purely so `TestWrapInterceptor` keeps compiling. Once the test moves to `httptap/` and is updated to use the `*RW` names directly, those aliases die with the file — they do not need to be carried forward.

### Task 8: Decide the fate of the summary-writer global (RESOLVED → Variant B)

- **Description:** The global `SetHttpSummaryWriter` (Task 4 renames it to `SetSummaryWriter`) is a legacy hook. Its functionality is a strict subset of the per-handler option `WithSummaryWriter(w)`, which already exists and takes precedence over it. The plan calls this out as a discrete decision the engineer **must not make unilaterally**. Two execution variants:

  **Variant A — Keep the global (default; safe).** `httptap/summary.go` exists as in Task 4; `SetSummaryWriter` is public; the precedence rule (per-handler `WithSummaryWriter` > global `SetSummaryWriter` > `os.Stdout`) is preserved verbatim from the source. `TestSummaryWriter` keeps all six subtests. This variant is one move + one rename; no behaviour change.

  **Variant B — Drop the global.** Delete `httptap/summary.go` before Task 4 lands. The package-level `summaryMu`, `summaryOutput`, and `summaryWriter()` helper disappear; `cfg.summaryWriter` becomes the sole knob. Inside `NewPayloadHandlerWithOptions`'s defer, the precedence collapses to: per-handler `cfg.summaryWriter` > `os.Stdout` (no middle tier). `TestSummaryWriter` loses three subtests (`summary_is_written_to_configured_writer` which tests the global directly, `unset_uses_global_default`, `global_unchanged_after_handler_built`) — these are deleted, not rewritten. `option_overrides_global` is rewritten as `option_overrides_default` (asserts the per-handler writer is used over `os.Stdout`). `discard_silences_summary` keeps its assertion shape. `no_options_equivalent_to_legacy_constructor` loses the `SetHttpSummaryWriter`/`SetSummaryWriter` setup but keeps the byte-equivalence assertion. The `body_cap_race_condition` test no longer needs `WithSummaryWriter(io.Discard)` to avoid global races (but keeping it does no harm). README/CLAUDE.md (Task 10) drop the `SetSummaryWriter` mention entirely.

  **Pros / cons summary** (for the user to decide):
  - Variant A keeps the public surface a strict rename of the existing one — easier mental model, no second behaviour delta to document in CHANGELOG.
  - Variant B genuinely shrinks the API by one global and removes the package-level mutex / mutable writer — better for a library, and the option already does the job. The only "loss" is process-wide one-call configuration, which nobody is using because the user is the sole consumer and configures per-handler.
  - This is a one-way break either way (Task 7 already breaks all the old `Http`-prefixed names); folding the global deletion into the same breaking version is cheaper than doing it later.
  - Maintainer's bias (per the user prompt) leans towards drop; final call sits with the user.
- **Acceptance Criteria:**
  - [x] RESOLVED: Variant B. Maintainer confirmed via prompt before execution. Zero downstream consumers of `SetHttpSummaryWriter` verified by external grep.
  - [x] `httptap/summary.go` is never created. The three deleted subtests (`summary_is_written_to_configured_writer`, `unset_uses_global_default`, `global_unchanged_after_handler_built`) are not rewritten. `SetSummaryWriter` does not appear in any code, comment, README, or CHANGELOG except as a one-line "removed" entry under Breaking changes.
- **Pitfalls & edge cases:** If the engineer picks unilaterally, Task 8 is failed and re-review must reject. Variant B is **not** a refactor-as-you-go — the call has to be made before code is touched, because the test surface differs. ## Trade-offs

- **No shim in root vs documented break.** Locked decision #3. A shim is impossible (cycle) and we accept the break loudly in CHANGELOG. Cost: any consumer beyond the user has a one-line-per-call-site migration. Benefit: the middleware is now opt-in rather than always-pulled, and the public surface stops stuttering.
- **Stutter-removal rename (`NewHttpPayloadHandler` → `httptap.NewPayloadHandler`).** Locked decision. Cost: every README example and downstream call site changes. Benefit: idiomatic Go (no `httptap.HttpFoo` because the package name already says `http`).
- **Flat layout inside `httptap/` (no nested `internal/`).** Decision #4. The interceptor, RW wrappers, and redact helpers are unexported within `httptap`, which is the same encapsulation strength as a nested `internal/`. Flat layout means one less directory hop and zero re-exports. Trade-off: a future `httptap_v2` would need to be a sibling rather than a swap, but YAGNI.
- **Tests stay in `package httptap` (white-box).** They poke at `interceptor`, `cappedWriter`, `buildBodySnippet`, `defaultPayloadConfig`, `summaryOutput`, and the `*RW` types — all unexported. A `package httptap_test` (black-box) split would require either exporting these (pollutes API) or rewriting half the tests to drive through the public surface (large incidental refactor). White-box keeps the diff to a rename.
- **Variant A (keep the global) vs Variant B (drop it).** Open question, Task 8. Recommend B (one less piece of mutable global state, the option is a strict superset), but the user has final say. The recommendation costs nothing structural — Variant A is fully spec'd as the fallback.
- **One bundled commit vs many small commits.** Decision: source moves (Tasks 2–5) in one commit, test move (Task 6) in one, breaking delete (Task 7) in one, docs (Tasks 9–10) in one. This keeps each commit individually buildable (after Task 6, source-side green; after Task 7, root-side green) and the breaking commit isolated for `git revert` if needed. Not splitting the source move further because the four files are interdependent and partial commits are not buildable.
