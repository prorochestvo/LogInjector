# Task Breakdown

## Overview

Add a lightweight, production-grade HTTP access-log middleware to complement the existing
`NewHttpPayloadHandler` / `NewHttpPayloadHandlerWithOptions`. The existing handler is a
debug-oriented full-dump (request line, headers with redaction, request body, response
status, response headers, response body, plus a per-request summary). It allocates two
buffers per request, tees the request body, dispatches to `*Logger` (with the logger's
own goroutine fan-out), and is too expensive for hot prod paths. The new handler is a
single-line access log targeting a plain `io.Writer`: status, method, path, duration —
nothing else. No body capture, no header dumping, no goroutines, no `*Logger` coupling.
Patterned after the lightweight middleware in `fx_rate_monitor/internal/gateway/middleware`,
plus the duration field that fx_rate_monitor lacks. Both handlers coexist; the existing
`NewHttpPayloadHandler` keeps its name and behaviour unchanged for backward compatibility.

## Assumptions

- The library remains pure Go, `CGO_ENABLED=0`, no new dependencies.
- Backward compatibility is non-negotiable: `NewHttpPayloadHandler`,
  `NewHttpPayloadHandlerWithOptions`, and every `HttpPayloadOption` keep their current
  signatures and behaviour byte-for-byte.
- Consumers wanting the new handler routed into `*Logger` will compose it themselves via
  the existing `logger.WriterAs(level)` adapter; the new handler does not take `*Logger`.
- Query strings, client IP, request IDs, request/response sizes, user agent, referer,
  and any header capture are explicitly out of scope for the first cut. Adding them later
  through additive `...Option` overloads is acceptable and does not require API breakage.
- Optional `http.ResponseWriter` interfaces — `http.Flusher`, `http.Hijacker`,
  `http.Pusher` — must continue to work for downstream handlers (SSE, websockets,
  HTTP/2 push). Exotic interfaces (`io.ReaderFrom`, `http.CloseNotifier`) remain
  unsupported, matching the existing handler.
- Status `0` from a downstream handler that writes a body without calling `WriteHeader`
  is normalised to `200`, mirroring `net/http`'s own defaulting behaviour and the
  existing interceptor. Status truly stays `0` only when the handler hijacks without
  ever writing — that case is rendered as `000` in the log (see Task 3).
- The new handler lives in the root package alongside the existing middleware, in a
  new file `middleware_access.go`. The shared optional-interface helper extracted in
  Task 1 stays in `middleware.go` next to the existing composite types.

## Tasks

### Task 1: Refactor the optional-interface composite types around a shared base

- Description: Today `wrapInterceptor` in `middleware.go:568` returns one of eight
  composite types parameterised over `*interceptor`. The new access-log handler needs
  the same eight-way switch over a *different* base (a lightweight status-only wrapper,
  added in Task 2). Duplicating eight types plus the switch would double the surface
  and leave two near-identical hijack-flag paths that can drift. Decision: extract the
  composites and the switch around a small unexported marker interface that both bases
  satisfy.

  Concretely:
  - Add an unexported interface in `middleware.go` that captures the only behaviour the
    composites need from their base beyond `http.ResponseWriter`:
    ```go
    // hijackMarker is implemented by interceptor types that need to record when a
    // Hijack has happened so callers can suppress post-handler writes that assume the
    // ResponseWriter is still live (Flush in the payload handler; no-op in the access
    // handler — the log line still emits).
    type hijackMarker interface {
        http.ResponseWriter
        markHijacked()
    }
    ```
  - Add `markHijacked()` to `*interceptor` (sets the existing `hijacked` field) and to
    the new wrapper from Task 2. Both already embed `http.ResponseWriter`, so the
    interface is satisfied automatically modulo the new method.
  - Replace the eight `*interceptor`-typed composite structs (`flusherInterceptor`,
    `hijackerInterceptor`, …, `allThreeInterceptor` at `middleware.go:472-563`) with
    eight composite structs parameterised over `hijackMarker`. Rename them to drop the
    `interceptor` suffix — they're no longer interceptor-specific:
    `flusherRW`, `hijackerRW`, `pusherRW`, `flusherHijackerRW`, `flusherPusherRW`,
    `hijackerPusherRW`, `allThreeRW`. Each embeds a `hijackMarker` field named `base`
    and stores the optional-interface delegates as before.
  - Each `Hijack()` method calls `v.base.markHijacked()` then delegates to its hijacker,
    same shape as today.
  - Replace `wrapInterceptor(i *interceptor, w http.ResponseWriter) http.ResponseWriter`
    with `wrapWithOptionalInterfaces(base hijackMarker, w http.ResponseWriter) http.ResponseWriter`.
    Same eight-way switch, same ordering, same "underlying has none → return base
    directly" fallback. `newInterceptorWithConfig` at `middleware.go:252` updates its
    single call site to use the new name.
  - Keep the helper, the marker interface, and the composites in `middleware.go` (not
    a new file); they belong with the existing handler conceptually and there is no
    win in splitting them out.

- Acceptance Criteria:
  - [ ] `NewHttpPayloadHandler` / `NewHttpPayloadHandlerWithOptions` behaviour is
    byte-identical: every test in `middleware_test.go` passes unmodified, including the
    Flusher / Hijacker / Pusher pass-through tests and the post-handler `Flush after
    hijack` guard at `middleware.go:151`.
  - [ ] The eight composite types and the switch live in exactly one place; no
    duplication for the new handler.
  - [ ] No new exported symbols. `hijackMarker`, `markHijacked`, the renamed composites,
    and `wrapWithOptionalInterfaces` are all unexported.
  - [ ] `go vet ./...` and `gofmt -l .` are clean.

- Pitfalls & edge cases:
  - The renamed composites must keep the existing case ordering in the switch
    (most-specific first). Reordering changes nothing functionally, but the diff is
    easier to review if order is preserved.
  - `markHijacked()` on `*interceptor` must set the same `hijacked` field the existing
    `defer` at `middleware.go:151` reads; do not introduce a parallel flag.
  - `wrapWithOptionalInterfaces` must still return the bare `base` (as
    `http.ResponseWriter`) when the underlying writer has none of the three optional
    interfaces — the zero-allocation fast path. Don't allocate a composite struct in
    that branch.
  - Embedding `hijackMarker` (an interface) inside the composite structs would let the
    composite satisfy `http.ResponseWriter` "for free" via method promotion. Prefer a
    *named* field (`base hijackMarker`) and write the four-line `Header() / Write() /
    WriteHeader()` forwarders explicitly only if Go's method promotion through an
    interface field does not work for you here — it does, and the existing code
    already relies on it (`*interceptor` is embedded by pointer today). So: keep
    embedding, just embed the interface instead of the concrete pointer.

- Complexity: Medium

- Code Example:
  ```go
  type hijackMarker interface {
      http.ResponseWriter
      markHijacked()
  }

  func (i *interceptor) markHijacked() { i.hijacked = true }

  type flusherHijackerRW struct {
      hijackMarker
      flusher  http.Flusher
      hijacker http.Hijacker
  }

  func (v *flusherHijackerRW) Flush() { v.flusher.Flush() }
  func (v *flusherHijackerRW) Hijack() (net.Conn, *bufio.ReadWriter, error) {
      v.hijackMarker.markHijacked()
      return v.hijacker.Hijack()
  }

  func wrapWithOptionalInterfaces(base hijackMarker, w http.ResponseWriter) http.ResponseWriter {
      f, hasF := w.(http.Flusher)
      h, hasH := w.(http.Hijacker)
      p, hasP := w.(http.Pusher)
      switch {
      case hasF && hasH && hasP:
          return &allThreeRW{hijackMarker: base, flusher: f, hijacker: h, pusher: p}
      // ... seven more, same ordering as today ...
      default:
          return base
      }
  }
  ```

### Task 2: Implement the lightweight status-capturing wrapper

- Description: Add a small `http.ResponseWriter` wrapper in `middleware_access.go` that
  records only the status code and a "header was set" flag. No body buffering, no
  request body tee, no config struct. Counterpart of `fx_rate_monitor`'s
  `httpResponseWriter` from `internal/gateway/middleware/logger.go:36-56`, sitting
  alongside the existing fat `interceptor` rather than replacing it.

  Struct shape:
  ```go
  type accessResponseWriter struct {
      http.ResponseWriter
      statusCode  int
      wroteHeader bool
      hijacked    bool
  }
  ```
  Implements:
  - `WriteHeader(code int)` — records `code` on first call only; subsequent calls are
    silent no-ops and do not forward to the underlying writer a second time. Mirrors
    `net/http`'s "WriteHeader more than once" handling and `fx_rate_monitor`'s wrapper.
  - `Write(b []byte) (int, error)` — if `!wroteHeader`, calls
    `WriteHeader(http.StatusOK)` first, then forwards `b` to the underlying writer
    unchanged. Returns the underlying writer's `(n, err)` unchanged.
  - `markHijacked()` — sets `hijacked = true`. Satisfies the `hijackMarker` interface
    from Task 1.

  Compile-time assertions in the same file:
  ```go
  var _ http.ResponseWriter = (*accessResponseWriter)(nil)
  var _ hijackMarker = (*accessResponseWriter)(nil)
  ```

- Acceptance Criteria:
  - [ ] `accessResponseWriter` is unexported, declared at the bottom of
    `middleware_access.go` per the "public surface on top, internals at bottom" memory
    rule.
  - [ ] `WriteHeader(code)` records on the first call only; second and later calls are
    silent no-ops (`wroteHeader` gate, then early return).
  - [ ] `Write(b)` triggers an implicit `WriteHeader(http.StatusOK)` if no header was
    set, then forwards `b` unchanged. Return value matches the underlying writer.
  - [ ] When fed through `wrapWithOptionalInterfaces` from Task 1, `http.Flusher`,
    `http.Hijacker`, `http.Pusher` are transparently exposed iff the underlying writer
    has them.
  - [ ] `markHijacked()` sets `hijacked` so the field is queryable from the deferred
    log path in Task 3 if needed (currently the field is not read by Task 3 — see Task
    3's "hijacked path" note — but having it set is correct semantics and costs
    nothing).

- Pitfalls & edge cases:
  - Don't double-fire `WriteHeader` when `Write` is called after an explicit
    `WriteHeader`: the implicit branch is gated on `!wroteHeader` only.
  - Do not call `WriteHeader` from the embedded `http.ResponseWriter` again on the
    duplicate path — the upstream stdlib `http` server will already log a warning if
    you do; the wrapper's job is to swallow the duplicate.
  - `statusCode` is zero before any `WriteHeader`. Task 3 renders this as `000` for
    the hijacked-without-write case; do *not* default it to 200 in the struct.

- Complexity: Easy

### Task 3: Implement `NewHttpAccessHandler`

- Description: Add a new exported constructor in `middleware_access.go`:
  ```go
  func NewHttpAccessHandler(out io.Writer, next http.HandlerFunc) (http.HandlerFunc, error)
  ```

  Behaviour:
  - Validate `out != nil` and `next != nil`; return `(nil, fmt.Errorf("out is nil"))`
    / `(nil, fmt.Errorf("next is nil"))` respectively. Matches the style of
    `NewHttpPayloadHandlerWithOptions` returning an error on `logger == nil`.
  - Per request:
    1. Snapshot `r.Method`, `r.URL.Path`, and `time.Now()` *before* calling `next`.
       Snapshotting `Path` matters because downstream middleware may rewrite it; we
       log what came in, not what the handler routed to. (The existing payload
       handler logs the rewritten path; the new handler intentionally diverges here
       because access logs are more useful when they describe the inbound URL.)
    2. Wrap the response writer:
       ```go
       arw := &accessResponseWriter{ResponseWriter: w}
       wrapped := wrapWithOptionalInterfaces(arw, w)
       ```
    3. `defer` the log emission (see format below).
    4. Call `next(wrapped, r)`.
  - Log line format (locked, single line, terminated with `\n`):
    ```
    2026-05-30T14:23:11.123456789Z [200] GET /api/v1/rates 12.345ms
    ```
    Built with one `fmt.Fprintf` call:
    ```go
    fmt.Fprintf(out, "%s [%03d] %s %s %0.3fms\n",
        time.Now().UTC().Format(time.RFC3339Nano),
        status, method, path,
        float64(elapsed)/1e6,
    )
    ```
    - Timestamp: `time.Now().UTC().Format(time.RFC3339Nano)`. No `log.Logger`
      wrapper — `log.Logger` takes a mutex per call (free serialisation) but uses
      `LstdFlags` (second-precision local time), which is the wrong trade-off for a
      production access log. Manual UTC RFC3339Nano is cheaper and more useful in
      log aggregation. See the concurrency note in godoc below.
    - Status: `%03d`. Zero-padded to three digits for column alignment and grep
      friendliness (`grep ' \[5[0-9][0-9]\] '`). Matches the fx_rate_monitor
      reference at `internal/gateway/middleware/logger.go:29`.
    - Method / path: `%s` / `%s`, no quoting, no escaping. If a caller routes a path
      containing a space or newline, that's their problem and matches stdlib
      `net/http` behaviour.
    - Duration: `%0.3fms` with `float64(elapsed)/1e6`. Fixed unit (always
      milliseconds), three decimal places. Chosen over `time.Duration.String()`
      because: (a) `Duration.String` allocates more (it goes via
      `appendInt`/`appendFrac` into a stack buffer then to string conversion), and
      (b) variable units (`12.3µs` vs `12.3ms` vs `1.2s`) break column alignment and
      complicate log aggregation. Consistent with the existing summary writer at
      `middleware.go:137`.
  - Hijacked path: the log line still emits with whatever `statusCode` was captured.
    If the handler hijacked without writing, `statusCode` is `0`, rendered as `000` —
    an unambiguous signal of "request happened, status unknown / hijacked". Do *not*
    suppress the log. There is no equivalent of the payload handler's post-handler
    `Flush` to skip; the deferred log call is the *only* thing that runs after
    `next`, so the `hijacked` flag does not gate anything in this handler. The flag
    is still set (Task 2) for symmetry and future-proofing.
  - If `next` panics, the `defer` still fires and emits the line with whatever status
    was last captured. Do *not* `recover` — let the panic propagate up to whatever
    panic-handling middleware sits above us. This matches how access logs work in
    every other framework and matches the payload handler.

  Godoc (verbatim wording — keep this short, no marketing fluff):
  ```go
  // NewHttpAccessHandler returns an HTTP middleware that emits one line per request
  // in the form:
  //
  //   2026-05-30T14:23:11.123456789Z [200] GET /api/v1/rates 12.345ms
  //
  // It is the production access-log variant: status, method, path, duration; no
  // header dumping, no body capture, no goroutine fan-out, no *Logger coupling.
  // For the debug full-dump variant see NewHttpPayloadHandler.
  //
  // out is the destination for log lines. The handler writes synchronously; out
  // MUST be safe for concurrent Write calls (an *os.File or os.Stdout qualifies;
  // a raw *bytes.Buffer does not — guard it externally). The status field is
  // zero-padded to three digits; duration is rendered in milliseconds with three
  // decimal places. Path is r.URL.Path captured before next runs, so downstream
  // path rewrites do not affect the logged value.
  //
  // out and next must both be non-nil; otherwise a descriptive error is returned
  // and the handler is nil.
  //
  // Format is intentionally fixed. Anything beyond these four fields (request ID,
  // client IP, JSON output, structured fields) must go through a separate
  // ...WithOptions constructor in a future change, not by mutating this signature.
  ```

- Acceptance Criteria:
  - [ ] Compiles under `CGO_ENABLED=0 go build ./...`.
  - [ ] `out == nil` returns `(nil, error)`. `next == nil` returns `(nil, error)`.
    Both errors mention the offending parameter by name.
  - [ ] Default 200 when `next` calls `Write` without `WriteHeader`: line contains
    `[200]`.
  - [ ] Explicit non-200 status is captured: e.g. `WriteHeader(404)` → `[404]`.
  - [ ] Repeat `WriteHeader` is ignored: first call wins.
  - [ ] Query string is NOT in the logged path: `/x?secret=1` logs `/x`.
  - [ ] Timestamp matches `^[0-9]{4}-[0-9]{2}-[0-9]{2}T[0-9]{2}:[0-9]{2}:[0-9]{2}(\.[0-9]+)?Z`.
  - [ ] Duration field matches `[0-9]+\.[0-9]{3}ms`.
  - [ ] Hijacked-without-write emits `[000]` and still appends the line.
  - [ ] Underlying writer's optional interfaces (`Flusher`, `Hijacker`, `Pusher`)
    are transparently exposed via the wrapper produced by Task 1.
  - [ ] `go vet ./...` and `gofmt -l .` are clean.

- Pitfalls & edge cases:
  - `r.URL.Path` is captured by value (string) into a local before `next` runs — do
    not read it from `r.URL.Path` inside the `defer`, or downstream rewrites silently
    change the log.
  - `time.Now().UTC()` in both the `defer` and the start snapshot — using `.UTC()`
    once and `time.Since` for the elapsed avoids the location-allocation surcharge.
    Prefer `start := time.Now()` and `elapsed := time.Since(start)`; only the
    *printed* timestamp needs `.UTC().Format(...)`.
  - `out` must be safe for concurrent `Write`. `*os.File` is fine (POSIX `write(2)`
    on a single FD is atomic for line-sized writes, and the runtime serialises
    `os.File.Write` via its file lock). `*bytes.Buffer` is *not* fine — the test
    suite (Task 4) must guard the buffer with a mutex; that is the test's problem,
    not the handler's. Document this in godoc (already in the verbatim block above).
  - Don't pre-allocate a per-request scratch buffer for the line. The whole point is
    that `fmt.Fprintf` straight into `out` is the cheapest path; introducing a
    `bytes.Buffer` pool would add complexity without measurable gain at one
    Fprintf-per-request.

- Complexity: Easy

## Execution Order

1. **Task 1** — refactor the eight composite types and the switch onto the shared
   `hijackMarker` interface. Doing this first lets Task 2's wrapper plug in cleanly
   and confines the highest-risk change (touching the payload handler's
   optional-interface path) to a single dedicated task.
2. **Task 2** — `accessResponseWriter`. Smallest unit; verifiable in isolation against
   the wrapper test fakes.
3. **Task 3** — `NewHttpAccessHandler`, composing Tasks 1 and 2.
4. **Task 4** — tests + benchmark. Engineer may write the table of subtests in
   parallel with Task 3 if they prefer table-driven test-first; the plan ordering
   lists tests after implementation to make code review easier.
5. **Task 5** — doc passes. Last because godoc tends to track the final shape of code.

Tasks 1 and 2 can technically be developed in parallel by a single engineer (Task 2
does not depend on Task 1 *compiling*, only on the `hijackMarker` interface signature,
which is decided in this plan), but committing Task 1 first keeps the diff to the
existing payload handler isolated and easy to revert.

## Risks

- **Task 1 refactor silently breaks the payload handler.** The eight composite types
  and the `hijacked` flag are the subtlest code in the existing handler. Mitigation:
  every test in `middleware_test.go` must pass unchanged at every commit; do not
  modify those tests as part of Task 1. If the refactor turns ugly (e.g. method
  promotion through the embedded `hijackMarker` field misbehaves in some edge case),
  the fallback is to keep the eight payload composites as-is and add a parallel set
  of seven composites for the access handler. That fallback doubles the surface but
  preserves correctness; do not ship a half-refactor.
- **`out` writer is not thread-safe.** The handler writes synchronously without an
  internal lock. If a caller passes a raw `*bytes.Buffer` and serves concurrent
  requests, the logs will interleave or panic. Mitigation: spelled out in godoc;
  the race test in Task 4 uses a `lockedWriter` to make this requirement visible.
- **Future option creep.** Once shipped, requests for "just add IP" / "just add
  request ID" / "just add JSON format" will arrive. The handler's signature
  `NewHttpAccessHandler(out, next)` is intentionally bare. Additive growth goes
  through a parallel `NewHttpAccessHandlerWithOptions` constructor like the payload
  handler did — *not* by changing the existing signature. Stance documented in godoc.
- **Format lock-in.** Picking `<RFC3339Nano> [%03d] METHOD PATH %0.3fms` as a
  non-configurable format means anyone wanting logfmt / JSON / Apache combined rolls
  their own. Explicit trade-off; flagged in godoc so no one is surprised.
- **Allocation regression.** The handler is sold on cost. The Task 4 benchmark with
  `b.ReportAllocs()` and the ≤ 5 allocs/op budget catches innocent-looking changes
  (e.g. someone adding a `bytes.Buffer` to "pre-format" the line) that would erase
  the perf win.

## Trade-offs

- **`io.Writer` instead of `*Logger`.** Consumers wanting fan-out, level routing,
  hooks, or any other `*Logger` feature must pipe through `logger.WriterAs(level)`.
  Cost: one extra line at wire-up. Benefit: handler is usable without our `*Logger`,
  with stdlib `log.Default().Writer()`, with `lumberjack`, or with anything else
  implementing `io.Writer`. Net win for a production handler.
- **Manual timestamp instead of `log.Logger`.** `log.Logger.Output` would give free
  per-call serialisation (its internal mutex), saving the godoc thread-safety
  caveat. We chose manual UTC RFC3339Nano formatting because: (a) `LstdFlags` is
  second-precision local time, which is the wrong default for a production access
  log; (b) the manual path is fewer allocations and one less import dependency on
  `log`; (c) the thread-safety burden moves to the caller, which is honest — most
  real `out` writers (`*os.File`, `os.Stdout`) are already safe.
- **No client IP, no request ID, no query string.** Each has a real defensible use
  case, but each also has a real footgun (IP behind NAT/proxy, request ID header
  name varies, query string leaks tokens). Shipping bare keeps the surface honest;
  adding them later as opt-in options is cheaper than backing them out once shipped.
- **Refactor (Task 1) over duplication.** Eight composite types times two handlers
  is sixteen types; two near-identical switches drift over time. The `hijackMarker`
  interface is a four-line surface that both bases satisfy trivially, with zero
  extra allocations on the hot path. The fallback to duplication (see Risks) exists
  only if the refactor surfaces a behaviour delta we cannot fix.
- **No goroutine for the log write.** The existing payload handler dispatches the
  write through `*Logger`'s goroutine-per-handler fan-out, which lets the request
  finish before the log is durably written. The new handler writes synchronously.
  If `out` is slow (full pipe, dying disk), every request stalls on it. Acceptable
  for an access log — slow `out` is an ops problem, not a middleware problem — and
  the existing fat handler stays available for callers who insist on async via
  `*Logger`.
- **Status `0` rendered as `000` for hijacked-without-write.** Could have rendered
  as `---` or `-` for human readability, but `000` keeps the field at a fixed
  three-character width and matches the `%03d` format used elsewhere. Trivially
  greppable (`grep ' \[000\] '` = "find hijacked requests").
