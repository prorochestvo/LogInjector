# Task Breakdown ‚Äî Fix Swallowed Errors & Correctness Bugs

## Overview

`github.com/prorochestvo/loginjector` is a public, dependency-light Go logging
library. A logging library that silently eats errors is a fire alarm with the
battery removed, so this plan's headline is the four `errors.Join` no-op bugs in
`json.go`. Around them sit eight more verified findings: a `println` that prints a
pointer instead of a message, two dead regexes, an HTTP middleware that reports
status `0` for the implicit-200 path, a file-rotation handler that forgets the
on-disk size after a restart, a couple of ignored-error nits, a doc/behavior
mismatch on `Fatal*`, ceremonial signatures, and the goroutine-per-write hot path.

The work splits into three buckets:

1. **Real error-swallowing / correctness bugs** (findings 1, 2, 4, 6) ‚Äî fix the
   behavior, prove it with a test.
2. **Low-risk cleanups and doc fixes** (findings 3, 5, 8b, 7) ‚Äî no behavior change
   visible to consumers; tighten code and docs.
3. **Performance** (finding 9) ‚Äî benchmark first, then a conservative inline
   fast-path for the 0/1-sink case, leaving the concurrency model intact.

Two findings (7 `Fatal*` semantics, 8a `NewLogger` error return) touch **public
API stability**. The default decision for both is *do not break the API* ‚Äî fix the
docs, leave the signatures. Those decisions are surfaced explicitly in Trade-offs
for the user to confirm or override.

This is a planning document only. No Go code is written here; the snippets are
illustrative sketches for the implementing engineer.

## Assumptions

- API stability is a hard constraint: any change to an exported signature or to
  documented runtime behavior is breaking and must be flagged, not done silently.
  Where a finding asks "should we break X", the plan's default answer is "no, fix
  the docs instead" unless the user says otherwise (findings 7, 8a).
- The concurrency model in `WriteLog` (goroutine-per-sink + `WaitGroup`) is load
  bearing ‚Äî the `‚Ä¶ForRaceCondition` tests exercise it. The perf task only adds a
  fast-path; it does not remove the concurrent path.
- `bytes.Buffer.Write` never returns an error in practice (its godoc guarantees a
  nil error). Findings 5 is therefore a convention/lint fix, not a behavior fix ‚Äî
  it must not change the happy path.
- Tests run with `CGO_ENABLED=0 go test -race ./...`; that is the gate.
- `internal.LineTrace()` already parses stack traces by hand; the regexes at
  `internal/stacktrace.go:11-12` are confirmed unused (grep returns only their own
  declarations). Deleting them changes no behavior.
- For new/refactored tests, follow the "one `Test*` per method, scenarios as
  subtests" rule. Existing tests that predate the rule are left as-is unless a fix
  forces a touch.
- `println` (the builtin) writes to **stderr**, not stdout ‚Äî relevant to how
  finding 2 is tested.

## Tasks

### Task 1: Fix the four `errors.Join` no-op Close bugs in `json.go`

- **Description:** All four close-after variants drop the `Close()` error on the
  floor. At `json.go:24`, `:30`, `:52`, `:59` the deferred func calls
  `errors.Join(err, c.Close())` and discards the result. `errors.Join` returns a
  *new* error; it does not mutate the named return `err`. The fix is to assign the
  result back to the named return inside the defer:
  `err = errors.Join(err, c.Close())`. Affected functions:
  `JsonEncodeAndClose`, `JsonDecodeAndClose`, `JsonEncodeAndCloseEx`,
  `JsonDecodeAndCloseEx`. This is the headline bug ‚Äî a logging library silently
  losing errors.
- **Acceptance Criteria:**
   - [ ] All four defers in `json.go` assign back: `err = errors.Join(err, c.Close())`.
   - [ ] A test constructs a `WriteCloser`/`ReadCloser` whose `Close()` returns a
         sentinel error and asserts the returned `err` wraps it
         (`errors.Is(err, sentinel)` is true) for each of the four variants.
   - [ ] A test asserts that when the operation itself errors AND `Close()` errors,
         the returned error wraps **both** (operation error + close error).
   - [ ] `JsonEncodeAndCloseEx` / `JsonDecodeAndCloseEx`: the `raw []byte` return is
         still populated as before when only the close errors (i.e. the body still
         encoded/decoded; close error is additive, not replacing).
   - [ ] Existing `TestJsonEncodeAndClose*` / `TestJsonDecodeAndClose*` tests still
         pass unchanged.
   - [ ] `CGO_ENABLED=0 go test -race ./...` green.
- **Pitfalls & edge cases:**
   - The named returns are `err` (and `raw` for the Ex variants) ‚Äî the assignment
     must target the named return, not a shadowed local. Don't introduce `:=`.
   - Order matters for joined errors: the operation error comes first, the close
     error second (`errors.Join(err, c.Close())`), matching the existing handler
     pattern in `handler.go:103`. Keep it consistent.
   - Do not change the close-on-success behavior: the closer must still close even
     when the body op succeeded.
   - The mock closers in tests are new stub structs ‚Äî add a compile-time interface
     assertion (`var _ io.WriteCloser = (*errCloser)(nil)`).
- **Complexity:** Easy
- **Code Example:**
  ```go
  func JsonEncodeAndClose(w io.WriteCloser, v any, indent ...int) (err error) {
      defer func(c io.Closer) { err = errors.Join(err, c.Close()) }(w)
      err = JsonEncode(w, v, indent...)
      return
  }
  ```

  pointer pair like `(0x.., 0x..)`), not the error text. Fix: `println(err.Error())`

- **Acceptance Criteria:**

   - [ ] A test asserts the printed output contains the error message text.
   - [ ] `CGO_ENABLED=0 go test -race ./...` green.
- **Pitfalls & edge cases:**
   - The builtin `println` writes to **stderr**, not stdout. The test must capture
     stderr (e.g. swap `os.Stderr` via `os.Pipe`) ‚Äî capturing stdout will find
     nothing. Restore `os.Stderr` in `t.Cleanup`.
   - Keep the fix minimal: stay on the builtin `println` (the other helpers in this
     file use `log`/`panic`/builtin deliberately ‚Äî `CloseOrPrintLn` is the

     point of having a distinct helper.

     guard stays).
   - This is `CloseOrPrintLn`'s own test ‚Äî it currently has none; add one
     `TestCloseOrPrintLn` with subtests (`error case`, `nil case`).
- **Complexity:** Easy

### Task 3: Default the middleware response status to 200 for the implicit-200 path

- **Description:** In `middleware.go`, a handler that writes a body without calling
  `WriteHeader` (the common implicit-200 case) leaves `interceptor.code == 0`. The
  logged response line at `middleware.go:113` becomes `<proto> 0 ` (empty
  `http.StatusText(0)`) and the stdout summary at `middleware.go:45` prints `[0]`,
  while the client actually received 200 (net/http defaults to 200 on first
  `Write`). Fix: mirror net/http ‚Äî in `interceptor.Write` (`middleware.go:93`), if
  `i.code == 0`, set `i.code = http.StatusOK` before recording. Optionally also seed
  `code` in `newInterceptor` (`middleware.go:65`), but the `Write` guard is the
  authoritative fix because a response with no body and no `WriteHeader` is a
  genuine 200 too ‚Äî see pitfalls.
- **Acceptance Criteria:**
   - [ ] A handler that only calls `Write` (never `WriteHeader`) results in a logged
         response status of `200` and the summary prints `[200]`.
   - [ ] A handler that calls `WriteHeader(404)` (with or without a body) still logs
         `404` ‚Äî the explicit code is not clobbered.
   - [ ] The fix lives in `interceptor.Write` (guard on `i.code == 0`), so an
         explicit `WriteHeader` before `Write` wins.
   - [ ] `CGO_ENABLED=0 go test -race ./...` green.
- **Pitfalls & edge cases:**
   - A handler that writes **nothing** and never calls `WriteHeader` still gets a
     200 from net/http, but `interceptor.Write` is never called, so `i.code` stays
     0. Decide the policy: seeding `code = http.StatusOK` in `newInterceptor` covers
     this case too and matches reality. If both the seed and the `Write` guard are
     applied, they must not conflict ‚Äî seed to 200, let `WriteHeader` override, and
     the `Write` guard becomes redundant-but-harmless. Recommend: seed in
     `newInterceptor` AND keep the `Write` guard (defense in depth, both cheap).
   - Do not set `i.code` in `Write` *after* a real `WriteHeader` already set it ‚Äî
     the guard `if i.code == 0` prevents that; 0 is not a valid HTTP status so the
     sentinel is safe.
   - The interceptor's `Bytes()` (`middleware.go:113`) reads `i.code` ‚Äî verify the
     timing: `Bytes()` runs in the deferred goroutine after `nextFunc` returns, so
     the code is settled by then.
   - `TestNewHttpPayloadHandler` is the existing test ‚Äî extend it with subtests
     rather than adding a parallel `Test*` (it already follows the single-func
     pattern). Add an `implicit 200` subtest and an `explicit non-200` subtest.
- **Complexity:** Easy
- **Code Example:**
  ```go
  func newInterceptor(w http.ResponseWriter, r *http.Request) (http.ResponseWriter, *http.Request, *interceptor) {
      i := &interceptor{
          code:           http.StatusOK, // default; overridden by WriteHeader
          payload:        bytes.NewBufferString(""),
          // ...
      }
      // ...
  }

  func (i *interceptor) Write(b []byte) (int, error) {
      if i.code == 0 {
          i.code = http.StatusOK
      }
      // ... (see Task 4 for the response.Write error handling)
  }
  ```

### Task 4: Seed `CyclicOverwritingFilesHandler` size from the on-disk file

- **Description:** `CyclicOverwritingFilesHandler` (`handler.go:91-134`) tracks
  `fileSize` as an in-memory `uint64` starting at 0 on every construction
  (`handler.go:92`). The real on-disk size of the current target file is never
  read. After a process restart the handler opens the existing
  `prefix.00000001.log` in append mode but believes it is empty, so it must write a
  fresh `maxFileCapacity` bytes on top of the existing content before rotating ‚Äî
  the file can grow to roughly 2√ó the cap (or worse across multiple restarts). Fix:
  on construction, `os.Stat` the current target file and seed `fileSize` from its
  actual size. (Seeding once at construction is sufficient since the handler owns
  the file thereafter; do not re-stat on every write ‚Äî that defeats the in-memory
  counter's purpose and adds a syscall to the hot path.)
- **Acceptance Criteria:**
   - [ ] On construction, the handler stats `path.Join(folder, fileName)` and seeds
         `fileSize` from its size; a missing file seeds 0 (no error).
   - [ ] A test pre-creates `prefix.00000001.log` with content near `maxFileCapacity`
         bytes, constructs the handler, writes one small message, and asserts a new
         index file (`prefix.00000002.log`) appears (rotation happened) instead of
         the first file overgrowing past the cap.
   - [ ] A `stat` error other than "not exist" is surfaced ‚Äî it must not be silently
         swallowed (repo "no skipped errors" rule). Decide: return it from the first
         `Write`, or fail-soft to 0 with the error joined into the first write's
         return. See pitfalls for the recommendation.
   - [ ] Existing `TestReinitCyclicOverwritingFilesHandler`,
         `TestCyclicOverwritingFilesHandler`, and
         `TestCyclicOverwritingFilesHandlerForRaceCondition` still pass.
   - [ ] `CGO_ENABLED=0 go test -race ./...` green (including `-race` on the
         concurrent test).
- **Pitfalls & edge cases:**
   - **Construction returns `io.Writer`, not `(io.Writer, error)`** ‚Äî that's the
     existing public signature and must not change. So a stat error cannot be
     returned at construction. Options: (a) capture the stat error in a closure var
     and join it into the first `Write`'s returned error; (b) fail-soft to
     `fileSize = 0` on any stat error except surface non-`IsNotExist` errors via
     the first write. Recommend (a)+(b): `os.IsNotExist` ‚Üí seed 0 silently;
     any other error ‚Üí seed 0 but stash the error and `errors.Join` it into the
     first `Write` return. This honors the no-skipped-errors rule without breaking
     the signature.
   - `TestReinitCyclicOverwritingFilesHandler` uses a **fresh** `t.TempDir()` and
     starts at index 1 with no pre-existing file, so it is unaffected ‚Äî confirm by
     re-running it. The new behavior only kicks in when the target file already
     exists at construction.
   - The handler always starts at `index = 1` (`handler.go:93`); seeding only the
     *current* file's size means after a restart it appends to index 1 even if
     higher-index files exist from prior runs. That is the existing
     index-resets-to-1-on-restart behavior; this task does NOT change index
     selection ‚Äî only the size accounting for the current file. Document this scope
     boundary in the godoc so the limitation is explicit, not surprising.
   - `bytes.TrimSpace(msg)` plus a newline is what's written; the seeded size must
     count actual on-disk bytes (use `FileInfo.Size()`), not the logical message
     length.
   - Watch the `‚Ä¶ForRaceCondition` test: the stat happens once at construction
     (before any goroutine writes), so it does not introduce a race. Keep the stat
     outside the `writer.h` closure's per-call path, or guard it with the same
     mutex ‚Äî recommend doing it at construction time, before the `writer` is
     returned, so no locking question arises.
- **Complexity:** Medium
- **Code Example:**
  ```go
  func CyclicOverwritingFilesHandler(folder, fileNamePrefix string, maxFileCapacity uint32, maxFilesInFolder int) io.Writer {
      index := 1
      fileName := fileNamePrefix + "." + fmt.Sprintf("%0.8X", index) + "." + defaultFileExtension

      var fileSize uint64
      var seedErr error
      if fi, err := os.Stat(path.Join(folder, fileName)); err == nil {
          fileSize = uint64(fi.Size())
      } else if !os.IsNotExist(err) {
          seedErr = err // surfaced on first Write, not swallowed
      }

      w := &writer{
          h: func(msg []byte) (int, error) {
              err := seedErr
              seedErr = nil
              // ... existing open/write/rotate logic, errors.Join'd into err ...
          },
      }
      return w
  }
  ```

### Task 6: Stop ignoring `i.response.Write(b)` error in the interceptor

- **Description:** `middleware.go:94` discards the return of `i.response.Write(b)`.
  `*bytes.Buffer.Write` never returns a non-nil error, so behavior is unaffected,
  but it violates the repo's "no skipped errors" rule (`bytes.Buffer` is not on the
  documented exception list). Fix minimally: capture and join the buffer write
  error into the error returned to the caller, consistent with how the file
  handlers join (`handler.go:103-110`). Keep it idiomatic; do not over-engineer
  (no retry, no logging here).
- **Acceptance Criteria:**
   - [ ] `interceptor.Write` no longer discards the buffer-write return; the buffer
         error (if any) is joined with the downstream `ResponseWriter.Write` error
         and the combined error is returned.
   - [ ] The returned byte count remains the count from the real
         `i.ResponseWriter.Write` (the value the HTTP layer cares about), not the
         buffer's.
   - [ ] `go vet ./...` clean; no `_ =` discard of an error in this function.
   - [ ] Existing `TestNewHttpPayloadHandler` still passes.
   - [ ] `CGO_ENABLED=0 go test -race ./...` green.
- **Pitfalls & edge cases:**
   - This is folded into the same function edited in Task 3 (`interceptor.Write`) ‚Äî
     do both edits together to avoid a churny double-touch. Order in the plan keeps
     them traceable, but the engineer edits `Write` once.
   - The contract of `io.Writer.Write` is "return n from the perspective of the
     destination" ‚Äî return the `ResponseWriter`'s `n`, not the buffer's, so callers
     see the bytes that actually went to the client.
   - Don't let a (theoretically impossible) buffer error short-circuit the real
     write to the client ‚Äî write to the real `ResponseWriter` regardless, then join.
- **Complexity:** Easy
- **Code Example:**
  ```go
  func (i *interceptor) Write(b []byte) (int, error) {
      if i.code == 0 {
          i.code = http.StatusOK
      }
      _, bufErr := i.response.Write(b)         // bytes.Buffer never errors, but no skipped errors
      n, wErr := i.ResponseWriter.Write(b)
      return n, errors.Join(bufErr, wErr)
  }
  ```

### Task 7: Clean ceremonial `&v` indirection in `json.go` (8b)

- **Description:** `JsonDecode` (`json.go:20`) calls `json.NewDecoder(r).Decode(&v)`
  where `v` is already an `any` interface ‚Äî taking its address yields `*any`, which
  `encoding/json` then has to unwrap again. Same noise in `JsonEncodeEx`
  (`json.go:41`, `encoder.Encode(&v)`) and `JsonDecodeEx` (`json.go:47`,
  `Decode(&v)`). This is internal noise, NOT a signature change ‚Äî the exported
  signatures keep `v any`. Pass `v` directly. Safe cleanup.
- **Acceptance Criteria:**
   - [ ] `JsonDecode`, `JsonEncodeEx`, `JsonDecodeEx` pass `v` (not `&v`) to the
         encoder/decoder.
   - [ ] No exported signature changes.
   - [ ] All existing `json_test.go` tests pass unchanged (decode-into-pointer and
         decode-into-map cases both still work).
   - [ ] `CGO_ENABLED=0 go test -race ./...` green.
- **Pitfalls & edge cases:**
   - This is behavior-preserving only because callers already pass a pointer (or a
     map/slice) *as* `v`. `Decode(v)` and `Decode(&v)` behave the same when `v` is a
     non-nil pointer wrapped in an interface, but `Decode(&v)` would *replace the
     interface value* if `v` were a nil interface ‚Äî verify the existing tests cover
     decode-into-`*struct` and decode-into-`map`/`*any`. If any test relies on
     decoding into a bare `any` (no pointer), keep that case green; if it breaks,
     that's a real semantic the cleanup must preserve ‚Äî stop and flag.
   - Encode side (`JsonEncodeEx`): `Encode(&v)` vs `Encode(v)` produce identical
     JSON for the same underlying value; the `&v` was pure noise. Confirm with the
     existing `TestJsonEncodeEx*` tests.
- **Complexity:** Easy

### Task 8: Make `Fatalf`/`Fatal` godoc match actual behavior (7)

- **Description:** `Fatalf` (`logger.go:189`) and `Fatal` (`logger.go:199`) carry
  godoc "writes a ‚Ä¶ log message and exits the program", but the bodies `panic` (a
  recoverable signal ‚Äî deferred funcs run, and `TestLogger_PrintAndFatal` already
  recovers from it). This is PUBLIC API: changing `panic` ‚Üí `os.Exit` would alter
  runtime behavior for every consumer (defers would stop running, recover would
  stop working). The **safe** fix is documentation-only: make the godoc accurate ‚Äî
  state that the methods log the message then `panic` (not `os.Exit`), so deferred
  functions still run and the panic is recoverable. Do NOT change behavior. The
  "should it actually `os.Exit` or be renamed `Panicf`/`Panic`" question is deferred
  to the user (see Trade-offs).
- **Acceptance Criteria:**
   - [ ] Godoc on `Fatalf` and `Fatal` accurately states: writes the message, then
         `panic`s (recoverable; defers run); it does NOT call `os.Exit`.
   - [ ] No change to the method bodies ‚Äî they still `panic(m)`.
   - [ ] Godoc starts with the identifier name and ends with a period (repo
         convention).
   - [ ] `TestLogger_PrintAndFatal` still passes unchanged.
   - [ ] `gofmt -l .` empty; `go vet ./...` clean.
- **Pitfalls & edge cases:**
   - Resist the temptation to "fix" the behavior to match the name ‚Äî that's the
     breaking change the user must sign off on, not a silent docfix.
   - Keep the godoc honest about *why* (recoverable, defers run) so consumers who
     read "Fatal" and expect process death are warned.
- **Complexity:** Easy

### Task 9: Benchmark `WriteLog` and add a 0/1-sink inline fast-path (9)

- **Description:** `WriteLog` (`logger.go:122-169`) spawns a goroutine per handler
  AND per matching hook on **every** call, plus a `WaitGroup` and a mutex-guarded
  error slice ‚Äî goroutine-per-write on the logger's hot path. Scope this
  conservatively: (a) add a benchmark (`-bench`) to quantify current cost across a
  few sink counts (0, 1, N hooks/handlers); (b) add a cheap fast-path ‚Äî when the
  total number of sinks to write for this call is 0 or 1, write inline (no
  goroutine, no `WaitGroup`, no mutex) and return the single error directly; keep
  the existing concurrent path for 2+ sinks. Do NOT rip out the concurrency model ‚Äî
  the `‚Ä¶ForRaceCondition` tests depend on it.
- **Acceptance Criteria:**
   - [ ] A benchmark exists (e.g. `BenchmarkWriteLog`) covering 0 sinks, 1 sink, and
         several sinks; runnable via
         `CGO_ENABLED=0 go test -bench=. -benchmem -run=^$ .`.
   - [ ] Fast-path: when the number of sinks that will actually be written for this
         call is 0 ‚Üí return early; 1 ‚Üí write inline and return that write's error
         directly (no goroutine/WaitGroup).
   - [ ] The fast-path preserves ALL current behavior:
     - below-min-level early return still returns `(0, nil)` and still fires exact-
       match hooks (the hook-fires-below-minimum asymmetry at `logger.go:146-159`);
     - error aggregation is equivalent (single error returned directly for 1 sink;
       `errors.Join` for 2+);
     - the returned `n` is `len(message)` on the success path, `0` below minimum.
   - [ ] The 2+ path is unchanged (still concurrent + joined).
   - [ ] `CGO_ENABLED=0 go test -race ./...` stays green, including
         `TestLogger_WriteLog`, `TestLogger_Hook`, and both `‚Ä¶ForRaceCondition`
         tests.
- **Pitfalls & edge cases:**
   - "Total sinks for this call" is not just `len(handlers)`. It is: (matching hooks)
     + (handlers, but only if `level >= minimumLogLevel`). Below the minimum the
     handler count is 0, so a single matching hook is the only sink ‚Üí fast-path.
     Above the minimum with one handler and no matching hook ‚Üí 1 sink ‚Üí fast-path.
     Compute the effective count correctly *before* choosing the path, or you'll
     mis-route and break the asymmetry.
   - The exact-match-hook-fires-below-minimum behavior is the trickiest invariant.
     The fast-path must fire those hooks even when handlers are gated out. Re-read
     `logger.go:146-159` ‚Äî hooks loop runs first, THEN the min-level gate. Preserve
     that ordering in the fast-path.
   - Counting matching hooks requires a pass over `l.hooks` anyway; do it once,
     collect the target writers into a small slice, then branch on its length plus
     the handler contribution. Avoid double-iterating.
   - Hold `l.m.RLock()` for the whole computation as today ‚Äî the fast-path must not
     drop the read lock early or it races with `Hook`/`Unhook`/`SetMinLevel`.
   - Inline write must still respect that each sink is a mutex-guarded `writer`
     (registration wraps them) ‚Äî the fast-path calls `w.Write` directly, which is
     already safe; don't add a second lock.
   - Returning the single error directly vs `errors.Join(err)` of one element: both
     are equivalent for callers using `errors.Is`, but returning it directly avoids
     an allocation and is the point of the fast-path. Confirm any test that asserts
     on the *exact* error type (not `errors.Is`) still passes ‚Äî prefer `errors.Is`
     assertions if a test is brittle.
   - Keep `n = len(message)` semantics identical; don't return the sink's reported
     byte count.
- **Complexity:** Medium
- **Code Example:**
  ```go
  // sketch: collect sinks, then branch on count
  var sinks []io.Writer
  for _, h := range l.hooks {
      if level == h.Level {
          sinks = append(sinks, h.Writer)
      }
  }
  if level >= l.minimumLogLevel {
      sinks = append(sinks, l.handlers...)
  } else if len(sinks) == 0 {
      return 0, nil // below min, no matching hooks
  }

  switch len(sinks) {
  case 0:
      // below-min already handled above; here means at/above min with no handlers
      return n, nil
  case 1:
      _, err := sinks[0].Write(message) // inline, no goroutine
      if level < l.minimumLogLevel {
          return 0, err
      }
      return n, err
  default:
      // existing concurrent fan-out + errors.Join
  }
  ```
  > Note: the sketch's `n`-vs-`0` return logic must match the current contract
  > exactly ‚Äî verify against `TestLogger_WriteLog` before finalizing the branch
  > shape. The hook-fires-below-min case (1 hook, level below min) must return
  > `(0, hookErr)`.

## Execution Order

## Risks

- **API stability ‚Äî `Fatal*` (Task 8 / finding 7).** The default is docs-only. If
  the user later wants real fatal semantics (`os.Exit`) or a rename to
  `Panicf`/`Panic`, that is a **breaking change** for consumers (defers stop
  running; `recover` stops working; existing `TestLogger_PrintAndFatal` recovery
  would break). Flagged, not done. Owner decision required before any behavior
  change ‚Äî see Trade-offs.
- **API stability ‚Äî `NewLogger` error return (finding 8a).** `NewLogger` returns
  `(*Logger, error)` but never returns a non-nil error (`logger.go:15-31`).
  Dropping the error return is a **breaking** public-API change (every caller's
  `l, err := NewLogger(...)` would fail to compile). Default: KEEP the signature.
  Optionally tighten the godoc to note the error is currently always nil but
  reserved for future validation. No code change to the signature in this plan ‚Äî
  see Trade-offs.
- **Rotation seed scope (Task 4).** The fix seeds only the *current* index file's
  size; it does not change the index-resets-to-1-on-restart behavior. A folder with
  pre-existing higher-index files from a prior run will still append to index 1
  after restart. This is an existing limitation, now documented but not solved ‚Äî
  solving it (scanning for the highest existing index) is out of scope and would be
  a separate, riskier change. Risk: a user might expect "resume at the latest file";
  the godoc must set the expectation honestly.
- **Fast-path correctness (Task 9).** The hook-fires-below-minimum asymmetry is
  subtle; a wrong sink count routes the call down the wrong branch and silently
  changes which sinks fire. Mitigation: the acceptance criteria explicitly pin the
  below-min + matching-hook case, and the existing `TestLogger_Hook` /
  `TestLogger_WriteLog` / `‚Ä¶ForRaceCondition` tests must stay green under `-race`.
- **Test capturing stderr (Task 2).** `println` writes to stderr; a test that
  captures the wrong stream will pass vacuously or hang on the pipe. Mitigation:
  acceptance criterion names stderr explicitly and requires restore in cleanup.
- **`bytes.Buffer` "error" handling (Task 6) reads as dead code.** A reviewer may
  flag `errors.Join(bufErr, ...)` where `bufErr` is always nil as pointless.
  Mitigation: a short comment explaining it satisfies the no-skipped-errors rule and
  guards against a future swap of the buffer type.

## Trade-offs

- **`Fatal*`: fix docs vs. change behavior (default: fix docs).** Chose
  documentation accuracy over changing `panic` ‚Üí `os.Exit` because the latter is a
  silent breaking change to public runtime behavior ‚Äî consumers who `recover()` or
  rely on `defer` cleanup (and the library's own test) would break. The honest
  godoc costs nothing and removes the trap. *Open question for the user:* do you
  want a future `Panicf`/`Panic` alias (non-breaking, additive) plus a deprecation
  note on `Fatal*`, or real `os.Exit` semantics (breaking, needs a major version)?
  Default until you say otherwise: keep behavior, fix docs only.
- **`NewLogger` signature: keep `(*Logger, error)` (default: keep).** Chose
  backward compatibility over signature minimalism. The always-nil error is dead
  ergonomics, but removing it breaks every consumer at compile time. Keeping it
  leaves the door open for future validation (e.g. rejecting a nil handler or an
  invalid level) without another breaking change. *Open question for the user:*
  accept the always-nil error as a reserved-for-future contract (document it), or
  schedule its removal for a future major version? Default: keep + document.
- **Rotation: seed current file only vs. full index recovery (default: seed
  current).** Chose the minimal, low-risk fix (stat the current index file) over
  scanning the folder to resume at the highest existing index. The minimal fix
  stops the over-cap overgrowth ‚Äî the actual reported bug ‚Äî without re-architecting
  index selection or risking the `‚Ä¶ForRaceCondition` coverage. Full
  resume-at-latest is a larger, separately-planned change.
- **Fast-path: inline 0/1 vs. worker pool / no goroutines at all (default:
  inline 0/1).** Chose the surgical fast-path over a bigger redesign (worker pool,
  removing concurrency, sync fan-out always). The inline path captures the common
  single-sink case (the default logger has exactly one handler) with near-zero
  risk, while the `‚Ä¶ForRaceCondition` tests keep guarding the concurrent path for
  2+ sinks. A full sync rewrite would be simpler to reason about but would remove
  the parallelism the tests assert and could serialize a slow sink (e.g. Telegram)
  behind a fast one ‚Äî not worth it without data. Benchmark first, then decide if
  more is warranted.
- **Stderr-only `println` in `CloseOrPrintLn` (default: keep builtin).** Chose to
  keep the builtin `println` (just fixing `.Error()`) over switching to `fmt`/`log`,
  because the four `CloseOr*` helpers are deliberately differentiated by output
  channel/dependency; collapsing them would be a behavior change for consumers who
  picked `CloseOrPrintLn` specifically for its stderr-builtin profile.
